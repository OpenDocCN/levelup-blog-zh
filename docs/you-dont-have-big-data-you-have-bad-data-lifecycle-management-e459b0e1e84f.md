## 存储并不总是便宜的

![](img/9e53710b88c83b166681d92798f552d2.png)

[数字图像 NeONBRAND 数字营销，[https://unsplash.com/photos/KZs5Bt5VDng](https://unsplash.com/photos/dDvrIJbSCkg)

我们正处于数据黄金时代。[全球数据球](https://www.statista.com/statistics/871513/worldwide-data-created/)在 2010 年产生了大约 20 亿字节，在 2020 年产生了大约 64 亿字节(x32！).数据量呈指数级增长。

作为个人，我们生产和消费数量惊人的数据。作为一家公司，我们希望利用这些来做出正确的决策和更智能的产品。鉴于此，很容易陷入仅仅为了存储而存储数据的陷阱。在当今的市场条件下，数据团队在总拥有成本方面将面临越来越多的挑战。“存储便宜”可能是一个神话。让我们找出原因。

# 🤔所有的模糊开始的地方

让我们得到一个关于“大数据”来自哪里的小提示。它指的是 3 V:速度(你需要数据的速度)、体积(大小)和多样性(你的数据来源的多样性)。

在本文中，我们将只关注一个最被低估的维度:体积。

# 💨ELT 的数据收集和衰落

如今事实上的策略是捕获几乎`any`数据，因为我们有“潜在的”未来用例，但没有活跃用户/用例。如果我们不抓住它，它就没了。

这就是过去几年中 [ELT](https://glossary.airbyte.com/term/elt#:~:text=elt/) (提取-加载-转换)大放异彩并接管他的老兄弟 [ETL](https://glossary.airbyte.com/term/etl-vs-elt/) (提取-转换-加载)的地方，因为我们需要一种更灵活的方式来接收数据，而不需要转换和建模。但是滥用这种模式最终会收集到大量未使用的数据。

# 🎌常见陷阱

下面是我反复看到的最明显的一个:

*   用于开发/实验目的的数据拷贝太多，没有截止日期:数据像幽灵一样留在那里。
*   对一些原始的`csv` / `json`文件不做任何文件转换:我们有许多更适合存储和查询的文件格式(例如:parquet，ORC，Avro…)。
*   太多无用的快照。
*   太多暂存管道数据未被清理。
*   数据仓库中有太多的重新加载，而没有利用 lake house/对数据湖的直接查询。
*   捕获的数据太多，根本用不上。

最后一个很难解决，因为我们将来可能会有一个用例。然而，这个未来“什么时候”会发生？用以下问题挑战自己:

*   相对于运营开销，这个用例真的有价值吗？
*   如果我们现在不捕获这些数据，有什么方法可以回填吗？

首先拥有“知识”的方法将有助于决定您是否需要首先捕获这些数据。

> 但是等等，存储不是很便宜吗？

是的，它是。但是因为它太容易储存，所以我们倾向于滥用它，坏的模式也很容易使用。存储大量的数据仍然会增加。

# 🫰embracing·费诺普斯

如上所述，有不同的方法来解决这个问题，但是从第一天开始就要采用 FinOps。

这意味着，例如:

*   创建云预算警报。
*   创建消费仪表板。

对于后一个问题，问自己以下问题？

*   最大的对象存储桶是什么？
*   最大的桌子是什么？
*   这个大桶/表最后一次被访问是什么时候？

清晰的监控和警报将有助于你发现事情何时失控，而不是在账单已经存在时才做出反应。

此外，对你的成本有一个清晰的看法总是你的领导所喜欢的，因为它更容易计划预算(包括人员和资源)。

奖励:如果您使用 AWS S3，您应该检查[智能分层](https://aws.amazon.com/s3/storage-classes/intelligent-tiering/)。它会根据您的配置自动将您的数据移动到更便宜的存储设备。

# 🩹ETL 来救援了吗？

除了 FinOps，我们也看到了 ELT 范式是如何被误用的。ETL 也有帮助，因为你可以做一些转换的 T2，而不是存储原始数据。

例如，如果您将一些源连接起来，只是将它们转换成更合适的文件格式(例如，parquet)，那么一旦转换完毕，保留原始文件就没有什么价值了。

无论如何，如果您的存储成本激增，牢记 ELT 的权衡并重新思考是一件好事。

# 💸一切都是便宜的，直到它成为优先事项。

如您所见，存储的最大危险是我们都知道存储很便宜，因此，我们列出的所有不良模式都不是重点关注的重点，直到您的信用卡预算耗尽。

然而，今天，您可以从数据之旅的第一站就开始预算和监控。没有必要在这方面发疯，但这应该随着您的用例以及数据成熟度的发展而发展，以便您保持低成本和高附加值。
