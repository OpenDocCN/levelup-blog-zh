# 如何更频繁地写作能让你成为更好的程序员

> 原文：<https://levelup.gitconnected.com/how-writing-more-often-can-make-you-a-better-programmer-46694d75225>

我已经从事专业编码工作大约 5 年了，我也在差不多相同的时间里继续写作，尽管工作量要小得多。在这段时间里，我学会了发现这两种工艺之间的一些关键相似之处。尽管这两个领域表面上看起来有很大的不同，但我真的相信，提高我的写作技能会对我思考代码的方式产生溢出效应。

尽管它们在创造性/技术性方面处于对立的两端，但从本质上来说，写作和编程都是关于交流的——只不过受众大不相同。

我了解到，作为一名作家，你的工作是以一种广大读者易于理解的方式传达你的思想和观点。你需要放下你的论点，或者构建你的故事，这样它就可以毫不费力地从头到尾。你的写作需要清晰简洁，不能有太多不必要的偏差。我们被告知，写作的黄金法则是让事情简单，而且[越少越好。](https://medium.com/personal-growth/george-orwells-six-rules-for-great-writing-4db6d31ff136)

编程是非常相似的，一旦你意识到你的受众到底是谁。十有八九，你的目标读者不是电脑，而是阅读你代码的人。你的队友。多年后维护你的代码的人。听起来很奇怪，甚至可能是你未来的自己。坦白地说，我感到尴尬的是，有多少次我在写完自己的代码后仅仅几个星期就阅读了它，却完全不能理解它。

作为一名软件开发人员，你工作的很大一部分就是为那些完全没有背景知识的人编写易于理解的代码。通常，您的目标是通过编程概念和架构对现实世界中的实体进行建模。

编程语言毕竟是*语言*。它们定义了语法和词汇，就像现实生活中的语言一样。在构建应用程序时，它们为您提供了大量的工具。你有责任找到尽可能好的结构和语法来表达你的每一步逻辑。就像你在写作中表达自己时会找到合适的词一样。

现在，我不是建议你让斯蒂芬·金写下一个超级可扩展的百万美元应用。我也不是建议软件开发人员应该突然放弃他们的职业，去追求成为 NYT 畅销书作家。我不想贬低编写高性能代码、构建和管理复杂系统或拥有专业技术知识的重要性——所有这些都需要数年甚至数十年的经验。所有这些都有助于成为一名高效的开发人员。

但是我相信拥有一个好代码的心智模型是一个很好的基础。我相信任何形式的写作都有助于强化这种心智模式。尤其是当你在杂草中日复一日地写代码的时候。很多时候，高性能的代码不一定等于难看的代码。通常有很多方法可以在两者之间达成妥协。但是你必须积极地朝着它努力。

最终，为了做好这两件事，你需要清晰地思考。而*那个*就是无论你做什么，它都会跟随着你。我确实发现写作在很多不同的方面对我来说都是一种宝贵的财富，我鼓励更多的人涉足写作。

另一方面，通常是这样的见解让我重视劳动力的多样性，尤其是工程方面。我们倾向于认为工程是一个高度技术性的职业，几乎没有创造的空间。但这并不完全正确。有创意背景的人通常会带来他们自己独特的经验和直觉，甚至更有可能为整个产品做出独特的贡献。

不出所料，我给新开发人员(也给任何技能水平的程序员)的推荐书目是罗伯特·马丁的[干净代码](https://www.amazon.ca/Robert-Martin/dp/0132350882/ref=sr_1_1?dchild=1&keywords=clean+code&qid=1597465946&sr=8-1)。强烈推荐你去看看。
