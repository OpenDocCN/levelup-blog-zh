为什么我们甚至需要单例模式？考虑这样一种情况，您只想创建该类的一个实例。例如，假设您在项目中调用一个 API，因为每个 API 的第一步都是验证客户端。身份验证成功后，服务器会生成一个会话，该会话会存在一段时间。

想象一下，对于每一个请求，您都在创建一个会话，这在很多层面上都是错误的。例如，一个脸书的登录到注销是一个会话。现在，如果你喜欢一个帖子，你的会话将到期，你必须再次登录。如果您再次添加一篇文章，您将创建一个新的会话，该会话会将您从系统中注销。

您应该只为所有其他请求创建一个会话。一旦你登录，你可以做许多喜欢，评论和帖子，而不用关闭会话。

这就是我们可以使用单例模式的地方。

没关系，没有必要惊慌，因为我们有解决这种情况的办法。我们有两个关键词来帮助我们:

> **易变和同步**

## 易变的(简而言之):

在多线程场景中，当前正在访问 CPU 的线程可以缓存一些资源，如属性、变量等。资源存储在 CPU 的高速缓存中，以加快处理速度。当它拥有这些变量时，thread 可以修改这些变量的值。

现在的问题是，其他线程仍然可以访问内存中的旧值。这里我们有一个不匹配的问题，所以为了避免这个问题，我们使用了可变的关键字。它确保缓存数据在 RAM 上更新，反之亦然。

因此，如果资源被同时执行的多个线程修改，它们仍然可以拥有更新的值。

## 同步(简单地说):

同步一次只允许一个线程访问资源。在被分配的线程完成其任务之前，没有其他线程可以访问该资源。线程异步执行。同步通过使线程同步资源来帮助我们。
