# 自卑感导致对复杂代码的盲目崇拜

许多最近才开始 web 开发生涯的人可能不会记得这一点。虽然现在不像以前那么糟糕，但在不久前的一段时间里，以编写网站代码为生的人自称为软件工程师被认为是令人憎恶的。在一些国家，如加拿大，你仍然不能，除非你持有工程学位。技术论坛上充斥着愤怒的帖子，人们互相辱骂，仅仅是因为有人在编写 HTML、CSS 和一点 PHP 或 JavaScript 时，竟敢自称为软件工程师。见鬼，正是这种语言的名字，JavaScript 被用来作为反对称之为编程的论据，因为很明显它只能是一种脚本语言，而编写脚本不能被认为是软件工程。

聪明的解决方法？ **Web 2.0。**从网站转向网络应用。但任何花了几个小时研究这个问题的人都会很快意识到，**这其实都是营销。Web 1.0 是单向的——建立网站的人也是直接或间接分享内容的人。Web 2.0 真正做的只是向消费者开放网络，他们突然发现自己能够成为创造者，而不需要依赖开发者。这就是今天的网络使用现状。Web 3.0 的现状与其说是现实，不如说是一个湿梦，所以与这个故事没有真正的关联。**

然而，Web 2.0 或当前的 Web 与我们今天在软件开发中所处的位置有着千丝万缕的联系。为了让 web 更像一个本地应用程序，许多聪明人想出了越来越复杂的解决方案，而这种复杂性催生了新一代的 web 开发人员，他们现在觉得更有资格称自己为程序员、工程师等等。这本身就不是问题。我个人认为，从 Web 1.0 开始，他们每个人都有资格获得这个头衔。

> 问题是，更多的时候，我们增加代码的复杂性并不是因为功能需要，而是为了满足我们的自我或平息我们的冒名顶替综合症。

但事情并没有就此结束。我们更进一步。更进一步说，在一个你从 HTML 中链接的 CSS 文件中编写普通的 CSS 被认为是不酷的，老派的，这是你永远不应该在面试中提到的。不。你要么使用 UI 框架，如果你这样做了，你必须能够与其他七家公司进行比较和对比，要么做一个长达一小时的演示，介绍 JSS 相对于顺风或 Bootstrap 的优势。打着*的旗号“我们现在正在构建应用，没有额外的复杂性你无法构建应用”*。我讨厌叫醒人们，**但是不管是不是应用程序，它仍然在浏览器中运行，因此它是一个网页**。通常情况下，你正在构建的所谓的应用程序，本质上是一个表单，几个按钮导致由 URL 驱动的视图。猜猜还有什么可以用同一句话形容？一个网站！是的，这包括 pwa(它似乎失去了受欢迎程度和浏览器支持，而不是增加)。

现在，我在企业级 web 平台上工作了多年，我同意复杂性确实进入了画面，但出于非常不同的原因，例如试图为用户解决复杂问题的复杂业务逻辑，或者为了帮助团队之间标准化编码实践而引入的某些 T2 复杂性，其中主要是可重用性，这恰好是我的下一个观点。

# 当干燥成为唯一的规则

我还没有发现任何一个编程教程不强调 DRY 是软件开发中的规则。无论你做什么，不要重复。永远不会。因为如果你这样做了，你和每一个敢于复制粘贴他们知道已经在别处工作过的东西的软件工程师都会有一个特别的地狱。

保持你的代码简单意味着它有一些现成的好处，使它易于阅读、扩展、维护、替换和重用。修卡，对吗？然而，不知何故，大量的程序员似乎对简单的代码感到厌恶。似乎有一种与软件开发中的过度工程化相关的自豪感，我不知道为什么。有些人声称简单的解决方案对工程师同事来说不够令人印象深刻，就好像 web 开发软件是为工程师同事开发的一样，其他人说**这是一个古老的策略，让一个人在他们的工作中不可或缺**，因为没有人想碰它，因此最初的工程师不能被解雇。当然，也有例外，但是没有客户会关心你的超级复杂的、几乎不可读的 200 行代码的解决方案，它需要 7 个 Babel 插件和 60 行额外的 Webpack 配置才能在当前的浏览器上运行。我在这里可能有点夸张，但你会明白的。我希望！关于哪一点，为什么解决方案需要 200 行难以理解的代码？

# 早熟优化综合症

你知道吗？有人认为他们的代码会在接下来的三个冰河世纪中存活。到第一个版本时，所有人都将早已消失，但他们的代码……嗯，这将在每个可以想象的浏览器和元宇宙的下一个 18.995 主要版本中运行良好，并适应每个新功能*“应用程序”*可能需要 500 年，他们为之编码的公司甚至已经从历史书上消失了。

持续的或机会主义的重构发生了什么？我告诉你。每个人都买了马丁·福勒的书，因为把它放在书架上很酷，但有多少人读了它，又有多少人采纳了可能是有史以来最重要的建议？可能 1%或更少。

相反，令人愤怒的是，工程师们经常面临一个需要解决的问题，但由于过度热情、自负和缺乏商业头脑，他们没有选择更明显的解决方案，而是选择花费两倍到十倍于必要时间的时间，通过客观的过早优化来*【避免技术债务】*。令人担忧的是，通常这种技术债务完全是虚构的，它基于软件从未打算迎合的场景，或者更糟的是，业务从未计划实现或甚至考虑的软件开发方向。我们都同意技术债务不是代码中最大的事情，通常被认为是潜在的浪费，另一方面，过早的优化肯定是浪费，因为它…过早！

# 缺乏商业头脑

我经常想，如果软件开发人员在他们曾经为之编写代码的公司中突然被赋予 CEO 或 CTO 的角色，他们会如何做出决定？因为**很少软件工程师关心成本**，这很可怕，有时我真的会问自己，在工程领域缺乏商业头脑是否是导致软件成本和其他许多东西膨胀的原因。什么时候成本效益成了——比如说——网络开发中的禁忌？这是在工程对话中很少提到的事情。如果有什么不同的话，它总是被看作是用来对付开发者的武器或鞭子，对于我来说，我不明白为什么。

软件是用来解决问题的。如果这个问题可以用 50 万美元而不是 120 万美元可靠地解决，那么很明显，商业目标是以更低的成本做到这一点。**软件开发中的每个决策都需要成本效益分析**，即使它不会直接导致金钱成本。如果从角度移动到反应是一个工程要求，那么是什么推动了这个要求？这是否会带来更好的应用程序，是否意味着团队可以更快地开发功能，企业是否能够留住更多人才或更快地建立新团队？

照片由 [Ruthson Zimmerman](https://unsplash.com/@ruthson_zimmerman?utm_source=medium&utm_medium=referral) 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

但它甚至不一定是一个首要问题。它可以是简单的事情，比如花时间进行重构或优化。这是在正确的时间做出的正确的努力吗？如果是这样的话，收益至少可以弥补成本，但是在错误的时间花时间做同样的事情，成本会使公司陷入财务困境。

另一个很少考虑的方面是**在错误的问题上花费时间和金钱**，可能会以意想不到的方式对开发者产生反作用。其中之一是压力增加，因为最后期限不能总是被移动，并且要满足业务承诺，这可能导致筋疲力尽。

第二个是现有的**工程师花费了公司过多的成本**，从而减缓或完全阻碍了公司工程资源的增长。简单来说，什么比较好？一个每年花费 100 万美元对所有事情进行过度设计的小团队，还是两个具有良好商业头脑的大团队，遵循 KISS 和机会主义的重构，保持公司的愿景和客户的快乐？就我而言，我投票赞成后者。我更愿意在一家成功公司的创收产品中看到技术债务，而不是在一家失败公司的未发布产品中看到一个干净的仓库，里面装满了创新的编码标准，优化到最后一行代码。

# 现实远远超出了代码

我来自一个国家，在 90 年代，共产主义政权垮台后，制造了无数完全由工程师驱动的产品。最让他们困惑的是没有人想要他们。国内市场也没有。它们极其昂贵，过度设计，设计拙劣，并且没有真正迎合用户的需求。

在某些方面，今天的许多软件工程师正在犯同样的错误——被代码弄得过于兴奋，迷失在细节中——并且忽略了一点，即他们是为了他们的用户提供伟大的产品，而不是为了他们自己和他们的工程同行，他们是为了盈利而这样做的。这是否意味着我们必须走捷径？绝对不行。**质量至上，但实用主义也是如此。**

不幸的是，太多时候，我们的应用程序充满了错误，因为我们创建了太复杂的代码，以至于在开发后仅仅几周，我们就无法在没有再次重构的情况下解决自己的问题。我们创建的代码比特性所要求的更难编写和测试。文档化变得不可能，知识转移变得无效，我们最终解决的问题更多的是关于我们自己挖的代码坑，而不是关于客户。

> 如果代码是我们用来与计算机交流的语言，那么为什么我们都如此不顾一切地给自己增加难度呢？

因为无论编码变得多么时髦，**它只是我们用来为客户解决问题的语言**。是的，其中许多都是需要认真思考的复杂问题，但这就是问题所在。我们应该把所有的精力都花在找出解决方案上，而不是试图理解过度设计的官样文章，再多写一些。当我们想出办法的时候，**我们需要保持简单，专注于我们试图解决的问题**，而不是通过解决可能永远不会发生的问题来倾家荡产。面向未来，虽然听起来很棒，但这是一种需要非常小心的方法，永远不应该被用作花费时间编写没有人要求或没有预算的代码的默认借口。

> 为复杂的问题编写简单的代码并不可耻，但是为简单的问题编写复杂的代码可能会可耻。
