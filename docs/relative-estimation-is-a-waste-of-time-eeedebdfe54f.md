# 我们需要停止所有的相对估计。

相对估计没有价值。零值。

评估完全是浪费时间，不仅仅是浪费时间，实际上对敏捷和软件交付是有害的。

你可以声称估算是基于对过去行为的观察，但是软件是一个不断扩展的世界，我们今天面临的挑战不是我们明天将面临的挑战。我们经常实施一些以前从未实施过的东西。

这些猜测的问题在于它们总是错的。这是我们都知道的事情，但出于某种原因，我们表现得好像我们不知道。

# 混乱报告

![](img/f57f1e0164d77bff6e1b1028766c46cb.png)

照片由[玄阮](https://unsplash.com/@darthxuan?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

> “古代的罗马桥是非常低效的结构。以现代标准来看，他们用了太多的石头，因此，建造起来需要太多的劳动力。多年来，我们学会了更有效地建造桥梁，使用更少的材料和更少的劳动力来完成同样的任务。”
> 
> —汤姆·克兰西，所有恐惧的总和

[混沌报告](https://www.csus.edu/indiv/v/velianitis/161/chaosreport.pdf)是一项基于 IT 项目成功率和项目管理策略的研究。这项研究经常被引用为 IT 项目成功率的事实上的权威。成功的项目被定义为满足所有要求，按时完成，并且在预算内。

桥梁通常会在预算内按时建成，而且不会倒塌。另一方面，软件很少按时或在预算之内。

混沌报告指出，软件开发世界中 80%的项目要么失败，要么交付得太晚，以至于被归类为失败。

*   所有软件项目的平均时间超限: **222%**
*   所有软件项目的平均成本超支: **189%**

这是为什么呢？

是因为我们不知道如何评估软件。

我曾经是一个全球性组织的 [LeSS](https://less.works/) 转型的一部分。范围是 120 名工程师，已经分成十几个功能团队，并且已经实践了一些年的 Scrum 形式。当时，我是一个特性团队的 Scrum 主管。

不到两年后，尽管有许多敏捷教练、顾问、工程师认证，以及以前的“敏捷”经验，我还是能看到它不起作用。我检查了每一英尺的每一次冲刺，发现每一次估计都完全错误，每一次冲刺都没有坚持，速度从来不一致，范围不断变化。

在一些极端的情况下，团队估计他们将完成比他们实际完成的多 500%的工作。

在不到一年的时间里(转型进行到一半)，引入了 PI 规划(产品增量)来尝试解决我们的问题。不幸的是，这只会让事情变得更糟。我将把我对 PI 规划的批评留到另一篇文章中。

有趣的是，我为这 12 个功能团队收集的数据与混沌报告中的统计数据几乎完全一致。

一个特别的团队确实以相当准确的估计脱颖而出。幸运的是，我与该团队的一名工程师有着密切的工作关系，所以我想知道他们是如何做到的。喝着啤酒，他们承认篡改了数字，因为他们的经理试图微观管理他们团队所做的一切。

# 估计都是谎言

![](img/f30ad2166b8394977963108e99ac21fe.png)

照片由[布雷特·乔丹](https://unsplash.com/@brett_jordan?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

> “故事点是为了混淆持续时间而发明的，这样某些经理就不会在评估上给团队施加压力”
> 
> — Ron Jeffries(大约在 1996 年，极限编程软件开发方法论的三位创始人之一。

我遇到的每一个软件工程师都曾向我抱怨过经理们是如何把他们的评估当成合同协议的。流行的敏捷扩展方法，比如 Less 和 SAFe，只是通过几个因素夸大了这个问题。

如果工程师知道这些估算是编造的，没有用，为什么我们的管理层不知道呢？每个人都在依赖一些不起作用的东西。

Woody Zuill(他以普及暴民编程而闻名)提出了#NoEstimates 标签，试图围绕他们的问题发起一场革命。

> “#NoEstimates 是一个主题为探索软件开发决策的[时间、精力、成本]替代方案的标签。这就是用“无估计值”做决定的方法
> 
> —伍迪·朱尔

一旦你给一个项目添加了估算，你就不能再保持可持续的进度，因为一旦你开始估算，你就有了最后期限，这些最后期限不是基于事实的，一旦你有了不切实际的最后期限，你就让人们每天工作 12 个小时来满足这些最后期限。

# 速度

![](img/49973b627a9f57ae51e945ac8b462dbc.png)

[*速度定义*](https://www.agilealliance.org/glossary/velocity/) *:在每一次迭代结束时，团队会将与该次迭代中完成的用户故事相关联的工作评估相加。这个总和叫做速度。知道了速度，团队就可以根据与剩余用户故事相关联的估计，并假设剩余迭代中的速度将保持大致相同，来计算(或修改)完成项目所需的估计时间。*

Velocity 允许您使用过去的数据，通过已完成的故事点与剩余的故事点来预测完成日期。

下面是一些用你的 Sprint 数据尝试的作业:

以一个正在进行的项目为例，使用斐波那契估算的故事点，通过比较一段时间内完成的服务点和剩余的服务点来推断预计的完成日期。

重复上述步骤，将所有故事点转换为增量值(1，2，3，4，5)

对所有值为 1 的故事重复上述步骤。

你可能会发现所有 3 种计算都得到了相同的答案…

如果你也是这种情况，那我们为什么还要费心去评估呢？每个故事可以默认具有值 1。

# 什么是故事？

![](img/fe2be588da16f6f119a3d8958b9cf97e.png)

照片由[S O C I A L C U T](https://unsplash.com/@socialcut?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

故事是描述客户做有价值的事情的简短叙述。没有时间的概念，也不应该是一个功能需求的变相写法。

优先权是你比较故事的唯一价值。

“我们估计这个故事需要 3 周时间。”

“对不起，这不符合我们的 2 周冲刺。你能不必要地把它分解成不同的故事，以便适应我们的冲刺吗？非常感谢。”

客户不在乎交付的是 1 层还是 10 层。

# 我们为什么要测量？

![](img/33912382f3abace359d43c98fb84b70b.png)

[廷杰伤害律师事务所](https://unsplash.com/@tingeyinjurylawfirm?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

经理们基于一种误解做出商业决策，认为评估是准确的。他们认为这个数字是一群工程师凭空想象出来的，有一定的真实性。

即使在一个我们可以更精确地估计的完美世界中，达到你的估计也没有价值。如果评估是如此重要，以至于你设置了里程碑来驱动你的项目，并且根据你达到那些里程碑的事实来判断你，那么当你为客户构建了错误的东西却按时完成时会发生什么呢？

时间是衡量成功的错误标准，只会被估计所激怒。停止测量时间，而是测量交付给客户的价值。

# 心理因素

![](img/c241f9939d2ef332f361e47185416c84.png)

照片由[腾雅特](https://unsplash.com/@tengyart?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

工程师动机很少被关注相对评估的敏捷和转换实践所涵盖。

现代敏捷方法经常引入复杂的系统来跟踪团队是否完成了他们在产品增量计划或冲刺计划中估计的工作。然后，工程师们花费大量时间谎报他们的估计，不做承诺，并试图将通用指标游戏化。这导致了巨大的动机问题。自我效能是已建立的心理健康模型(Zimmerman 解释自我效能的模型)的一部分，是你能做某事的信念和围绕它的动机。如果你的时间表和估计没有意义，你花了太多的时间撒谎，那么你就不能建立信念结构或做这件事的动机，这会影响你尝试和做这件事的兴趣。我们需要能够围绕我们所遵循的系统建立信念结构。
