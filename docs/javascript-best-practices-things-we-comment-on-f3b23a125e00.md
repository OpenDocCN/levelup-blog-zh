JavaScript 是一种非常宽容的语言。编写可以运行但有错误的代码很容易。

在本文中，我们将研究注释代码的最佳实践，包括我们可以注释的内容。

# 注释代码段落

代码段可能会收到注释来澄清它们。

任何澄清它们并且不重复代码的东西都是好的。

# 在代码意图的层次上编写注释

代码意图的同层注释是好的。

因此，无论我们注释什么，都应该在我们解释的代码之前。

# 将文档工作集中在代码本身

我们应该一直编写自文档化的代码。

这意味着变量名和常量名必须有意义。

同样，函数、类、对象和属性的名称也需要是描述性的。

例如，我们可以编写以下内容:

```
let numApples = 1;
```

现在我们知道上面的变量存储了苹果的数量。

我们可以对其他需要命名的东西做同样的事情。

# 将段落评论的重点放在为什么而不是如何上

我们应该在评论中关注我们为什么做某事，而不是如何做。

代码已经解释了我们如何做某事，所以我们不需要在注释中重复。

# 使用注释让读者为接下来的内容做好准备

我们可能还想告诉读者在我们的代码中会发生什么。

这样，他们可以阅读注释，知道代码后发生了什么。

# 让每条评论都有价值

评论应该是有用的。否则，它们应该被写入。

我们应该把更多的精力放在使代码可读上，而不是写更多的注释。

# 记录惊喜

如果我们通过查看代码发现一些不明显的东西，那么我们应该在注释中记录它们。

这样，从事代码工作的开发人员就不会落入陷阱。

# 避免缩写

不是每个人都理解缩写，所以我们不应该使用它们。

只能使用最常见的方法。

# 区分主要意见和次要意见

我们也可以定义不同级别的注释。

为了做到这一点，我们可能想用不同的符号。

例如，我们可以写:

```
// check strings that are to be deleted
// ... determine number of strings in the table
```

在上面的代码中，我们将第一行定义为主要注释，第二行定义为次要注释，用符号`...`表示。

# 注释任何避开错误的内容

我们可能想对我们在代码中发现的一个变通方法进行评论。

同样，我们可以对语言或环境中未记录的特性做同样的事情。

然而，周围不应该有太多这样的东西。

# 证明违反良好的编程风格是正当的

如果我们不得不违背一个好的编程风格，那么我们必须证明它是正当的。

这会让每个人都知道这不是马虎。

# 不要评论复杂的代码，而是重写它

我们不应该写注释来解释复杂的代码。

如果我们可以通过重写使它们更清晰，那么我们应该这样做。

通过这种方式，任何从事代码工作的人都会感谢我们让他们的生活变得更加轻松。

# 注释数字数据的单位

如果数字数据的单位不完全清楚，那么我们应该通过对它们进行注释来使它们清楚。

我们也可以将单元放在变量名中，而不是添加注释，这可能更好，因为我们不必添加注释。

# 注释允许的数值范围

可以通过注释来限制允许值。

然而，我们真的应该在代码中这样做，这样我们就不会遇到处理无效数据的问题。

# 注释编码含义

由于 JavaScript 没有枚举类型，我们可能希望使用注释来解释编码的含义。

我们可以用注释来解释每个值的含义。

# 输入数据的注释限制

如果输入数据有限制，那么我们可能希望在评论中解释它们。

但是，我们也应该在代码中处理它们，以便确保输入数据符合限制。

# 结论

我们可能想要评论很多事情，比如输入限制和证明违反良好实践的合理性。

段落注释应该用来解释为什么，而不是我们如何做事。
