# 为什么自动化让你作为软件工程师更有效率

> 原文：<https://levelup.gitconnected.com/why-automation-makes-you-more-effective-as-a-software-engineer-9a44c1e6b664>

作为一名软件工程师，你想在工作中或者在你的个人项目上取得更大的成就吗？将某些任务自动化是完成更多任务的最好方法之一，而不必比现在支付更多的人或工作更长的时间！

> **自动化** —减少流程中人为干预的技术。通过预先确定决策标准、子流程关系和相关操作，并在机器中体现这些预先确定，减少了人工干预。

有许多行业和研究领域都包含自动化。汽车制造、软件、医疗保健，不胜枚举。作为一名软件工程师，我一直在寻找自动完成任务的方法，但是有一些事情需要记住。让我们来谈谈您应该利用自动化来完成重复性或手动任务的一些好处和原因，这可能会改变游戏规则。

# **您可以完成更多工作**

毫无疑问，如果你将部分工作自动化，那么你将能够在一天、一周、一月或一年内完成更多的工作。将工作移交给虚拟机或 Docker 容器或其他自动化工具将使您能够进行多任务处理，并在许多不同的上下文和线程中加倍努力。

> 曾经多次运行同一个命令，或者您是否发现自己在一周内运行了一次或多次命令？

这可能是一个自动化的任务，它会向您发送结果或生成输出，一旦准备好就可以使用，而不是需要在终端或其他系统中看管命令。

自动化这些平凡而重复的任务的另一个好处是，你将有更多的时间处理工作中有趣的问题。不需要运行该查询来生成报告，现在您可以获得报告并深入研究数字或调查生产中的资源利用率，以便在问题或潜在错误变成更大的问题或停机之前发现它们。腾出你的时间来学习或发现对你和你的公司来说都是非常有益的。

# 你可以更好地利用你的时间

与前一点一样，做更多的事情，还有一个方面是更好地利用你的时间。如果您是一名高级工程师，那么花时间指导其他初级工程师或更广泛地研究架构选择，是比每天为您的产品或 C 级利益相关者运行一些 SQL 查询更好的提供价值的方式。

随时准备帮助团队中的其他人会有你可能都没有意识到的好处。一天中多次切换上下文是令人疲惫的，你越能限制手动任务的一次性中断，你就有越多的时间停留在更困难的技术问题上。

# **别人可以从你的努力中获益**

不仅你和你未来的自己会对自动化重复任务感到高兴，你的同事或团队成员也会欣赏这一点。拥有一个可以被引用的脚本将允许每个人在此基础上进行构建，并鼓励自动化文化。如果做得好，所有这些都可以作为构建模块，以便完成越来越大的任务(想想 Unix 设计的[原则](https://paulvanderlaken.com/2019/09/17/17-principles-of-unix-software-design/))。

你喜欢度假，对吗？好吧，如果你是唯一一个执行任务的人，而且只是在你的脑海里，那么你就是在破坏自己去度假。将它自动化，这样当你在海滩上闲逛时，其他人也可以使用它，而不必考虑“我运行了星期五的月末报告作业吗？”。他们将能够运行您的脚本，并能够添加到它，如果需要为未来的请求。

# 风险

自动化很棒！它是世界上所有钉子的锤子，对吗？！？嗯，不是*确切的说是*。

> 能力越大，责任越大

我们概述了一些明显的好处，但在某些情况下，更多的自动化并不是答案。

**镀金你的自动化**

许多技术人员陷入了过度设计或制造系统*可以使用的新方法*的陷阱，然后在遇到之前实施解决方案。必须每周生成一份报告？如果销售人员也要求生成条形图，该怎么办？最好将它构建到脚本中…对您正在解决的用例有一个清晰的定义，并且不要走得太远，这是开始自动化的最佳方式，所以要小心范围蔓延和为您的自动化创造新的需求。

**自动化不重要或一次性的任务**

当开始考虑自动化一项任务时，考虑它是否会再次被执行，是否仅仅做这项任务比为它写一个脚本要少得多。我们正试图*增加*我们必须奉献给其他计划和团队成员的时间，所以摆弄一个不会再次运行的脚本并不能让我们达到那个目标。

**过度自动化(有时有一些人的参与还是好的)**

将一些事情自动化是很诱人的，这样你就不必执行*任何*动作。有时这是可以做到的，也是有意义的，但其他时候最好还是包括一些人类互动，以保持每个人的大脑都参与进来，并了解在出现错误的情况下该做什么。

> 眼不见；心不想

考虑一下你是想让这个过程变得更好，还是觉得它很平凡，需要顺其自然。如果你想继续做得更好，那么自动完成一项没有人参与的任务会让你的团队和公司很快忘记它。

**错误发生时没有警报或监视器通知**

一旦任务被自动化，如果它运行时没有您定期检查的输出，那么您应该投资于监控或警报。系统故障和 cron 作业过早退出的频率很高，以至于您的自动化没有确定性。找出一个失败的任务可以让你快速知道解决问题的最佳方式，而不会有太多的噪音。

**利用技术解决人员问题(避免与其他部门互动)**

添加更多的技术来解决一个通信问题或一个有问题的软件产品不是答案。直面问题，进行艰难的对话，而不是拖延不可避免的事情，而不是自动避免冲突。

总之，寻找机会通过一些实用的自动化来赢得一些时间，但要注意概述的风险，否则你会发现自己工作量增加了一倍，却没有任何好处。
