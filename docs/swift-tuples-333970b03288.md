# 什么是元组？

## 定义

元组表示为单个复合值，是一组由逗号分隔的多个值。对于元组中使用的值的数量和值的类型没有限制，这为我们提供了使用各种数据类型(例如，整数和字符串)及其排列(例如，整数数组)创建任何长度的元组的**灵活性**。这里有几个展示这种灵活性的例子。

由于 Swift 支持类型推断，所以在后面使用这些元组的时候，你可以找出这些元组的具体类型，就像上面评论中提到的那样。有趣的是，一个元组可以包含在另一个元组中。

# 使用元组的好处

## 简单申报

元组可以很容易地声明。不像`struct`和`class`，你不需要在声明一个元组之前写任何特别的东西。每当你需要使用它的时候，就在那里声明它。不需要额外的管理费用。

此外，有几种方便的方法来访问元组中的单个元素，这使得使用起来非常愉快。

## 元素的灵活性

元组的灵活性在其定义中讨论。对于给定的元组，可以定义任意多的元素。此外，它在元素的数据类型方面也非常灵活，比如整数、字符串、整数数组，甚至元组本身。

由于元组是其元素的复合值，其灵活性使其成为理想的值容器，特别适合于本地使用(例如，在函数中)。

## 易变性

元组是值类型数据。因此，当它们被声明为变量(即，当值可以改变时)而不是常量(即，当值不能改变时)时，我们可以改变元组中的一些或所有元素，只要类型保持不变。下面的代码片段提供了一个简单的例子。

# 使用元组的缺点

## 有限的“可扩展性”

元组本身并不是真正的标准数据类型。它们对于简单的相关值组非常有用。然而，根据设计，当涉及复杂的数据结构和函数时，它们不能被使用。

相反，您应该认为元组是一个没有任何实现细节的`struct`。它只是提供了一种方便的方法来将相关的值组合成一个变量，并在适用时单独访问它的元素。它实际上不允许您向元组添加函数，因此它的“可伸缩性”是有限的。

## 不符合可散列协议

元组的另一个主要缺点是它不符合[散列](https://developer.apple.com/documentation/swift/hashable)协议(在这里了解更多)。这意味着元组不能是字典中的键或集合中的元素，因为`Dictionary`的键和`Set`中的元素应该符合 Hashable 协议。下面的代码片段向您展示了相关信息。

# 结论

元组作为包含一组相关值的紧凑变量非常有用。它可以提供比函数返回的单个值更多的信息。但是，在需要额外的数据结构和功能的情况下，它有一些局限性，限制了它们的使用。然而，它是一个方便的值容器，具有简单的构造、灵活性和可变性，因此在适用的地方鼓励使用它。
