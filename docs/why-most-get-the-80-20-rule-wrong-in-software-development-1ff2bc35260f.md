# 为什么大多数人在软件开发中误解了 80/20 法则

> 原文：<https://levelup.gitconnected.com/why-most-get-the-80-20-rule-wrong-in-software-development-1ff2bc35260f>

## 如何应用帕累托原理！

![](img/8dcee8ac60e3e8bf4d31d00ce605320c.png)

马文·迈耶在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

这一切都始于意大利，当时维尔弗雷多·帕累托观察到意大利大约 80%的土地被 20%的人口所拥有。他提出了一个广为人知的原则，即[80-20 原则](https://en.wikipedia.org/wiki/Pareto_principle)，该原则指出，大约 80%的后果来自 20%的原因。

> 大多数结果(80%)来自少数原因(20%)。

各个行业对这一规则有自己的解释:

*   在商业中，20%的投入产生了 80%的产出。
*   在销售中，20%的客户贡献了 80%的销售额。

这个原理表明了投入和产出之间，努力和结果之间的不平衡。

在本文中，我们将讨论如何将 80-20 法则应用到软件开发的各个方面。

# 斯坦迪什集团“混乱”报告 2002

一般的经验法则是“80%的用户使用 20%的功能”。这份报告超越了这种说法，深入挖掘。

Standish Group 2002 年“混沌”报告展示了软件功能的使用:

*   7%的功能总是被使用。
*   13%经常使用。
*   有时候是 16%。
*   19%很少使用
*   45%的功能从未使用过。

这表明团队是如何浪费大量时间来开发用户不需要/不想要的功能的。大多数产品经理都知道这条规则，但他们仍然会犯这个错误。分析没有得到足够的重视，导致产品开发偏离其主要用例。

# 最有价值球员

最小可行产品是我们大多数人都误解的概念。我见过许多产品经理，他们想在 MVP 本身中增加很多特性。

MVP 应该解决最重要的用例，并且很好地解决它们。

> 大多数主要问题(~80%)应该在 MVP (~20%)中解决。

正如帕累托原则所说，大多数结果来自少数原因，MVP 是覆盖 80%用例的原因。

# UI/UX

> 20%的视图是 80%最常用的用例。

适用于 MVP 的规则在这里同样适用。开发人员工作并花费大量时间的大多数视图都没有被用户访问或很少被用户访问。

创建一个好的用户界面和 UX 可能会让你兴奋，但最重要的是你需要问的首要问题是"*这个视图是否在主用例的路径上？*”。

# 发展中

> 80%最广泛使用的特性是用 20%的努力开发出来的(代码行)

因为我是一个开发者，我可以证明这一点。因为我是一个开发者，我可以证明这一点。我大部分时间(80%)都花在了琐碎的任务上。所有的附加功能、极限情况、由于这些极限情况而产生的错误以及许多这样的任务花费了大部分时间来开发，但是不属于最广泛使用的功能。

# 质量保证

QA 的表现是通过他/她在一个版本中发现的问题/bug 的数量来判断的。这是业内最不合法的习俗之一。

> 80%的用例只有 20%的 bug。

QA 发现的大多数问题都存在于很少/从未使用的 20%的用例中。开发人员花费时间来解决这些问题，但并没有带来太大的改善。

一个有效的项目经理可以根据对用户体验的影响而不是对功能的影响来区分这些 bug 的优先级。在 bug 清理期间记住这一点有助于纠正开发路径。

# 敏捷

敏捷方法的思想是尽快交付产品/功能，以获得反馈并相应地纠正执行过程。团队通常开始关注那些不属于最广泛使用的特性的特性。

即使在敏捷执行中，帕累托原则也是正确的。在这个过程中，人们希望开发新功能，然后在不知道影响的情况下不断改进这些功能。

为了避免这种情况，应该在每个 sprint 开始时对所有特性的使用情况进行分析。

# 包扎

在软件开发的每个方面，帕累托原则在很大程度上都是正确的。

这个原则听起来像是常识。但我从未在实践中见过。我试图与产品经理和团队成员争论。我只成功过几次。

专注于接受努力和原因之间的不平衡，根据任务的影响对任务进行优先排序，可以帮助团队表现得更好，取得更多的产出。有时候，我们需要退后一步，看看更大的图景。
