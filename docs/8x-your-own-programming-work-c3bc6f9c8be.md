似乎我们总是顺便提到“10 倍程序员”这件事——但最大的影响往往发生在中间。

所以今天，我想谈谈一些简单的步骤和习惯，任何程序员都可以实现这些步骤和习惯来增加我们工作的价值——这些都在我们自己的手中，每次我们写新代码或编辑旧代码的时候。

# 确保其他人能够理解并使用它

在大多数较大的公司中，会有适当的规则来处理基本的编码准则，并确保每个人都在同一页面上——但这仍然为恶意的遵从或实际上有帮助和有成效的协作留下了大量空间。

我曾经处理过在技术层面上简单的代码，但是在实践和逻辑层面上却无法理解。

当其他人可以检查你的代码并调试它时，你为他们节省了一个小时，也为你自己节省了一个小时，公司会很高兴重要的事情不会因为除了你之外没有人能够修复它而中断。人为的工作保障是一个笑话，但当你实际做的时候就不好笑了。

# 确保其他人根本不需要使用它

易于调试的代码很好，根本不用调试更好——这不足为奇。

如果您保持较小的更改，编写适当的测试，并且可能在代码投入生产后进行一轮手动测试——您的一两个小时将会得到很好的投资，对每个人都有好处。

# 让你的代码自动说话

天啊，记录输出，这是我生活中的秘密爱好。如果您曾经调试过这样一个系统，它会弹出一个类似“发生了错误”的提示，或者甚至没有提示，只是完全停止工作——那么您就会知道在漫长的一天工作后，日志文件会感觉像是一个热水澡。

日志消息很棒，万一有东西坏了，自动提醒也很棒——非常有用。

我曾经在没有人知道错误发生的系统上工作过，直到有一天我们查询了系统背后的数据库，发现了逻辑上不可能存在的数据星座。

自动化警报和监控报告的最大好处可能是，在最好的情况下，您甚至不必接触代码。如果您的电子邮件收件箱中有类似“无效输入数据”的内容，那么您可以直接找到源，并找出为什么链中的前一个系统将不正确的数据写入源表。这可以为你节省*几个小时*的调试时间，即使系统并没有出错——所以写那些该死的日志消息和错误报告吧。

如果你觉得很有趣，那么编写一个工具来生成每日监控可以节省你大量的时间，否则你就要花费大量的时间来手工检查单个的错误报告。

# 自动化你的自动化

两个字:建管道。

还有两个词:预提交钩子。

这两者在几乎所有情况下都是很好的工具，只有很少的缺点。在现代公司中，您可能会发现构建管道具有自动化的工作流，用于构建、测试和将您的代码部署到开发/测试/生产环境中——但是许多公司仍然停留在没有这些救命稻草的世界中。

但是，这并不意味着您不能自己使用它们，在某些情况下，一个简单的 PowerShell 脚本与预提交挂钩可以在解决方案层面上实现这一点。想想类似“在提交之前，构建解决方案并运行测试，只有在一切顺利的情况下才提交变更”。

但理想情况下，你可以用合适的管道工具，比如 Azure DevOps 之类的工具，做合适的工作。

# 让事情可以从外部配置

与上面的日志消息一样:当您不必接触实际代码时，它节省了大量时间。这里有一个很好的例子:

*   我们曾经有过这样的服务器，在那里，运行整个管道并把更改投入生产需要大约两到三个小时。在一个经常设定或打破最后期限的系统中，等待关键变更上线的时间太长了。
*   但是由于环境变量，我们可以在服务器本身的简单配置文件中维护重要的配置。
*   这意味着对于一半的更改，我们可以在构建管道中进行更改，但甚至不必运行它，因为我们可以登录到服务器，在配置文件中复制该更改，然后用它重新启动服务器。
*   下一次构建管道正常运行时，它会自动推出更改。

这可能不是最干净的工作流程，但是它为我们节省了大量的时间，因为我们在实际的项目工作中来回地做着改变。

# 学会有效地帮助他人

我不知道你怎么想，但是那些因为你问问题而对你大喊大叫的人是我最讨厌的。尽管我喜欢尝试，但我就是无法让自己享受这种感觉，这在我们的科技世界里是一种流行病。

我真的很喜欢这句名言“除非你能用简单的术语解释，否则你不擅长某事”——这是我的座右铭。

我已经浪费了无数的时间给那些知道我的问题的答案但拒绝帮助我的人，或者给了我一半的答案，完全是为了让我因为没有立即知道另一半而显得愚蠢。这很烦人，我甚至不再为此生气，并试图找到其他知道如何进行实际团队合作的人。不用说:当人们开始像另一个障碍一样在你周围工作时，你没有尽最大努力。

# 学会与非技术人员交流

在前一点的基础上:有些人真的不知道如何与非 it 人员谈论技术。

在这两个世界之间做一名翻译实际上是一项超级有用的技能，可以让偏离轨道的会议回到正轨，并带来实际的进展。

你可以很容易地将不重要的细节抽象出来，把注意力集中在真正重要的部分，然后更详细地解释它们。这里有一个简单的“黑客”:普通人认为他们知道服务器是什么，所以你可以很容易地说类似“如果我们这样做，那么服务器就会有问题，事情就不会工作”。每个人都明白这一点，甚至不需要了解 Oracle 怪癖的复杂性。

# 确保你的假期有保障

我有很多关于人们去度假的恐怖故事，当他们在塞舌尔群岛度假时，事情发生了惊人的失败。

我很幸运，通常是接受的那一方——因此，我可以告诉你一些我度过难关，尽我所能的有趣故事。有一次，当一个重要的系统发生故障时，我排在第三位，原来的所有者正在某个偏僻的地方休假，他的休假替代者正在休病假。我对管理该系统中的一些配置几乎一无所知，现在突然之间，我们不得不修复一个没有可用错误消息的完全中断。三天零一个周末后，我们终于成功了，这很有趣——但如果有适当的假期准备，这可能根本不是问题。

# 要点:简单的步骤可以给你的工作结果带来巨大的变化

我希望这篇文章能有助于展示一些简单直接的习惯和方式是如何极大地提高你工作的影响力的——即使没有增加实际的产出。
