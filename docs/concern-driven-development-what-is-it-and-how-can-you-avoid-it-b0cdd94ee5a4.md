# 关注驱动开发:什么是关注驱动开发？如何避免关注驱动开发

> 原文：<https://levelup.gitconnected.com/concern-driven-development-what-is-it-and-how-can-you-avoid-it-b0cdd94ee5a4>

## 不要让 CDD 阻止你的团队取得成功

![](img/9c12de2964cd5a19d36e6ffbb345fc88.png)

信用:Canva

软件工程师在整个开发周期中与许多未知因素一起工作。所以，有顾虑是开发人员日常工作中不可避免的一部分。

当担忧伴随着克服它们的解决方案时，它们是有益的。它能抚平别人可能看不到的皱纹。这也有助于思考可能的解决方案并选择最佳方案。

然而，如果你总是有没有解决方案的顾虑，或者你的顾虑的唯一解决方案是不做工作以避免失败，那么你正在遵循关注驱动开发(CDD)方法。

我已经看到个人和团队成为关注驱动开发的牺牲品，而没有意识到他们正在这样做。

当团队遵循 CDD 的方法时，他们会尽力延迟或停止进度。他们问了许多“如果……会怎么样”的问题来延长讨论。他们不是查看解决方案或寻求帮助，而是一个接一个地表达关切。

如果团队被要求评估工作，他们会提供异常高的数字。他们预计需求会失去成本效益分析，并由于过高的估计而被降低优先级。而且，如果它仍然获得批准，由于臃肿的数字，他们有很多悠闲的时间。

## 关注驱动开发方法的缺点

CDD 可以减缓团队内部的创新。创新是最不确定的领域。周围有很多未知是正常的。如果开发人员对未知感到不安，不知道如何驾驭它们，他们会感到失落。最终，他们会选择远离任何创新工作。

遵循 CDD 的另一个巨大的缺点是，它很容易挫伤热情和积极的团队成员的积极性。这样的开发人员总是准备卷起袖子，专注于解决方案而不是问题。然而，团队内部持续的摩擦和来自他人的阻力会浪费他们的创造力，让他们失去动力。

众所周知，CDD 方法也会影响工作文化。如果允许团队继续使用这种方法，将直接影响他们的长期绩效。团队不会有重大的交付，因为他们会很乐意通过一连串的关注来推动。CDD 没有鼓励高绩效，而是导致了逃避工作。

CDD 也能创造一个充满敌意的工作环境。它鼓励人们提出问题，而不必提供替代解决方案。它在团队成员中制造冲突。个人专注于强调为什么其他人的想法可能会失败，而不是合作交付高质量的产品。

这样的心态极具破坏性，会扼杀团队的生产力。

抱怨很容易。不用提供可能的解决方案，继续提出问题更容易。当团队没有对缓慢的进展或保持他们的产品为未来做好准备负责时，他们很快就会陷入关注驱动开发的陷阱。

## 领导者可以做些什么来避免关注驱动的发展？

![](img/9e9752cf118e63a79c73bc65a10d938e.png)

信用:Canva

领导人必须对 CDD 的做法保持谨慎。他们应该花时间了解是什么导致团队有如此多的顾虑。

也许开发人员没有得到适当的支持，他们不相信他们的领导会在失败时袖手旁观他们，或者工程师不能胜任这项工作。

无论如何，领导者必须为团队定下基调，鼓励开发人员关注解决方案，而不是问题。一个有效的方法是奖励积极主动的行为和创造力。这可以激励那些准备卷起袖子的人。同时，它将成为团队中其他人的榜样。

领导者还必须确保团队有一个清晰的愿景。他们应该持续跟踪目标的进展。这将促使团队专注于重要的事情。这将促使他们不断进步，而不是陷入成千上万的担忧之中。

营造一个让开发人员感到安全的环境，可以大大避免顾虑。这可以鼓励团队承担风险，而不用担心后果。当团队不害怕失败时，他们将更有可能专注于寻找解决方案，而不是隐藏在担忧背后。

最后但同样重要的是，领导者应该为他们的团队提供适当的培训和指导。他们必须确保开发人员能够胜任这项工作，并具备必要的技能。有了适当的支持和指导，团队可以在未知的情况下航行，而不会感到迷失。

# 最后的想法

谨慎使用时，担忧是好的。然而，团队或开发人员使用关注作为不做工作的借口会严重阻碍交付。领导者需要对它保持警惕，并采取必要的步骤使他们的团队远离关注驱动的开发方法。
