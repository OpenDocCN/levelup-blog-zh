## 它们到底是什么意思

软件工程师的生活是建立在他们的代码运行的基础上的。他们引以为豪的事实是，他们的代码正在生产中运行，没有任何错误或异常。但如果不是呢。

> 如果他们的代码第一次在 prod 中运行就出错了怎么办？
> 
> 如果他们的代码导致堆问题怎么办？
> 
> 如果他们的演示没有任何深度怎么办？

他们撒谎。是的，他们撒谎。

他们会为了保全面子而撒谎。软件工程师为了摆脱这种棘手的局面，会说一套标准的谎言。

# 它以前是有效的

嗯！从来都不是。这部分代码从未经过测试。希望在跨团队参与会议上不要出现错误。但是它出现了。我暴露了。

现在所有人都盯着我，想知道为什么会引入这么简单的 bug。每个人都很惊讶为什么这段代码没有被正确测试。

唯一能挽回面子的方法就是代码之前能正常运行，现在不行了。

所以，是的，

> “它以前是工作的”

这有望缓解压力。

# 我想我会完成代码，然后添加注释

没有机会。我讨厌评论。我不得不写完整的代码，然后添加注释，向一些笨蛋解释这段代码做了什么。为什么？仅仅因为一些书上说它们是好的软件工程原理。哼！

我会躲过这颗子弹的。等我写完这段代码，下一个任务就会分配给我。没人会记得评论部分。

*几个月后，我开始了新项目*:这到底是什么代码？没有适当的注释，怎么会有人写出这样的代码呢？我如何理解这是做什么的？

如此循环往复。

# 这不是我的责任

> 这是别人开发的
> 
> 这不是软件问题
> 
> 这段代码是另一个团队的问题

但是在内心深处，我知道这是我的责任。但是现在，我没有时间去理解和修复代码。而且代码本身太复杂。

我该如何承担责任？我怎么能说我不懂代码呢？这是软弱的表现，对吗？

# 我不知道这个功能

我确实知道这件事。第一天我就在我的追踪清单上看到了它。

但我不知道如何在本周内完成这项工作。

现在经理们向我询问任务的最新进展。我有两个选择。要么说，根据我的能力水平，这项工作需要时间。或者只是说

> “我不知道这个功能”

这个星球上的每个工程师都会选择第二种方案。我也会。

# 我不擅长做报告

也许我不是，也许我是。但是我讨厌这样做。我讨厌画方框和圆圈。我想编码就是这样。让他们经理去做吧。当我成为其中一员时，我会开始这样做。在那之前，我只想说

> “我不擅长这个”

# 我会让 EOD 完成的

不会吧！EOD 不可能做到这一点。这至少需要一周时间。他一定是在开玩笑。我也会和他开玩笑。我会说它将由 EOD 完成，并在未来几天休假。

这会提醒他永远不要和我开玩笑，永远不要。

# 这是一个临时的解决方案

在这一点上，我没有更好的永久解决办法。这是我能做的最好的。如果他们找不到人来设计更好的解决方案，这个临时解决方案将在生产中运行一段时间。

# **这不是 bug 用户正在做错事**

任何人身伤害律师都会劝告人们 ***永远不要在任何类型的车祸后*** 认罪。相反，他们会要求责备对方。

每个软件工程师都发过类似的誓言——绝不承担责任。

如果代码在 prod 中失败，那是最终用户的问题。他们可能在进行 API 调用时犯了一个错误。

如果特性没有被正确开发，那是因为经理没有解释需求。

这总是他们的错。

一个小谎言不会伤害任何人。所以，要创新，继续撒谎。让我们想出新的酷方法来隐藏我们的不足。当代码崩溃时，让我们责怪其他人。让我们错过时间表，并为此指责政府。

毕竟，如果你不说谎，你就不是一个好的工程师…对吗？
