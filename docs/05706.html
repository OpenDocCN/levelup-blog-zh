#2 —找到问题的根本原因

编写干净的代码是非常困难的，如果不是作为一名程序员最困难的部分。这是一项需要经验的技能。为了掌握这项技能，你真的需要很好地理解代码是如何工作的以及如何维护的。

在这篇文章中，我将通过给你5个非常简单的提示来帮助你开始写更干净的代码，你可以马上开始应用。您应该专注于编写更干净的代码的原因非常简单:干净的代码有助于我们最大限度地减少阅读和试图理解代码所花费的时间。我们都知道难以阅读的混乱代码的后果。

1.吻

简单点，笨蛋。如果你希望你的代码是干净的，这就是你应该如何编码。因为我们花了大部分时间阅读代码而不是编写代码，所以我们应该尽可能地保持代码简单。这不仅让其他开发者更容易阅读和理解你的代码，也是为了你未来的自己。

虽然这个原则看起来很简单，但是执行起来非常困难。为了保持代码简单，您应该尽可能降低复杂性。这意味着越简单越好。然而，你应该永远不要为了让你的代码看起来更简单而牺牲可读性或可维护性。

适用于KISS原则的代码被认为是简单明了的，甚至可能是愚蠢的代码。然而，编写简单的代码是开发人员在编程方面最重要的素质之一。

有很多事情可以让你的代码尽可能简单。真正突出的一点是，您应该尽量避免抽象——很多时候这只会增加不必要的技术负担。

2.找到问题的根本原因

这样你只能抑制真正的错误

当你写代码的时候，最终不可避免的会遇到一些问题。您如何解决这些问题会对代码的清洁度产生很大的影响。

根据经验，你应该总是寻找问题的根源。

我见过很多出现在bug追踪器中的问题在代码中太深的层次上得到修复。让我们用一个例子来解释一下。

bug追踪器中的一个问题表明执行的查询无效。查询需要一个数值。

SELECT * FROM Products WHERE id = [numeric value]

错误的发生是因为向查询传递了一个空值，而不是一个数值。所以查询看起来像这样:

SELECT * FROM Products WHERE id = null

显然，ID字段不可为空，所以这触发了一个错误。通过将传递给查询的变量类型转换为整数，bug得到了“修复”。这使得查询的语法有效，但没有解决问题。

这个bug的“修复”并没有追踪到问题的根源，而是在一个太深的层次上实现的。修复这个bug的一个更好的方法是跟踪为什么将 null 值传递给查询并修复它。

为了找到问题的根源，你应该问很多为什么的问题。

为什么这个查询无效？因为ID为空。

为什么传递给这个查询的ID为空？因为它由另一个函数传递。

为什么这个函数传递一个空值？因为它没有从API得到有效的响应。

这种技巧被称为“五个为什么”,有助于找到问题的根本原因。

3.有意义的名字

有意义的名字，或者更确切地说是不言自明的名字，对于代码的可读性是极其重要的。例如，对于一个第一次看代码库的新开发人员来说，它们非常重要。

选择好的名字需要时间，但节省的时间比花费的时间多。花些时间找出有意义的名称，因为这将提高代码的可读性，并使其他开发人员更容易理解。

4.你需要测试

干净代码的特征之一是它是健壮的。编写测试有助于使您的代码更加健壮。尽管测试不会给你百分百保证没有错误的代码，但它可以帮助你捕捉尽可能多的错误。你应该写测试的原因有很多，但那是以后的事了。现在，你需要做的就是确保你写的测试也是干净的。

什么是干净的测试？三件事。可读性，可读性，可读性。

这是罗伯特·c·马丁的《干净的代码》一书中的一句话，很好地总结了这一点。您不仅应该编写可读的生产代码，而且您的测试代码也应该是可读的。

尽管可读性非常重要，但是你的测试也应该是快速的、独立的和可重复的。

5.一致性

如果你想有一个干净的代码库，一致性是关键。我喜欢这句名言“持续的坏好过持续的好”。尤其是在代码方面。如果你以某种方式做某事，以同样的方式做所有类似的事情。

如果你已经决定对你的变量做骆驼大小写，坚持下去。想用空格代替制表符？太好了！无论你在代码中做什么，至少要坚持做下去。

不一致的问题来自于时间摧毁软件这一不可避免的事实。一个软件存在的时间越长，使用它的人越多，混乱就越多。

如果你在一个团队中工作，你应该做的第一件事就是想出编码惯例。通过这种方式，您可以确保所有代码都以相同的方式编写。

一致性对代码库的可维护性有很大的影响。不一致的代码需要更多的时间来阅读和理解，从而使维护更加耗时和昂贵。最重要的是，以一致的方式编写的代码将加速代码审查。
