## 有时候，你需要给自己一些休息

让一个应用程序在生产中运行意味着它们正在影响其他人的生活。这是一个巨大的责任，在能够处理待命电话之前，需要很好地理解它。

通常有两种通用的方法来了解您的应用程序是否一切正常:**主动监控**——您可以查看仪表板以检查错误和问题，以及**被动监控**——通常称为警报。当系统通知你一个正在进行的问题时。

后者通常比前者好。你在技术部门工作。因此，你希望事情是自动的。此外，最好在问题开始后就知道有问题，而不是只是随便检查问题，这可能需要更长的时间，从而增加事件的影响。

您可能会认为，让系统中发生的每个问题都有一个警报，可以解决检查仪表板以检测任何问题的问题。

事实并非如此。

让最小的事情触发警报会让你[警觉疲劳](https://en.wikipedia.org/wiki/Alarm_fatigue)。根据[维基百科](https://en.wikipedia.org/wiki/Alarm_fatigue#Changing_alarm_sounds_and_parameters),**警报疲劳**或**警报疲劳**发生在一个人暴露于大量频繁的警报(警报)并因此对它们变得不敏感的时候。脱敏会导致更长的响应时间或错过重要警报。”

被警报淹没不会给你带来任何好处，无论是对你的应用还是你的健康。你不会给予提醒所需要的关注，随着时间的推移，你会感到更加疲惫。

这里有五个迹象表明你有警觉疲劳，即使你不知道这是一件事。

# 1.您忽略了重复出现的警报

警报应该是规则的例外。你不应该有一个定期触发的警报，并且不需要人工干预。这些误报意味着你应该更好地调整你的闹钟。

如果你开始有一些警报触发，你不应该做任何事情，最常见的事情是你开始忽略这些警报。当这种情况发生时，真正的警报响起只是时间问题，而你忽略了它，因为你认为这只是另一个假警报。

# 2.当弹出新的警报时，你会很生气

闹钟被触发并不是世界上最美好的经历。它应该让你担心正在发生的事情，并担心它的影响。

如果到了这一步，你不再担心，而是变得愤怒，感到一股挫败感，等待(再次)触发警报，那么你就不再关心你的应用程序了。你不希望任何问题发生。你在祈祷没有警报被触发。

# 3.当发生重大事故时，您会收到太多警报，以至于无法跟踪它们

总有一天，你会遇到一件重要的事情，需要你在这一天的大部分时间里集中注意力。想象一下，试图解决一个每 X 分钟就会触发大量不同警报的问题。

我知道这不容易。

当你不再阅读什么样的警报被触发，而只是确认它来继续寻找问题时，你就到了一个点。

最终，你会发现问题所在。但是所有的警报都关闭了吗？当您查看警报日志时，您甚至不知道哪些警报已经解决，哪些还没有解决。你需要主动检查你的系统。

# 4.你没有一天没有至少一个警报

我怎么强调都不为过:**闹铃应该是一种例外**。如果每天都有警报被触发，您要么需要检查您的警报策略并调整最常见的警报，要么需要检查是什么潜在问题触发了它们。你可能不会每天都遇到关键问题。那么，您为什么会收到警报呢？

# 5.没有解决问题的愿望

当您发现是什么破坏了您的应用程序时，您所获得的成就感是继续寻找问题并提供可能的最佳服务的最佳激励。然而，如果你开始有更多的警报在队列中，超出了你的处理能力，这将会打击你的积极性。当越来越多的警报进入分析行列时，你会感到不知所措，最终你会放弃。

失去解决问题的热情和关心你的应用程序对任何人都没有好处。

# 最后的想法

被动监控可能是了解您的应用程序何时出现需要解决的关键问题的最佳工具。它让您不必总是去查看应用仪表盘，就能知道当前是否有任何问题。

避免在最小的事情上闹铃。这不是目标。如果你不这样做，你将最终被困在你自己建造的笼子里，并开始忽视本该是最优先的事情。

不要设置一个你不愿意研究它为什么被触发的警报。一个好的经验法则是，如果你想在凌晨 4 点被叫醒，就设置一个新的闹钟。

不要把警报当作灵丹妙药。在关键情况下使用它们，包括与技术相关的情况(低 CPU、磁盘空间等。)和商业相关的。
