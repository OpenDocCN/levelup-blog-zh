JavaScript 是一种非常宽容的语言。编写可以运行但有错误的代码很容易。

在本文中，我们将研究注释控件结构和函数的最佳实践。

# 注释控制结构

控制结构之前的空间是注释的好地方。

我们可能想解释为什么我们要用循环或条件语句做一些事情。

# 在每个 if、case、loop 或语句块之前添加注释

如果我们想加入注释，我们应该把它们放在`if`、`case`、循环或块之前。

这是阐明这些结构的好地方。

# 注释每个控制结构的结尾

我们可能还想把它们放在每个控制结构的末尾。

# 将循环结束注释视为指示复杂代码的警告

我们的循环通常不需要太多解释。

如果我们觉得我们必须用注释来解释它，那么它可能太复杂了。

因此，在那些情况下我们应该简化它们。

# 注释功能

在函数之前我们不需要很长的注释。

如果我们必须为每一个函数编写代码，那么人们将不会愿意创建更多的函数，而是让现有的函数变得冗长而难以阅读。

# 让注释靠近它们描述的代码

我们应该把注释放在它们描述的代码附近，这样它们更容易被维护。

此外，如果他们接近他们所描述的代码，我们知道它实际上是在评论什么。

# 在函数的顶部用一两句话描述每个函数

一两句话足以描述一个功能。

如果我们不能用一两句话来描述我们的功能，那么它可能太长了。

除了`get` 和`set` 访问器函数，其余的可能会用到一些描述。

# 在声明参数的地方记录参数

当参数被声明时，我们可能想要描述它们。

这样，如果需要的话，我们就知道他们储存了什么。

# 对该功能的局限性进行评论

如果有任何我们必须知道的限制，那么我们应该记录下来，这样我们就知道在调用它时会遇到什么。

# 记录函数的全局效果

如果一个函数改变了全局变量，那么我们应该知道。

可能很难发现我们在用全局变量做什么，所以注释可能会有所帮助。

# 记录所用算法的来源

如果我们使用了来自其他来源的算法，那么我们可能希望将其归因于该来源，以便人们可以在该来源中查找它们。

# 使用注释来标记我们程序的各个部分

我们可能还想用注释来标记程序的某些部分，以便以后可以找到它们。

# 描述课程的设计方法

如果我们设计一个类，那么我们可能想通过在类中留下一些注释来解释我们的设计方法。

设计理念、AP[方法以及被抛弃的替代方案都可以得到解释。

# 描述限制和使用假设

我们类的任何限制和使用假设都可以用注释来解释。

错误处理、全局效应和算法使用的问题都可以得到解释。

# 对文件的注释

代码文件的顶部也是放置注释的好地方。

# 描述每个文件的用途和内容

每个文件的目的和内容也可能使用一些解释。

例如，我们可以解释为什么一个文件中有多个类。

此外，我们可能还想解释一下为什么我们要以这样的方式将代码分成文件。
