*成为伟大工程师的道路并不总是平坦的。有时候，我们无法控制的环境会阻碍我们走向成功。其他时候，我们自己的行为成为阻碍我们前进的头号敌人。*

但是在我们进入这种行为的细节之前，让我们看一个受真实的人和事启发的故事，这个故事对于任何在软件工程团队工作的人来说可能都很熟悉。

# 梅布尔的问题是

![](img/ba93ec6fba695be1ef57ba2dd715bda6.png)

由 [Kobu 机构](https://unsplash.com/@kobuagency?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

亚伯是一名代码战士。她的头发从自然的灰褐色染成了鲜红色，这可能是她暴躁脾气的最好证明。还有那双强烈的绿眼睛，当争论一个技术观点或在无尽的代码行中进行检查，挑剔面向对象和类设计的最细微的细节时，它们的严厉几乎可以穿透一切。

她的优势深深植根于 Java，尽管团队的大部分成员已经转移到基于 Javascript 框架的新项目中，但她仍然是首席架构师，几乎是仍在生产中的最古老的遗留系统的唯一守护者。这个系统在很大程度上是她的创意，自 10 年前加入公司以来，她一直积极参与其中。

没有人喜欢在它上面做 bug 修复。对代码本身来说不算多，但主要是为了避免激怒梅布尔。

另一方面，她是工程界的一名有价值的成员。她深厚的技术能力超越了遗留系统本身无可争议的知识，团队非常欣赏她的洞察力。她经常参与新项目的头脑风暴会议，并提供有价值的反馈和想法，尽管有时她会就一些琐事发生激烈的争论，如变量命名或类布局。

梅布尔喜欢争论。她与她的老板、同事和直接下属争论。她与设计师和销售人员争论。这是她展示自己对公司及其产品充满热情的方式。她是个急性子，但却是个高产的女人。

至于她自己的事业，她相当满意。报酬丰厚，而且她在过去十年里取得了很多成就，这主要归功于那个遗留系统。她唯一的遗憾是，当其他人似乎都在使用新技术和框架时，她并没有真正开始使用新技术和框架。

每当他们见面时，她大学时最好的朋友总是谈论人工智能、数据科学和渐进式网络应用，因为这是他们每天在工作中所做的事情。她觉得自己慢慢地被抛在了后面，她一辈子也不知道为什么。

毕竟，她在大学里一直是最好的。哪里出了问题？

不幸的是，Mabel 的案例在软件工程界太常见了。让我们来看看为什么一些工程师最终走到了她今天的位置，以及可以做些什么来回到通往伟大的道路上。

# 编写只有你能维护的聪明的代码

![](img/363ede77ee2aa25529129863e7fa289c.png)

马库斯·斯皮斯克在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

虽然编写只有你能维护的代码确实让你成为公司不可或缺的一员，但它也无限期地将你束缚在你所编写的框架/产品上。

这意味着当有新的项目时，你不一定会参与其中，因此你会错过获得新技术应用知识的机会。

最终，你的“聪明的”代码会变得过时，或者成为遗产，随之而来的是你的不可或缺性。那你会怎么做？

成为团队不可或缺的一个更好的方法是帮助新的工程师在他们的职业生涯中成长，既通过让他们熟悉现有的代码库，又通过提供关于编写软件艺术的建设性反馈。

# 用“以我的经验”来赢得争论

![](img/e330aaf33d7418b8ab58464c40feda9a.png)

由[尼克·麦克米伦](https://unsplash.com/@nikarthur?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 拍摄的照片

每个人都有自己的生活经历和职业经历，包括上周加入你团队的应届毕业生。当你用“以我的经验”这个短语时，你实际上是在说“闭嘴，按我的方式做”。

尽管你对眼前问题的评估很可能是正确的，但有更有建设性的方法来表达你的观点。

不要用“根据我的经验”，你可以向团队提供你不同意背后的原因。你可以用过去事件的具体例子，这样每个人都可以从你的经历中学习，而不会感到受到威胁。

此外，一起解决问题将有助于团队建立信任，并且您可能会从经验中发现您提出的解决方案中的缺陷。

# 彻底的垃圾替代品

![](img/51a423cc41907c63d2256ea89f7fa785.png)

照片由[德鲁·海斯](https://unsplash.com/@drew_hays?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

当想出一个解决方案时，感觉它是最好的是正常的。比方说，当您以一种特别聪明的方式使用一个现有的算法，或者您设法用几行代码获得了一些比预期更好的性能增益时，情况尤其如此。

太好了，你可能是对的。你可能已经设计出了完美的解决方案。

然而，您仍然应该对备选方案保持开放的心态，无论是来自您自己还是来自其他团队成员，也许是通过代码评审反馈。

善于接受并不意味着盲目地听从收到的建议。在判断给定解决方案的适用性时，你的内部评论仍然是一个重要因素。以建设性的方式与作者讨论反馈肯定会成为双方作为工程师成长的机会，即使呈现给你的替代方案在你的问题的具体环境中是次等的。

# 做个混蛋

![](img/46c39c12f0f4c8e2d7a8845fb22a34bf.png)

照片由[泰勒·尼克斯](https://unsplash.com/@jtylernix?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

虽然这可能在短时间内很有趣，特别是当有意用于喜剧效果时(想到“抑制你的热情”)，但一直做一个混蛋对团队中的任何人都没有建设性，包括你自己。

混蛋的例子包括:在代码审查系统或技术文档中留下自鸣得意/讽刺的评论，在头脑风暴会议上表现得令人讨厌，不喜欢更多没有经验的团队成员，并且通常表现得好像你是团队中最好的工程师，比其他人高一英尺。

这是最难处理的行为，因为有时你甚至可能没有意识到。

例如，当你遇到一段“愚蠢”的代码时，你本能地说出一句荒谬的话，或者当你因为“太忙”而拒绝帮助一名初级工程师时，就会发生这种情况。这是很容易犯的错误，尤其是当你有压力或者心情不好的时候。这里最好的建议是，记住我们都是人，人类有感情，有时也会犯错误。包括你自己。

你越是混蛋，人们就越不愿意和你交往。团队互动越少，你向他人学习并成为更好的工程师的机会就越少。

# 结论

今天到此为止。我希望你喜欢这篇短文，并发现它很有见地。

如果你认为自己有上述任何一种行为，会发生什么？

别担心。对问题的认识已经是解决问题的第一步。解决你自己的行为永远不会太晚，自我意识和不断努力改进会让你立刻回到正确的道路上。

此外，不要害怕寻求帮助。你的经理很可能知道你面临的挑战，表现出积极主动的意愿去应对这些挑战肯定会有利于你的职业生涯。或者，你可以找一个导师或教练来帮助你度过你的特殊情况。有时候，当你离火太近时，局外人的观点可以提供一个不太明显的视角。

无论你选择哪一个，我都希望一切顺利，你会成为一名伟大的软件工程师。记住，关键是每天不断变得更好。

在那之前，继续做好东西！
