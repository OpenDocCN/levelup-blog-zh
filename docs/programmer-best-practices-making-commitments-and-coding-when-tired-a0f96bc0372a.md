要成为一名职业程序员，我们必须做很多事情才能成为一名优秀的程序员。

在这篇文章中，我们将看看我们如何做出承诺，以及为什么我们不应该在疲劳的时候编码。

# 说是

凡事都有可能，所以不能一直说好。在很多情况下，我们会遇到软件项目的问题。

因此，我们在做出任何承诺之前都要三思。做出承诺有三个部分。

首先，我们说我们会做。第二，我们的意思是我们的方式。最后，我们实际上做了我们说过要做的事情。有些人是说到做到的。

有些人说他们是认真的，但是没有做到。

甚至有更多的人承诺了他们不想做的事情。

我们不能相信言行不一的人。

# 认识不承诺

人们说的有些事情可以识别不承诺。

任何带有“需要”的句子，比如“我们需要完成这件事”，都是没有承诺的表现。

句子中的 Hope 或 wish 也表示不承诺。

还有，“咱们”也是和其他 2 一样的东西。

一旦我们寻找这些词，我们将很容易抓住不承诺。

# 认可承诺

承诺可以通过说某件事将在一个具体的日期前完成来发现。

如果它说得如此具体，那么我们就知道它要么会被完成，要么不会。我们可以做我们完全可以控制的事情。如果它依赖于别人，那么我们不能承诺它。

然而，我们可以承诺采取其他行动来实现我们的目标。我们可以和其他人合作，通过和他们交流来帮助我们完成我们需要做的事情。

如果有些事情做不到，那么我们仍然可以承诺一些能让我们更接近目标的事情。我们可以做一些让我们更接近完成的事情。

有时候，我们就是不能在我们设定的日期完成某件事。如果我们不能实现承诺的时间表，我们必须告诉利益相关者。

如果我们遇到了意想不到的技术问题，那么我们可以就此进行沟通，并决定行动方案。

# 如何说好

使用“尝试”这个词意味着我们可能会有所成就。

为了全力以赴，我们会有一个更具体的时间表。

# 编码

编码是一项具有挑战性和令人疲惫的活动。它需要一定程度的专注和专注，这是其他学科所不具备的。这是因为我们必须同时处理许多相互竞争的事情。

我们的代码必须有效。我们必须了解我们正在解决什么问题，并了解如何解决它，我们还必须确保我们的代码是解决方案的忠实代表。

此外，我们必须确保解决方案与语言、平台、架构以及当前系统的所有其他部分保持一致。代码还必须为客户解决问题。

我们必须与客户核实，看看解决方案是否真正解决了他们需要解决的问题。代码还必须很好地适应现有的系统。

系统的刚性、脆弱性或不透明性。它还必须得到良好的管理。

代码必须是其他程序员可读的。它必须以揭示意图的方式制作。这很难做到，也是一件很难掌握的事情。

处理所有这些事情很难。它们是长时间内必要的集中和专注。在团队或组织中工作的问题和干扰使事情变得更加困难。

如果我们不能充分集中注意力，那么代码就会出错，就会有 bug。我们不应该在累的时候编码。

# 疲劳时编码

当我们累的时候，我们不应该编码。

我们写的代码不会很好。如果我们有更多的精力，我们可能不会抓住很多问题。我们可能会认为有些东西有用，但这可能是侥幸，因为我们太累了，以至于错过了一些东西。

代码质量可能会很差，因为我们不能直接思考。当我们对代码有问题时，即使它可能工作，它也会咬我们一口。
