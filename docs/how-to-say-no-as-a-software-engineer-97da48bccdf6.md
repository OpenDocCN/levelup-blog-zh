# 作为一名软件工程师，如何说“不”

> 原文：<https://levelup.gitconnected.com/how-to-say-no-as-a-software-engineer-97da48bccdf6>

## 用一个词消除问题

![](img/081fc86202e834306e271ac1d5e16300.png)

由[路](https://unsplash.com?utm_source=medium&utm_medium=referral)上[车头](https://unsplash.com/@headwayio?utm_source=medium&utm_medium=referral)拍摄

作为一名软件工程师，我学到的最难的一课就是对各种请求说“不”。我总是对各种各样的请求说“是”;"当然，我会来检查你的电脑的."“是的，我可以添加该功能。”“好的……没问题。”然后，我会和我的经理或软件开发总监发生纠纷……“你必须告诉他们‘不’。”

## 解决纷争

作为一名初级开发人员，你已经知道了足够多的知识来编写一些代码，并且可以解决一些错误。您甚至可能参与过软件部署，并经历过与部署相关的问题。像网络应用程序的 CDN 缓存没有被清除，新的 Windows 更新在安装后杀死了应用程序，或者公司的反病毒软件标记了一个 DLL，你无法完成安装。如今，我们处理的是基于 web 的应用程序，因此有太多的问题需要解决，也有太多公司或客户需要的增强功能。

当我 20 多年前刚开始的时候，这是我手头上最耗时的事情之一。我们有一个客户端-服务器桌面应用程序，我们必须从午夜开始部署到公司的所有计算机上，直到凌晨 3 点左右结束。第二天早上和接下来的几天；由于我是一名初级开发人员，不知道如何说“不”，我四处排查人们遇到的任何问题…没有一个是因为我们安装的应用程序，但每个人都认为是。

我学到的经验是，如果我只是写下人们遇到的问题，我可以把它交给网络管理员/技术支持来解决。后来，我会告诉人们给我发电子邮件来解决这个问题(这是一种说“不”的隐晦方式)，让我把它列在一个需要修复的列表上，然后把这些问题转发给技术支持。我会将这些请求密件抄送给我的老板，这样他就可以和那些部门主管讨论询问技术支持部门，而不是软件开发部门。

我只是从来没有对公司的首席执行官说不…她对很多人大喊大叫，吓到了很多人。

## 证明文件

文档将确定一个特性是做什么的，它是如何工作的，以及我们为什么要添加它。

我曾在一些地方工作过，这些地方允许将文档作为瀑布方法的一部分预先完成，在一些地方，我得到的只是一个带有一些口头解释的抽象绘图，并被告知，“构建它。”文档将是每个项目成功的关键，因为你将确切地知道构建什么，QA 将知道如何测试。

我曾经有一个项目，用 ASP 和 VBScript 重写一个旧的 MS Access DB 应用程序…是的，没错；我有点老了。我知道 Access DB 是如何工作的，因为我已经在其中构建了各种功能并创建了一些报告，所以一切都进行得很完美。当我接受国际招聘副总裁对一份报告的口头解释并建立她想要的东西时，问题出现了。我尽可能礼貌地和她来回交涉，但她一直对我很失望，在她整个部门面前大声斥责我，并要求和软件开发总监谈话。

我安排了和每个人的会面。会上她发泄了一通，我用她的解释为我的工作辩护。长话短说，事情变得非常激烈，我被告知离开会议，又过了一个小时，我的老板过来对我说:“下次，把所有的事情都写下来。”没有文件解释报告应该如何工作，国际招聘副总裁无法记录她如何从不同来源提取数据，将它们放在一起，并让它们明白什么是对的，什么是错的。

开始时的巨大教训… ***写下来*** 。如果你在理解某人想要什么或如何运作上有任何问题；和他们一起记录下来。这样，如果他们抱怨工作不像他们想象的那样，你可以参考他们和你一起工作并签署的文件。这是对口头解释说“不”的最外交方式。

## 范围蔓延

当你在构建某些功能时，有一个时间限制，因为公司希望尽快推出该功能。在开发期间，您可能会收到添加和/或更改内容的请求，这会延长开发并影响部署。每个人的时间表都会被打乱。

这是范围蔓延。开发人员试图不惜一切代价避免这一点，但总会有一个项目或领导不会让步或妥协。他们想要全部，并且希望按时完成。这是无法回避的。

你。威尔。偶遇。它。

关于这一点，最好的方法是说“不”,并希望将其放在下一个版本的待办事项中，就是按时给他们评估，并提供一些替代方案。

*   要诚实；告诉他们需要多长时间才能完成
*   提议删除某些功能以适应变化
*   要求最小可行产品(MVP)——足够让他们开心
*   可以分阶段完成吗……类似于 MVP
*   集中解决问题……更多的开发者能帮助解决这个问题吗？
*   让他们知道它是否需要一个新的库或技术堆栈，你必须了解那里有什么以及如何构建它
*   最后，提醒他们兑换美元的时间…要花多少钱？

## 最后的话

随着时间的推移，你将学会如何对各种各样的要求和需求说“不”……只是不要做一个混蛋。最终，如果你正在构建内部或外部使用的软件；利益相关者(即首席财务官、CMO 或客户)将会得到他们想要的。

了解你的局限性，设定界限，并写下来。
