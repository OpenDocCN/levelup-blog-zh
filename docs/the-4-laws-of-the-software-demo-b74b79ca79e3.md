无论您是开发人员、产品所有者、业务分析师，还是软件公司的管理人员，您都可能目睹过失败的产品演示。在演示过程中，旨在展示软件新兴功能的东西通常会停滞不前。

为什么？因为软件开发的世界受以下不可避免的法则支配:

# 法则 1——演示越有组织，出错的可能性就越大

这个队正在取得巨大的进步。因此，在下一个 sprint 计划中，团队努力安排时间来展示产品。日历邀请函已发送给产品团队、业务团队、项目管理团队、营销团队和精选的高管团队成员。

对于少数雄心勃勃的人，甚至发出了新闻稿，还邀请了媒体成员。

预发布文档和偷偷摸摸的营销视频是为大演示日(D-Day)准备的。那是这个法律生效的时候。

一首老歌:

甚至软件巨头也不能幸免。

这条定律的另一面同样适用——T4，演示越即兴，就越成功。想看看软件的运行效果如何？在开发人员的隔间里找到他，并要求他做一个快速演示。通常情况下，它可以在五分钟内演示，并按预期工作。

# 法则二——受众越重要，失败的灾难性就越大

演示可能进行得非常完美，直到首席执行官(中间名:墨菲)不请自来地走进房间。仅仅是这种高权限的存在就会立即触发软件的异常行为。软件只需要*知道*一个有权否决项目的人物会导致它打嗝。

主任一走进来，软件就会执行带有有缺陷的 WHERE 子句的查询。突然，数据库中的每个人都有了相同的帐号。

还好这只是一个测试数据库。它只是一个测试数据库，对吗？

# 法则 3——没有以下短语的演示是不完整的

*“记住，这不是最终产品”——*这个不祥的免责声明是在会议开始时给出的。它有助于缓冲即将到来的失败带来的冲击。

*“它在我的机器上有效”——*开发人员对他们的主管嘀咕着，组织者在闲聊，以填补沉闷的空气。像祈祷一样，这可以小声说，也可以大声说给任何人听。

*“几分钟前还好好的”* —和之前差不多；意图是一样的。其效果同样是徒劳的。

*“我不知道它到底为什么不起作用”* —在检查了所有设置、权限和部署策略后，愤怒的开发人员对组织者说了这句话作为警告消息。翻译:*情况不妙*。

*“我不知道它到底是怎么工作的”* —总有一个未完成的功能没有准备好演示。但是，也有人坚持要看。

“我们在最后一刻做了一些改动”——主管对等待的人群说的老一套借口。根据预期规格列表的演示已经准备好了，但是我们加入了一些没有人要求的附加功能。现在，什么都不管用了。

*“你试过关机再开机吗？”*—技术团队之外的人总是提供这种建议。他们从某人那里听说的。他们被告知这很有趣。礼貌的笑还是算笑吧？一般来说，技术团队根本不理会这种说法。

*“我们应该延期吗？”*——不是因为沮丧或恼怒而说的。通常是有人想借此机会扮演英雄，把所有人从失败的程序中拯救出来。

*“这难道不是典型的演示吗？每次都这样！”—* 建议关掉机器的人也提供了另一条智慧。

*“是的，我们当然可以这么做。这是一个简单的改变”——*当开发人员正在排除故障时，主管正在*解释*软件的变化。不可避免地，有人要求改变特性。这是为了安抚观众和挽回面子的回应。无论是改变按钮的颜色，还是从 React 切换到 Angular，这种说法总是合适的。

# 法则 4——在所有人离开后，软件将立即发挥作用

在行动项目被分发出去，日程表被重新安排以进行下一次尝试之后，技术团队呆在房间里，一心想找出问题所在。

他们通常在每个人都出去后几分钟就这么做。

向团队发送电子邮件，吹嘘发现问题的速度有多快。在这一点上没人在乎。

# 解决方案

没有演示。

只要相信开发人员会完成工作。毕竟他们比产品负责人更懂行吧？他们不需要用户故事。他们不需要用户验收测试。他们只是需要时间编码。如果你不去管他们，他们可以在两周内全部完成。

停止要求演示。
