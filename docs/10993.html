开发者，屏蔽一个PR不是侮辱！

原文：https://levelup.gitconnected.com/developers-blocking-a-pr-is-not-an-insult-87d1261a8f7e?source=collection_archive---------1-----------------------#2022-02-03

现代软件工程师的一些常识性公关礼仪。

T 这里有一个匈牙利语的表达( osztja az észt )，翻译过来就是“分享大脑”，但是一旦翻译过来，听起来就完全不像原来的样子了。它的意思是有点讽刺，指的是一个人在没有人问的情况下给出“宝贵的建议”，经常这样做，而且往往没有真正的知识，最终看起来像一个傲慢的傻瓜。

我觉得最好不要做这些事情，尤其是冒着看起来像个傲慢的傻瓜的风险，所以我在近十年的时间里远离了公关礼仪这个话题，尽管在我软件开发生涯的头三年左右时间里，我对此有着强烈的看法。然而，现在我感到自信，虽然也许不是每个人都完全同意我对这件事的看法，但我至少不是在胡说八道，有些人甚至可能会学到一些东西。

提交你的第一份个人简历既有趣又令人害怕。突然，你不再被推到主分支，提交也不全是你的了。在某种意义上，拉请求就像在软件开发中失去童贞。有史以来第一次，你不是一个人在做，但你也不知道会发生什么。当谈到一般的公关礼仪时，我总是说黄金法则是，但我打算很快写得更详细，所以如果你感兴趣的话订阅或关注。类似地，另一篇关于PR模板的文章即将发表，所以也请留意这篇文章。在这个故事中，我将只坚持三个(虽然不是真的，继续读)选项，当你提交一个评论时，为什么它如此重要。

更深的责任感

这可能因工程师而异，通常与资历和经验有关，但是总的来说，任何时候一个人决定评审一个拉请求，他们必须对他们评审的代码负责。这并不是说如果生产中出现问题，他们自己必须解决问题，而是说在各个方面，人们会立即负责防止以下情况:

  • 生产过程中破坏某些东西的代码。
  • 不必要地重新发明轮子。
  • 采用完全不同的开发风格。
  • 引入了一个巨大的安全漏洞。
  • 介绍一种新的、未经验证的语言、库、包等。

PRs有两个主要功能:预防性和共享性。上面的列表主要处理防止代码中的问题和摩擦点，在工程师、团队或自动化工具之间。

然而，分享方面也同样重要。有人可能会说，评审是质量关，但它的副作用是知识共享。首先，提交PR的人分享他们对如何在代码中解决问题的理解，这是评审者可以学习的，而他们自己，如果决定评论，将分享他们的知识和专业技能。在一个真正协作的环境中，评审对参与其中的每个人都有好处。

有不止一种方法来获得公关

不幸的是，情况并非总是如此。人们会认为采购申请既可以被批准，也可以被拒绝(被阻止)。这将是一个公平的假设，除非出于某种原因GitHub决定更灵活一点，我认为这种灵活性在工程师之间产生的敌意比围绕tab和space的讨论还要多！目前，在审核PR时，您有以下选项:

  1. 无意见批准❌
  2. 无注释的阻止(更改请求)❌
  3. ✅批准并提出意见
  4. 带注释的块✅
  5. 只是评论❌

我在这里要说的是，根据封闭和开放源代码和PRs的经验，选项1、2和5实际上应该不存在或被忽略，我将解释每个选项的缺点和选项3和4的优点，我认为，它们应该是事实上的选项。

  1. 不加评论地批准——虽然我偶尔还会发现自己这样做，但这已经是一种习惯了。简单地批准而没有任何评论让我怀疑评审者是否真的审查了我的代码，或者仅仅是因为他们懒得去做而批准了它。当我简单地批准时，我会承认我可能很匆忙，或者我批准了一个我100%信任的工程师的公关。话虽如此，我们没有人是完美的，而且很明显我还没有真正花时间从他们的工作中学习，这是一个耻辱。
  2. 无评论屏蔽 —这本身就可以被视为具有攻击性和恶意。在要求改变时不提供任何有意义的评论就像说“去你的，我不喜欢你，你的公关很糟糕，因为我是这么说的”。
  3. 通过评论批准。这些都很棒。尤其是他们带着不止【LGTM】(我看不错)。带有评论的批准显示了对编码者工作的关心和考虑。即使是那些【LGTM】或【赫克】【GEFN】也显示了评审者花时间去理解正在发生的事情，找不到任何明显的问题，即使这可能不是世界上最好的解决方案，在一段时间内，它也是可行的，并且理解了所涉及的风险和潜在的技术债务。当评论变得更加详细时，比如一点赞美或者一段代码片段，进一步改进代码，额外的上下文或者对未来重构的建议，那就更好了！这种认可实际上是在说“我不担心在生产中看到这段代码，我对你有信心，如果你有时间，这里是我对我将如何做的想法。”这样的评论充满了尊重，这是一个极好的学习机会。
  4. 带注释的块。你可能认为它们不是，但它们就像“带注释的批准”一样伟大。见鬼。甚至更好。这意味着安全网起作用了，你有不想让你失败的队友，他们抓住了你。不仅如此，他们确切地知道什么会失败以及失败的原因，这只会帮助您更快地实现正确的解决方案。屏蔽一个公关或被屏蔽相关评论，不是一种侮辱，永远不要把它当成一种侮辱。这是一个不可思议的工具，我们都应该好好利用。
  5. 随便评论。这是我感觉几乎和没有评论的block一样烂的一张。现在，当然，这是一个微妙的问题，因为它确实涉及到谁做这件事以及为什么做这件事，但是这种微妙的差别通常是不可能察觉的，尤其是在一个远程团队中工作的时候。只是评论的客观问题是，它使公关处于不稳定状态，增加了它变得陈旧的机会，并且有点说你是一个弃权的评论者。在代码审查中，我认为弃权是个坏主意。你可能真的知道某些东西会损坏，也可能不知道，如果你既不认可，就不要在决定前查看或联系工程师。麻烦在于，在较大的团队中，当更多的人开始在没有批准或要求改变的情况下留下评论时，公关将成为禁区，没有人会批准，因为评论太多了，他们不愿意承担责任。只有在你对某件事提出建议或提出问题时，评论才可能起作用，而根据回答，你肯定会回来批准或要求改变。否则，就像狗在墙上撒尿一样，就到此为止了。

引擎盖里一切都好

当你第一次看到你的公关被封的时候，我知道你会有什么感受。你可能会心悸，开始像树叶一样颤抖，并且你会一时忘记自己的名字。很好，但是不要。下次不会了。一个人不应该对阻止公关感到害羞，也不应该因为被阻止而感到压力或侮辱。这不是你，甚至不是你的代码，这是建议的解决方案，而这个解决方案不是你，它只是一个建议的解决方案，你可能有也可能没有完整的上下文。

我们所有人——在职业生涯的任何阶段——经常会写出伟大的、平庸的和糟糕的解决方案。伟大的工程师不可能孤立地成长和发展。

比起参加React课程，我从审查PRs和审查代码中学到了更多关于React的知识。真实故事！

我们也不可能一直全速奔跑，当这种情况发生时，你会感谢你的队友，他们会发现你没有发现的问题。相信我，在公共关系中获得变更请求要比在一天中的任何时候在生产中接到重大事故电话好得多！在公关阶段，你不会有5000万用户盯着你，所有管理层都期待每10分钟更新一次。在这一点上，祈祷你有一个快速恢复管道到位；否则，你会有更多的白头发。

所以，继续，点击变更请求选项，它是有原因的，只要你正确地使用它。如果你想留下评论，可以在本文的最后留下。你最好在某个地方把它从你的系统里拿出来！😆
