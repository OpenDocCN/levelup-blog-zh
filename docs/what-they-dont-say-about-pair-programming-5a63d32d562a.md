有人喜欢，有人讨厌。有些人从未尝试过，有些人每天都做。结对编程。

喜欢它的人总是谈论它对知识共享和打破团队中的知识孤岛的积极影响。他们喜欢分享这样一个事实，结对编程(PP)可以改善团队中的协作，取代甚至加快新员工入职和拉动请求等流程。 [PP 还可以与 CI/CD 和基于主干的开发很好地集成](/why-are-so-many-people-against-trunk-based-development-a785d9322584?source=friends_link&sk=45c5c7c78107e4f8acc9415e3aacd287)

不喜欢它的一组人喜欢提到它感觉更慢，尤其是当两个开发人员的经验水平不同时。他们表示，这可能会干扰注意力，而且每个人的工作方式都不一样，这使得两个人在一台电脑上工作很有挑战性。

我在两个阵营都呆过，我觉得谈论结对编程中不常被谈论的事情是很重要的。

# **结对程序有好有坏的方法**

PP 没有一个放之四海而皆准的做事方法，但肯定有一些方法可以让 PP 变得更简单，也有一些方法让它变得更有挑战性。

[使用 PP 的显式风格，如 Navigator / Driver 或 Ping Pong](https://stackify.com/pair-programming-styles/) 有助于**保持会话的组织性和结构化**。

在会话不是导航器/驱动器类型的情况下，没有键盘和鼠标的每一半导致该对的一半实际上只是一个乘客。

与团队中许多不同的成员结对并轮流完成具有挑战性的任务**扩大了知识共享的影响**并增强了整个团队的协作。

在**不相容的配对**的情况下，试图用相等的时间与团队中的每个人配对会导致压力。

使用 Pomodoro(指定的编程时间，然后是短暂的休息)这样的技术，让**经常休息**，可以在编程过程中休息并提高注意力。

允许其他同事的打扰可以让两个成员都置身于“流程”之外。

每项任务都要求结对编程，这可能会在团队中引起更多的阻力。

只有对“复杂”任务进行结对编程，才会导致大多数任务被视为“不复杂”。

# **结对编程需要礼仪**

PP 最具挑战性的部分之一是，一个人要从戴着耳机、不受干扰地独自工作，转变为直接与另一个人一起工作。

这很有挑战性，因为这实质上相当于从一个人住到和室友一起住。你最终不得不学习其他人如何工作得最好，以及什么让他们感到舒服。他们必须做同样的事情。你最终会陷入一种互相让步的局面，在这种情况下，双方都需要在各种事情上做出妥协。

虽然和室友住在一起有礼仪要求，比如自己收拾，按时交房租，但 PP 也有类似的要求:

*   两个人中的一个人必须以另一个人能应付的速度移动，这意味着一个人的动作必须有另一个人的解释和确认。
*   在配对过程中，大约 50%的时间应该用来控制鼠标和键盘，而另外 50%的时间应该由你的搭档来控制。
*   耐心……非常有耐心。
*   感同身受，相信你的同事正在用他们的技能和经验尽最大努力。
*   乐于尝试自己和同事的想法。
*   对娱乐和学习的渴望。

# **结对编程很难**

一整天戴着耳机独自工作和一整天结对编程的区别在一天结束时非常明显。这是我觉得关于 PP 说的最少的部分，有时候…好累。

结对编程很有挑战性，需要很多额外的努力，就像所有其他事情一样，这是一项技能。这不是天生的，因此感觉像是一种锻炼。

结对编程给人一种当你工作时有人在你身后监视的感觉。

当许多软件开发人员觉得他们应该能够做一些他们做不到的事情时，这可能会重新点燃冒名顶替综合症。

分享你对大部分抽象话题的想法，并把它们用其他人可以理解的具体术语表达出来，这是一项挑战，也是一项艰巨的任务。感觉像是考试。

甚至像每台计算机上不同的键绑定和不同的开发人员使用不同的快捷键或工具这样的事情也会完全破坏开发人员在他们的环境中的舒适程度。

具有讽刺意味的是，使结对编程更容易的事情之一是在一天结束时，或者在编程之间的休息时间给彼此提供反馈，以便改善双方的体验。在一个主要由刻板内向的人组成的行业，反馈交流也不容易。

简而言之，这整件事并不容易

结对编程的优势是有据可查且不可否认的。

结对编程的挑战很少被记录，但同样重要。

我郑重声明，我支持结对编程。但我和其他人一样清楚，有时随之而来的挑战会让收益变得不值得努力。但我也相信，就像任何其他具有挑战性的事情一样，当你知道自己将面临什么时，事情总是会变得更容易，因为你可以为此做好准备，甚至有可能习惯它。

如果我们更经常地谈论挑战，我们也可以利用我们的智慧创造解决方案，这样我们可以获得好处，同时减少处理挑战性的一面……例如，间歇性地结对编程，而不是每天对每个任务编程。
