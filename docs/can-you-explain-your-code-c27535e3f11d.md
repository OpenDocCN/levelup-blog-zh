# 你能解释一下你的代码吗？

> 原文：<https://levelup.gitconnected.com/can-you-explain-your-code-c27535e3f11d>

![](img/45f875bf96c7dbb0ab96034d866ce3e6.png)

程序员经常把一个工作程序误认为是一个有效的程序。他们的代码可能是阅读和调试的噩梦。可能永远都跑不完。它可能充满了命名不当的变量。没关系。只要他们的代码能够工作——也就是说，能够产生正确的结果——他们就会很高兴，然后继续前进。

这是错误的态度。

让程序工作只是编写有效解决方案的一部分。其他因素包括:清晰度、意义和性能。因此，我们需要关注有效性，而不是只关注正确性。

编写有效解决方案的一个步骤是逐行解释你的代码，并大声朗读出来。这是一个很好的方法来衡量你的代码质量和你对它的了解，以及找到使它更好的方法。

你对自己的代码了解得越多，你就能写出越有效的解决方案。如果您知道每一行代码做什么以及为什么做，那么您就可以看到替代方案并做出权衡，这最终会导致编写有效的解决方案。这就是我们作为程序员的目标:写一个有效的解决方案，而不仅仅是一个可行的方案。

# 旨在解释

看起来很容易。“我当然可以解释我的代码，”你可能会想。

我鼓励你试一试:从程序的顶部开始，大声地向自己解释每一行代码。

这项任务往往比看起来更难。

那是因为解释和描述是有区别的。当你解释某件事时，你会超越表面层次去挖掘发生了什么和为什么。

以这行代码为例:

**句子 _ 字母=集合()**

为了解释这行代码，我会说:“**句子 _ 字母**跟踪所提供句子中的唯一字母，因为句子中可能有多个相同的字母。”考虑到问题的背景，这正是*做的事情* sentence_letters 做的事情(跟踪唯一的字母)以及*为什么*这么做(句子可能包含重复的字母)。

解释不同于描述。为了描述这行代码，我会说:“创建一个空集。”但这并不能告诉我太多。布景在做什么？为什么这很重要？

这是一个说明问题的简单例子:旨在解释。原因是有的。

# 通过知识测试

首先，解释你的代码是一个知识测试。如果你在把单词放在一行代码周围有困难，那么你可能需要重新考虑一个概念。

在未来的问题中，您会看到相同的概念、方法和数据结构。因此，意识到自己的知识差距并采取措施要比让它们随着时间的推移而恶化要好。

此外，对有问题的领域做一些研究可能会帮助你改进当前的解决方案。您可能会找到使用数据结构或方法的更好方法。或者你可能会遇到新的东西，这将极大地改善你的解决方案。

这给我们带来了下一个好处:解释你的代码来改进它。

# 让它变得更好

描述你的代码是捕捉错误和发现机会使你的代码变得更好的好方法。

这是因为当你大声朗读你的代码时，你使用了另一种感觉:你的听觉。这是一件好事:你用眼睛捕捉到的和你用耳朵捕捉到的可能非常不同。

如果你读了你的代码，听起来有些笨拙或者麻烦，那么它可能是并且应该被更新。

同时，工作和重新工作你的代码会导致你发现你忽略的东西，比如命名不当的变量。

# 一项需要培养的关键技能

找出你的错误和需要改进的地方，并采取相应的措施，这是一项关键技能的一部分:自我评估。这足以解释你的代码。

别人指出你的错误是一回事。但训练你的头脑和眼睛自己发现它们并修复它们完全是另外一回事。

同时，成功也是值得注意的。找到你做得好的事情——并理解为什么——这样你就会再做一次。

在你解释完你的代码后，制作抽认卡来真正地确定概念。我使用 Anki:一个使用间隔重复的免费开源抽认卡程序。我制作抽认卡来提醒我做得好的事情(所以我再做一次)，以及我需要改进的概念。这个想法是定期重温我在解释我的代码时学到或使用的核心思想。

# 让它发生

养成习惯最简单的方法就是把你想做的新事情(比如解释你的代码)和你已经做的事情联系起来。假设在你称一个程序完成之前你做的最后一件事是最后一次运行所有的测试。所以运行测试，然后解释你的代码。

我使用[解决问题清单](https://amymhaddad.com/why-you-need-a-programming-checklist)。它包含了在我称一个问题完成之前要做的小而重要的任务。我清单上的最后一项是“解释代码”使用清单是一种简单的方法，可以确保您的解决方案是准确和完整的，然后您才能称问题为“完成”此外，它使得融入一个新习惯变得容易，比如描述你的代码。

最近经常听到人们抱怨代码的质量。解释您的代码是一个小的时间投资，可以产生更好、更高质量的代码。你应该利用的是唾手可得的果实。
