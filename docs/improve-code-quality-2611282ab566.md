**编写质量代码是一门艺术**一旦你开始编写代码，这些代码也必须是可持续的、可追踪的和可理解的。

出现的问题是如何获得可读和高质量的代码，因为有一点很清楚，好的代码不会凭空出现。作为一名开发人员，你必须有针对性地进行编程。这通常在你写代码之前就开始了，它始于对整个开发过程的良好规划。

# **强大的心态**

编写高质量代码的基本前提是正确的心态。如果可读性对你来说不重要，你就不会产生可读的代码。

这样做的原因相对简单。作为一名开发人员，您主要关心的是找到一个功能或技术问题的解决方案，而更多的是找到一个智能的方法，并将其公式化为一个算法。

这意味着**重点是写代码**，而不是可读性。但是良好的可读性是必不可少的。完成实际任务后，您最终应该检查您编写的代码，再次优化它以提高可读性。

接下来就是清理代码、使用更合适的数据结构、正确的接口等等。当然，这是复杂且耗时的，因此这些步骤只在团队中执行，这些团队意识到这些步骤对于代码的进一步开发是如何相关的。

**因为你读代码的次数比你写代码的次数多！**如果你的代码不可读，那么它就不可持续，维护起来既困难又乏味。理解不可读的代码比理解可读的代码花费更多的时间。

# 童子军规则和根本原因分析

如果现在的心态是正确的，所谓的**童子军规则**是一个很好的起点。所以总是让代码处于比你发现时更好的状态的要求。

为了使对代码的更改可持续且有意义，您首先必须注意构建对代码的良好理解。仅仅浏览或肤浅地研究是不够的，你必须理解它，你还必须理解代码内部的联系。然而，代码易于阅读是有帮助的。

不要满足于你得到的第一个答案。试着深入挖掘，把事情弄个水落石出。这就是所谓的**根本原因分析**。这不仅是改进和优化代码的基础，也是重构和调试的基础。

当然，知道如何从专业和技术上削减代码是有帮助的。这意味着如何决定哪个代码片段负责什么。

一个好的规则是遵循**单一责任原则**，处理**耦合和衔接**。这里的另一个重要方面是意图和实现的分离，这通常在开发中通过接口来完成。

当然，您应该使用从技术角度来看有意义的合适的命名，并编写有用的注释。

# 自动化质量保证

要正确管理代码，需要版本管理，比如 Git 或 Github，或者 Bitbucket。像 Github 这样的服务提供了一整套自动化流程，例如使用静态代码分析或测试。

除此之外，还要使用度量和代码覆盖率，提供文档，正确处理问题，等等。你已经为围绕代码的生态系统打下了很好的基础。

# 测试，测试，测试

有许多微妙之处，尤其是测试，从不同的测试类型开始，经历基本的过程，例如 Arrange-Act-Assert (AAA)模式，并选择合适的测试框架。

虽然单元测试仍然相对容易编码，因为它们通常只涉及单个的功能，并且与更广泛的依赖关系隔离，但是集成测试的情况就完全不同了。

因为在这里，通常还需要包括数据库和消息队列等基础设施。那么就不可避免地要使用 Docker，只需按一下按钮就能自动建立和拆除基础设施。

通过 E2E 测试来测试组件和 UI 是至高无上的。E2E 测试主要通过网络浏览器进行，并使用诸如木偶师或硒等工具。这可能非常耗时。

# 前景中的专业知识

反过来，只有当参与开发的每个人都清楚实际要开发什么，即开发背后的愿景是什么，要解决的基本技术问题是什么，以及解决方案在开始时应该是什么样子时，所有这些措施才有效。

反过来，这需要反映领域专家专业知识的用例以及明确定义的需求。

# 敏捷开发过程

然而，这只是整个开发过程的一部分。现在，这通常是敏捷的，但是敏捷方法之间有显著的差异，比如 Scrum、看板或者极限编程。

都有不同的要求，不同的目标。无论你决定哪一个开发过程，最终，仅仅包括开发是不够的。你还必须将其他角色融入团队。

如果你正在为一个开源项目编程，你也必须让社区或用户本身参与进来。接下来的问题是如何处理反馈、问题、拉式请求等。

这在理论上听起来很容易，但实际上往往不那么容易。为此，有必要找到正确的解决方案，这因应用而异。
