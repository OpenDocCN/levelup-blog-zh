要成为一名优秀的程序员，我们应该遵循一些易于采用的习惯，以保持我们的编程生涯长久。

在这篇文章中，我们将看看如何探索解决问题的方法，我们可以通过创建原型和进行评估来构建软件。

# 什么是原型？

原型是在我们构建最终解决方案时将被丢弃的代码。

它只是用来探索我们最终解决方案不同方面的解决方案。

它可以是代码，也可以是其他东西，如图纸、线框等。

我们可以跳过最终解决方案中的大部分关键内容，包括结果的正确性、代码的完整性、代码的健壮性或风格标准。

我们的目标只是制作原型来探索我们系统的某些部分，这样我们就可以使用这些发现来构建最终的解决方案。

这意味着要尽快构建，以便我们可以实施最终的解决方案。

因此，我们可以用我们最熟悉的语言和库来构建它，这样我们就可以很快得到它。

# 原型架构

许多原型在构建时都考虑了整个系统的架构。

然而，原型的任何部分都不是功能性的。

在构建解决方案原型时，我们可能需要考虑一些特定的问题:

*   主要部门的职责是否定义明确且适当？
*   主要组件之间的协作是否定义良好？
*   耦合最小化了吗？
*   我们能识别潜在的重复来源吗？
*   接口定义和约束是可接受的吗？
*   在执行过程中，每个模块都有一个访问数据的路径吗？

# 如何不使用原型

我们永远不应该使用原型作为最终解决方案的一部分。它应该是一次性代码。

此外，我们不希望利益相关者在最终解决方案方面被误导，所以我们不应该向他们展示最终解决方案的进展。

# 估计

我们得估计一下，这样才能避免意外。这样，我们在构建东西的时候就不会遇到意外。

它不一定总是正确的。然而，它确实给了我们一个粗略的想法，事情需要做多长时间，或者我们产品的性能，以及我们在构建软件时必须考虑的许多其他事情。

因此，寻找一个大概的数字比什么都不想要好。

例如，如果我们估计一项任务需要 3 天完成，那么我们就知道它大概需要 2 到 5 天。

# 估算从何而来？

估计是基于问题的模型。首先，我们必须确保我们必须理解被询问的内容。

然后我们要根据问题建立系统的模型。我们可以在脑子里快速建立一个模型。

我们可以对一项任务可能需要完成的步骤进行建模，这样我们就可以大致了解我们需要什么样的评估。

模型让我们发现表面上不明显的潜在模式和过程。

我们也可能会重新思考最初提出的问题。

模型不准确是不可避免的。然而，这对于估计来说是没问题的。

我们必须将我们的模型分解成组件，这样我们的大脑才不会被淹没。我们必须发现描述组件如何交互的规则。

总有许多参数会影响系统的输出，所以我们必须识别它们。

然后我们必须给出每个参数的值，这样我们就可以从中得到结果。

我们只需要为每种情况设置一些值，这样我们就可以从中获得一个粗略的值。

所选择的参数应该是合理的。可能是经常遇到的案例，也可能是某些人可能遇到的边缘案例。

最后，我们计算出答案，并根据这些参数给出我们的估计。

因此，如果我们被要求估计一个 100MB 的文件在 50Mbps 的连接上传输需要多长时间，那么我们可以在假设没有限制、有限制、接收不好或线路条件不好等情况下粗略地计算一下。

我们可以使用原型和评估来给我们关于如何进行项目的粗略想法，也给涉众关于项目将如何进展的想法。

原型和评估都是以不同的方式建模，最终让我们做出决策。
