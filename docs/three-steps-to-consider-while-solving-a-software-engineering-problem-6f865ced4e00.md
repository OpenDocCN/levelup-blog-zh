我写这篇博客的目的是分享我在一年的实习中获得的经验。在我实习的初期，我经常接受任务，不管是一个特性还是一个 bug，阅读给定任务的内容，然后直接投入到代码中，因为我的实现无法解决问题，或者给系统中其他已经存在的特性带来了问题。

下面是解决软件工程问题时要遵循的三个步骤。

# **1。分析**

解决软件工程问题的第一步是分析你要解决的问题。分析涉及两个部分

1.  了解问题。
2.  考虑实现。

通常，当你为一个项目工作时，如果是一个特性，你会得到一个以规范形式的任务，如果是一个 bug，你会得到一个标签。说明书将包含客户的要求，而标签将包含客户面临的问题。

要做的第一件事是阅读规范或标签，了解需要做什么。如果你是这个项目领域的新手，一开始理解所有的东西似乎有点困难。您应该做一些研究，并与团队中的领域专家交谈，以熟悉该领域。如果它是一个 bug，您也可以尝试在开发环境中重现它，以获得更好的理解。

在清楚地了解问题之后，你必须想办法解决这个问题。如果是一个 bug，你必须调试或者使用任何其他方法来检测 bug 发生在哪里。如果是一个特性，你必须检查现有的代码，并对这个新特性将如何影响系统进行可行性检查。

您可能有多种方法来解决同一个问题，在这种情况下，您将不得不考虑用一些标准来评估它们，比如客户端的易用性、代码复杂度的降低、系统的稳定性等等。在你选择了你的最佳解决方案后，和你的队友讨论一下，看看他们是否有不同意见，这是一个很好的主意。如果找不到解决问题的方法，你可以再次尝试向你的队友寻求帮助。

# **2。流程**

您应该在分析下的所有调查和可行性检查完成后开始处理或编码解决方案。

首先，您需要根据系统中遵循的代码架构来构建您的代码，同时您还必须考虑系统中是否有任何代码可以在您的解决方案中重用。此外，您还必须考虑以可重用的方式编写您认为未来实现需要的代码。

另一个重要因素是编写可读的代码。当涉及到变量和函数的命名时，你必须非常重视。变量或函数名应该定义和描述其存在的目的。如果变量名或函数名不够详细，您必须添加注释来解释它们。通常，编写干净代码的项目级标准会得到维护。

完成实现后，重新检查代码，看看是否可以进一步简化和优化。还要检查是否有任何代码级别的错误没有处理，比如空点错误。

当您准备好实现时，不要忘记运行代码库中现有的测试用例。它将验证您没有影响任何其他现有的实现。

在确认您没有影响任何其他代码之后，您必须为您的实现的所有可能场景编写测试用例。这将有助于您找出在分析时是否遗漏了任何场景，也有助于其他开发人员不中断您的实现。之后，验证您已经实现的测试用例是否通过。

最后，将您的实现推到测试环境中，并在那里进行测试。

# **3。出席**

展示你的解决方案也非常重要，这涉及到沟通技巧。不管你做了什么，如果你不能用语言表达或表达出来，那将是一种浪费。

别担心，交流会随着经验/时间的增加而改善。我从实习中学到的另一个重要经验是，沟通不在于某一种语言的流利程度，而在于演讲的内容和语境。

作为一名软件工程师，你需要与质量保证工程师交流，从用户的角度来看，你的实现是如何工作的。此外，你应该能够通过文档或直接与客户互动，向客户清楚地解释解决方案。

这是我在实习中学到的提高解决工程问题能力的三个步骤。试试看，如果有帮助，请告诉我。
