Node.js 是编写应用程序的流行运行时。这些应用程序通常是许多人使用的生产质量应用程序。为了使维护它们变得更容易，我们必须为人们设定一些准则来遵循。

在本文中，我们将研究构建和运行 Node.js 应用程序时的一些安全性和性能考虑。

# 使用子进程时要小心

我们必须小心投入。这意味着我们应该清理以防止任何外壳注入攻击。我们不希望攻击使用我们从应用程序中创建的子流程实例来运行命令。

如果我们不采取预防措施，我们可以很容易地让攻击者从我们的系统远程运行命令，通过输入恶意输入未经整理的系统命令。

# 对客户端隐藏错误详细信息

客户不应该知道我们系统的部分或任何私人数据。因此，我们应该隐藏任何容易出错的信息。

我们应该确保我们不会将整个错误对象返回给客户端，其中可能包含一些敏感的应用程序细节。敏感的东西包括文件路径、第三方模块、应用程序的内部工作流程等。，攻击者可以利用该漏洞。

# 为 NPM 或纱线配置 2FA

我们可以使用多因素认证来访问我们的私有包存储库。这样，攻击者就不能发布被他们修改过的包。

如果这些攻击未被检测到，那么我们的包可能会暴露数据或做一些我们不希望攻击者对我们的系统做的事情。

# 修改会话中间件设置

我们应该删除任何让攻击者了解我们系统的信息。其中之一是我们使用哪个 web 框架来开发我们的应用程序。这是因为攻击者可以利用这些信息来查找如何攻击我们的系统。

因此，我们应该隐藏像`X-Powered-By`响应头这样的东西，它返回 Express 中默认使用的框架。

# 通过明确设置进程崩溃的时间来避免 DOS 攻击

如果错误没有得到处理，节点流程将会崩溃。许多人建议，即使错误被处理，进程也应该退出。

如果我们让我们的应用程序在输入某些输入时崩溃，那么攻击者可以找到什么输入使我们的应用程序崩溃的模式。这并不好，因为我们不想让他们想办法让我们的系统崩溃。

任何时候我们的应用程序崩溃，一个关键的警报应该被发送，让我们知道它。

# 防止不安全的重定向

我们应该验证用户输入以防止恶意重定向，这样攻击者就无法发起网络钓鱼来窃取用户凭据。

# 避免向 NPM 登记处泄露机密

秘密永远不应该出现在 NPM 的注册表里。否则，我们增加了向公众暴露秘密的机会。我们可以使用`npmignore`文件将特定的文件或文件夹列入黑名单，或者使用`package.json`中的`files`数组作为白名单。

# 不要阻塞事件循环

因为 Node.js 是单线程的，所以我们不应该运行任何需要很长时间的同步代码。这样，我们就不会阻塞事件循环，以便运行其他代码。

每当我们有长时间运行的流程时，我们应该使用异步代码，比如 promises，这样只有当应用程序得到结果时，这些类型的代码才会运行。

否则，一个请求就可以挂起成千上万用户正在使用的应用程序。

# 比起 Lodash 等第三方工具，更喜欢本地 JS 方法

如果 JavaScript 标准库中有可用的函数，那么我们应该使用它。这样，我们需要更少的依赖，因此，我们的应用程序将运行得更快。

原生 JavaScript 函数现在比实用程序库快 50%。

# 结论

我们应该验证子进程的输入，以消除攻击者输入和注入恶意命令的风险。应该检查不安全的重定向，这样用户就不会被重定向到恶意站点。

此外，我们应该隐藏任何可能对攻击有用的信息，比如使用了什么框架、堆栈跟踪等。秘密永远不应该出现在 NPM 注册表中。

就性能而言，我们应该尽可能地使用 JavaScript 标准库来提高性能，以减少所使用的依赖项的数量。
