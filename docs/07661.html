为什么要许可您的代码

Python有一个蓬勃发展的开源社区。随着如此多的用户将Python用于数据科学、机器学习，甚至web开发，与以前相比，有更多的机会使用社区中其他成员编写的代码。有很多pip包和GitHub repos的源代码，它们可能会改善其他有类似问题需要解决的Python用户的工作流程。但是没有适当的许可，社区不能从你的源代码中获益。

当我刚开始在大学发表自己的作品时，我对授权感到措手不及。我们总是知道开放源代码是免费使用的，但我不知道许可证的类型在源代码的使用方式中起着重要的作用。除了许可的复杂性，它很少出现在我的任何计算机科学课程中。我对许可源代码了解得越多，就越觉得应该向每个新程序员传授最基本的知识。虽然许可很快会变得令人困惑，但是对于大多数新程序员来说，理解最流行的开源许可以及它们存在的原因就足够了。

定义清晰的使用指南

你可能会说你不介意别人对你的源代码做什么，这完全没问题！然而，没有许可证，其他人下载和使用你的代码在技术上是不合法的。默认情况下，任何创造性的作品，包括代码，都受到专有版权的保护。这意味着你的代码是你的，任何人使用它都是不合法的。当我们发布Python代码时，我们希望其他人能够使用、贡献或修改它，并在他们自己的工作中使用它。在本帖中，我们将讨论Python项目中最常用的五种开源许可证，以及为什么你会选择其中的一种。我们将首先讨论许可许可证，然后讨论更具限制性的选项。

担保和责任

除了明确定义修改/分发作品的规则，许可证最重要的功能之一是保护作者或贡献者免受发布开源软件时可能出现的两种风险:

  1. 被要求为最终用户提供担保。
  2. 如果最终用户在使用您的软件时遭受损失，您将承担责任。

我们将在本帖中讨论的所有许可证都非常清楚，软件是“按原样”提供的，没有任何担保。这同样适用于免除你在使用你的软件时发生的任何事情的责任。

尽管所有这些许可证都涵盖了这些基础，但考虑这些事情是很重要的，因为它们是始终拥有公共源代码许可证的重要原因。

律师:我不是律师。我只是想帮助我的Python社区成员理解基本的许可，以便社区能够繁荣发展。网上有更深入的资源，如果你有具体的顾虑，请随时咨询律师。以下内容均不被视为法律建议。

许可许可证

三种最常用的“许可”许可证是MIT、BSD和Apache 2.0。当大多数人想到“开放源代码”时，他们可能会想到与MIT和BSD许可证所概述的规则相近的规则。事实上，这些许可许可证占2018年PyPI上Python库许可证的大约64%。

我经常在我的软件中使用非常宽松的许可证。我的大部分个人项目并不打算货币化，我不介意任何人使用它们。我通常也不介意有人可能在货币化/私人化的环境中使用我的源代码，这是非常宽松的许可所允许的。

麻省理工

MIT许可证是python源代码附带的最常见的许可证。它最初是在20世纪80年代由麻省理工学院(MIT)创建的，我一直将它视为默认的开源许可。它短小精悍，便于他人使用你的作品。

在我的第一个研究职位期间，我的PI告诉我只要一直使用MIT，除非为了兼容性我不得不使用不同的东西。一些非常著名的项目带有MIT许可证，比如Node.js和Ruby on Rails。以下是麻省理工学院许可证的文本:

特此免费授予获得本软件和相关文档文件(“软件”)副本的任何人不受限制地经营本软件的权利，包括但不限于使用、复制、修改、合并、发布、分发、再许可和/或出售本软件副本的权利，并允许获得本软件的人在遵守以下条件的情况下这样做:

上述版权声明和本许可声明应包含在软件的所有副本或重要部分中。

— 开源倡议

正如我们所看到的，MIT许可证非常宽松，允许用户对你的源代码做几乎任何事情。MIT许可证允许使用、修改或不修改地重新分发，并且如果您在另一个项目中使用源代码，只需要您保留版权声明和担保免责声明。

拥有这样一个开放许可的好处是其他程序员可以尽可能容易地使用它。它与另一个程序员可能使用的任何许可都兼容，而且几乎没有任何限制或重新许可的问题需要担心。这意味着更多的人最终会使用你的代码，因为给他们带来的不便被最小化了。

加州大学伯克利分校软件(Berkeley Software Distribution)

BSD许可证有多个层级。最流行的两个Python库是BSD two-clause和BSD three-clause。

双条款

BSD双条款允许使用、修改或不修改地重新分发，并要求您保留版权声明和担保免责声明。

只要满足以下条件，允许以源代码和二进制形式(修改或不修改)重新分发和使用:

1.源代码的再分发必须保留上述版权声明、此条件列表和以下免责声明。

2.二进制形式的再分发必须在随分发提供的文档和/或其他材料中复制上述版权声明、此条件列表和以下免责声明。

— 开源倡议

虽然在麻省理工学院和BSD双条款许可证明确允许的内容方面存在一些差异，但实际差异非常小。有些人选择MIT而不是BSD 2-clause，因为MIT更简单，没有层次，不会造成混淆。BSD-2条款是2018年排在麻省理工学院之后的PyPI上与Python库相关的第二受欢迎的许可证。

不管你选择MIT还是2-clause BSD，知道它们是兼容的是很重要的。这意味着您可以使用其中任何一个，并且以后您可以将带有另一个许可证的代码合并到您的项目中。

三分句

尽管BSD three-clause不在PyPI上使用的前几个许可证中，但您可能已经注意到它被一些Python最流行的开源库使用，如numpy和pandas。增加的附加条款如下:

3.未经事先明确的书面许可，不得使用版权所有者的姓名或其贡献者的姓名来认可或推广由此软件衍生的产品。— 开源计划

您大概可以想象为什么大型库会使用这种许可证。它保持了其开放源代码的特性，但是防止了“numpy”这个名字被用来推广某个人对它的修改版本。绝大多数程序员不需要担心这个条款。

阿帕奇2.0

Apache 2.0许可证是大型编程社区中最受欢迎的许可证之一。一些非常著名的项目带有Apache 2.0许可证，如Tensorflow、Swift和Kubernetes。虽然它与BSD 3条款非常相似，包括对名称的保护，但还是有一些细微的区别:

  • 如果您使用Apache 2.0许可证对文件进行了更改，您必须在每个文件上突出注释您所做更改的信息。
  • 在所有许可许可证中，Apache 2.0许可证拥有最好的专利保护。它是这些选项中唯一默认授予专利权的选项。

Apache 2.0许可证仍然允许贡献者修改和私有化您的源代码。因此，我看到许多Python开发者选择使用MIT许可证，因为它更简单、更简短。不得不记录源代码的变更也是其他开发人员的额外负担，这可能会阻止新的贡献者。总的来说，在我看来，Apache 2.0许可证给大多数Python程序员带来的好处并不多。

版权所有许可证

Copyleft许可证是开源许可证的“限制性”系列。它们被称为copyleft，因为任何使用带有该许可证的代码的项目都必须将相同的许可证带到新作品中；许可证“遗留”给集成它的项目。

尽管我们称copyleft许可证为“限制性的”，但我喜欢把这组许可证想象成一个公园。每个人都可以免费使用，但不允许私有化，并有额外的规则来确保它们保持开放。这些许可证的设计核心是防止有人获取源代码，对其进行更改，然后发布私有版本以获取利润。虽然这听起来很棒，但是在你的代码中添加更多的限制会有一些不好的地方，我们将会讨论。

最流行的左版权许可证，通用公共许可证(GPL)，是由自由软件基金会 (FSF)开发的。FSF是由理查德·斯托尔曼创建的，他过去和现在都是真正自由软件的坚定拥护者。他有一些充满激情的名言:

一个人做的每一个决定都源于这个人的价值观和目标。人们可以有许多不同的目标和价值观；名声、利润、爱情、生存、乐趣和自由，只是一个好人可能有的一些目标。当目标是帮助别人和自己时，我们称之为理想主义。我在自由软件上的工作受到一个理想主义目标的激励:传播自由和合作。我想鼓励自由软件的传播，取代禁止合作的专有软件，从而让我们的社会变得更好。— 理查德·斯托尔曼1998年

GPLv3

通用公共许可证(GPL)是最受欢迎的限制性许可证，使其成为2018年PyPI包第三受欢迎的许可证。GPL许可的关键属性是它是“病毒式”的。这意味着，如果我用GPL许可证发布源代码，而你想把它合并到你自己的作品中，除非你也在你的代码上使用GPL许可证，否则你不能这样做。与更宽松的许可证不同，这阻止了拥有专有软件的实体使用你的作品，因为任何拥有GPL许可证的代码都必须是公开的。

这个许可证确实很好地实现了其保持软件自由的预期目的。但它的病毒性质使它对不同类型的潜在用户有一定的限制。这意味着更严格的许可会降低可用性；尽管GPLv3的目的只是阻碍专有软件的可用性，但其病毒性质也对从事开放项目的开发人员产生了影响。为了合并你的代码，他们将不得不重新许可，而一些开发者可能不愿意这样做。如果我正在做一个项目，我想让尽可能多的人使用它，它可能会有一个麻省理工学院的许可证，我真的需要一个GPLv3依赖关系来考虑在一个更严格的许可证下重新许可我的项目。

每个许可证都有更多的含义，但是根据我的经验，这些是Python程序员最重要的区别。下面我将讨论一个使用GPLv3的具体例子。

LGPLv3

宽松通用公共许可证(LGPL)是标准GPL的一个更宽松的版本。当GPL强制你在GPL下许可整个项目时，LGPL允许你在你的项目的其他部分使用其他许可，如果你满足某些条件的话。这意味着有人可以将带有LGPL的源代码合并到专有软件中，然而这并不像MIT/BSD许可证那样简单。

为了使用带有LGPL许可证的源代码，您必须允许任何人修改、重新编译或替换带有LGPL的代码部分。这意味着用户必须能够使用修改后的代码下载、修改并运行您的软件。有一些方法可以在不公开整个代码库的情况下做到这一点，比如在一个单独的库中公开带有LGPL的源代码。

这意味着，虽然你的源代码有可能被合并到私有软件中，但你的实际工作将永远是公开的。您可以看到，选择许可证在很大程度上取决于您作为开发人员的个人优先级。

为什么不总是使用非常宽松的许可证呢？

当我第一次开始写Python软件时，我只是在所有东西上都贴上了麻省理工学院的许可证，并称之为好东西。但是，在开发一个处理无源微波数据的包时，我需要结合一个同事的项目的本地源代码，他在大学的另一个系工作。我有一个问题，她的代码没有许可证！我向她求助，请求她许可，这样我就可以安全地将它融入我的作品中。我建议她使用麻省理工学院的许可证，但她回来告诉我她将使用GPLv3的许可证。

她的理由是，她的研究是由公共资金资助的，因此她的源代码可能被用于私人场合以获取利润，这似乎是不道德和不公平的。这个推理对我来说很有意义，而且我也不介意将我自己的源代码重新授权给GPLv3以确保兼容性。毕竟，我在一个在我们的使命宣言中有“致力于开源科学”的实验室中，为处理地球科学数据而开发python库。从伦理上来说，使用一个保持代码严格开源的许可证并不是一个坏主意，但是严格的许可证很可能降低了我的包对某些群体的可用性。

包扎

我知道许可是复杂的，但是从这篇文章中得到的重要信息是，你可以根据几个关键的决定来做决定:

  1. 你能接受你的代码可能被合并到专有软件中，但更容易使用吗？
    - 是:进入第二题
    - 否 : GPLv3
  2. 如果你能接受专有软件，你能接受在私人环境中存储和修改代码吗？
    - 是:许可(MIT/BSD/Apache 2.0)
    - 否 : LGPL

使用许可许可证，像麻省理工学院，可能会使你的程序员同事的生活更容易。如果你对这两种方式都没有强烈的感觉，那么我建议你只对标准Python项目使用MIT许可证。在许多方面，这归结为一个道德问题，没有哪个选项是对的或错的。

重要的是要记住，这些许可证比我在这篇短文中讨论的要复杂得多。如果你有进一步的问题，不要犹豫寻求进一步的资源或律师，因为许可可能是非常重要的。

底线是不管你用什么许可证，最重要的是你有一个许可证。
