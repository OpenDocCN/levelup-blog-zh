我喜欢学习新的概念和我已经使用的系统部件。当我有问题时，我倾向于问谷歌教授，不可避免地会找到一个教程、文章或会议讨论来描述我正在寻找的东西。

问题是，我倾向于打开每一个结果，一遍又一遍地阅读相同的概念。所有这些都不需要编写代码。因为我喜欢学习，阅读比实际工作更容易掌握。

但是在 StackOverflow 之前，在 Google 之前，在我们称之为代码的东西的早期，你不能只是问别人如何解决一个问题。你必须自己去发现。如果你做不到这一点，我不会说你不是一个“真正的”程序员，但我会说，如果你*能*自己解决问题，它会让你充满在 StackOverflow 上永远找不到的满足感。

我是一个解谜的人:大部分是逻辑谜题，大部分是数独。我喜欢用我能找到的一些最难的数独游戏来挑战自己，因为即使它们可能要花我几个星期才能弄明白，但我知道当我找到答案时，我会再次感到兴奋。

探索性编程就是那种感觉，但是对于代码来说。

# 什么是探索性编程？

我称之为“探索性编程”，因为我把它比作探索性手术。检验时外科医生做探查手术，问问题无处可去。我在一个[非常相似的上下文](https://wiki.c2.com/?ExploratoryProgramming)中见过这个术语，所以我不会因为在这里创造了它而居功。

因为编程不是生死攸关的事情，所以当我们开始探索的时候，我们可以宽容得多，但是想法是一样的。当我们需要亲自观察系统的内部以了解它是如何工作的时候，我们就开始探索。

探索性编程类似于原型开发，或者务实的程序员的“追踪子弹”，有一个明显的区别:这个想法不是发布产品。我使用探索性编程作为学习工具。如果一个产品想法由此而来，那很好，但这不是编写代码的原因。

在我的工作中，我可能会使用探索性编程作为解决[尖峰](https://ancaonuta.medium.com/how-spikes-help-to-improve-your-agile-product-delivery-a0f104305911)的一种方式。工作之外，我可能会使用探索性编程来继续测试我的技能。

我发现我用例子解释更好，所以让我们在这里有一个，好吗？我从一个问题开始了我最近的一次探索性编程冒险:“我能完全用定制的 HTML 标签创建一个网站吗？”

在[我关于这个项目的文章](/death-to-div-creating-a-better-html-4775793270fc)中，我深入探讨了学到的教训以及我是如何解决这个问题的。然而，对于本文，让我们把重点放在探索阶段。

它始于我读到的一条 tweet，这条 tweet 让我写了一个只有自定义标记的快速 HTML 文件。我在我的浏览器中测试了一下，效果不错。没有 JavaScript，没有 CSS，只有纯自定义的 HTML。

所以我挑战自己，只用自定义 HTML 创建一个完整的单页网站。在这个过程中，我学到了很多东西——比如一些标签不能被忽略——我发现整个过程令人振奋。我不仅在编程，而且在编程的同时我也在激励自己。

我不会称之为解决倦怠的方法，但对我来说，它把乐趣带回了我喜欢做的事情中。

# 探索 WordPress

我最近使用探索性编程完成的另一个项目可能是完整过程的更好例子，从开始到结束。

有一天，我问自己，作为一个挑战，“有没有可能使用 WordPress 的核心钩子创建一个 WordPress 插件，让我可以完全访问 WordPress——永远不接触用户数据库？”

简而言之:我如何创建虚拟用户？这种类型的系统有许多可能的用途，但目前，我只关心我是否能做到。

## 该过程

最近，我将 XDebug 添加到我的指令清单中，作为一个顽固的“死亡和转储”调试器，我很痛苦地这样说，但是分步调试已经使我的开发速度提高了 10 倍。

尤其是对于像 WordPress 这样复杂的系统，能够一步一步地完成任何特定的流程不仅仅是有价值的，我甚至认为这是必不可少的。

当弄清楚如何*开始*探索性编程时，我做的第一件事就是弄清楚如何执行我想要修改的任务。在本例中，这是登录表单提交。

## 步骤调试登录表单提交

需要注意的是，在这个阶段，一切都是为了研究。我们在读代码。我们慢慢地移动，分析我们接触的每一行——声明的每一个变量，我们可以与代码接口的每一个可能的位置。

因为我在 WordPress 中工作，所以我会记下我在这个过程中发现的任何挂钩或可插入函数。两个钩子立刻映入我的眼帘，`wp_authenticate`和`authenticate`。

不必深入挖掘，假设第一个是执行用户定义的函数而不返回数据的操作，另一个是过滤器，与操作相同，但有返回值。

仔细阅读这些钩子让我想知道我是否可以以某种方式使用它们来拦截现有的 WordPress 用户类(`WP_User`)。我相信我可以，但我还不确定具体怎么做。

## 书写探针

我在阅读`wp_authenticate`行动时注意到的一件事是，这不是一个普通的行动。一般来说，对于一个动作，你不能修改任何东西。对于`wp_authenticate`来说就不是这样了，因为它是一个通过引用传递变量的动作。

我想知道是否可以修改这些变量，并干扰流中的其余调用。

所以我写了一个探针。

探针是我称之为对我测试的代码的修改。探针不一定是最终结果的一部分。他们可能在那里确保事情按照我们期望的方式进行。

在本例中，我编写了一个探针来修改用户名值，看看会发生什么。这很有趣，但我不知道如何使用它来做任何事情。未来的探索可能会发现我可以在那里做些什么，但在这一次，我没有。

我确实发现我可以使用`wp_authenticate`作为某种检查点。如果我在那里做一个检查，看看我是否传递了一个特定的用户名，我可以把钩子挂在`authenticate`过滤器上。这样，代码将只在我尝试虚拟登录时运行，允许虚拟登录在不干扰常规登录的情况下发生。

## 最重要的是，专注

我所做的每一件事，前进的每一步，都包含一个非常特别的规则:专注。

我们可以将每个问题分解成我们可以回答的离散事件、任务和问题的子集。总的问题是我是否能在不接触数据库的情况下登录 WordPress，但是在这个过程中还有更多小问题。

我可以在不接触数据库的情况下生成成功的登录表单响应吗？我可以在数据库表中没有用户的情况下创建会话吗？我可以向不存在的用户授予权限吗？如果我能做到以上所有，还有什么有用的吗？

我们可以进一步分解这些问题，直到你最终找到一个你能回答的问题。每当你找到一个你能回答的问题，那就是我们开始的地方。

这就是我们解决复杂问题的方式。不是通过知道所有的答案，而是通过学习如何*找到*答案。因为愿意尝试。错误足够长的时间来变得正确。

## 经验教训

一路上我回答了所有的问题吗？没有，但我回答了重要的问题。我证明了我能够在不接触用户数据库的情况下创建一个登录的用户会话。

仍然有一些问题要问和回答，但这超出了本任务的范围。我的目标是登录，我做到了。

我现在对 WordPress 登录系统了解得更多了——例如，在这个探索性编程之前，我根本不知道会话管理器进程。我知道肯定有一个，但我从来没有理由去找到它。现在，我不仅知道它在哪里，还知道它在接口层面是如何工作的。

从这篇文章中得到的重要的东西不是你可以不用接触数据库就能登录 WordPress。你应该明白，你可以通过提问和探索来解决复杂的问题。我们不应该害怕打破东西。

我们的工具让我们能够控制时间，我们的环境允许我们将实验放在受保护的桶中。这让我们可以在代码中自由地做任何我们想做的事情，而不用担心失败。如果一个实验失败了，我们可以恢复一个提交并尝试一个新的。

编程是一个谜题，谜题让生活更有乐趣。
