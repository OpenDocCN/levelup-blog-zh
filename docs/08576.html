DIV之死:创造更好的HTML

原文：https://levelup.gitconnected.com/death-to-div-creating-a-better-html-4775793270fc?source=collection_archive---------0-----------------------#2021-05-14

为什么每个人都喜欢的元素可能永远不会被使用。

大多数家庭都有一个房间，它的存在仅仅是因为四面墙围绕着它，现在它需要一个目的。这些占位房间最终会比你房子里的其他房间使用得更频繁，但这不是它们的初衷。它们的多功能性和独特性允许我们经常改变它们，对它们的用途没有预先的概念，总体上让我们感到自由。

div很像这些房间。这是一张白纸。它适合任何流程，允许我们完全控制它如何工作，并且可以成为我们告诉它成为的任何东西。整个网站可以用一个多一点的div创建。搜索“单个div设计”,你会发现无数的设计者炫耀他们的CSS能力，强迫单个div将形式变成他们想要的任何形式。

这些有时令人肃然起敬，但是我在这里要说的是，我们应该超越div去看一个没有类名或id的div的世界。自定义HTML元素的世界。语义HTML。关注点的分离，以及对基础的关注。

在阅读了Jamon Holmgren 的推文后，我创建了一个名为自定义HTML标签的项目。在这篇推文中，他反对使用带有类名的div来创建HTML组件，而是使用定制的HTML标签。

美在于简单。默认情况下，自定义HTML标签已经被分割。通过简单地删除类名前面的句点，我们可以保持与以前完全相同的样式。相应的HTML和CSS都更加语义化，字符也更少。

扪心自问:我这辈子给div赋过几个类名？在HTML中搜索这些组件时，是搜索div本身，还是搜索类名？

我在我的项目转变思想实验中探索了这些问题，发现我不仅能够在不使用任何div的情况下创建整个网页，而且我还能够缩小网站的大小，用适当的构建管道设计所有内容，并创建我所做过的最小的构建项目。

anubisthe jackle/custom-html-tags

github.com

但是可访问性呢？

我不是什么专家，也不会假装是。我的目标是创建这个项目，并解决出现的可访问性问题。为此，我在Twitter上找了尽可能多的可访问性专家，请他们浏览页面，并提供一些建议。

我还用多种可访问性扫描仪扫描了页面。它并不完美，但我觉得结果足以推翻自定义HTML永远不可访问的论点。

它是如何设计的

我不是设计师，我也不想在CSS上花太多时间，这也是我选择使用Tailwind CSS的原因。然而，我的意图不是在HTML中添加类，而是将关注点分开。因此，通过在样式表中使用apply，我可以将Tailwind样式直接应用于定制HTML，根据需要添加任何定制CSS。

讽刺的是，我觉得顺风的这种用法显示了它的真正潜力。太多的焦点放在了内联类等同于内联样式上，以至于我们完全忽略了apply特性。

通过适当的构建管道，我能够将样式表的大小缩小到运行页面所必需的大小。JavaScript和HTML也是如此。所有这些都被缩小了，加速了页面加载，以及所有的核心网站的活力。

你能用自己的风格做到这一点，而不是跟风？当然了。这肯定不是一个要求，但对于像我这样不想在这个项目中关注CSS的人来说，它很快，很容易设置，也很容易使用。简而言之:它刚刚生效。

JavaScript

这个项目唯一使用的JavaScript是为了增加可访问性。在a11y更新之前，定制的HTML标签只是默认为浏览器的div配置，这很有效。但是核心的HTML标签包含了被大多数开发者忽略的东西:默认的ARIA设置。

没有这些，辅助工具就不知道如何使用你的标签，也不应该被期望这样做。也就是说，用JavaScript创建定制元素并不难，然后根据需要给它们分配ARIA属性。

项目的不可行性

这个项目本身从来都不是为了实用。这是一个走极端的思想实验，看看有多大可能走得那么远。这个想法是，如果有可能在如此极端的水平上做到，我们完全没有理由不能在一个更小、更专注的水平上做到这一点。

这正是我发现的。一旦掌握了基础知识，不仅有可能做到极致，而且相对简单。在一个生产项目中，没有理由不为某些组件创建定制的HTML标签。

简而言之，在你的职业生涯中，绝对没有理由需要再次使用div。
