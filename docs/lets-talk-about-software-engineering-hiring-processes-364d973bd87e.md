# 招聘流程目标

首先，简单介绍一下面试结果的[二元分类](https://en.wikipedia.org/wiki/Binary_classification)。面试最终可以归结为四种可能的结果之一。

1.  真正积极的:你雇佣了候选人，他们在新的岗位上取得了成功。
2.  真正的否定:你不雇佣候选人，他们就不会在新的岗位上取得成功。
3.  **误报**:你雇佣了候选人，他们在新的岗位上没有成功。
4.  **假阴性**:你没有聘用该候选人，而他们本可以在新的岗位上取得成功。

我将在整篇文章中使用这些术语。

当然，目标是创建一个过程，最大化真阳性和真阴性的数量，最小化假阳性和假阴性的数量。这是雇主对他们招聘过程的看法。他们还必须确定是优先消除假阳性还是优先消除假阴性。

对于一个**大型组织**来说，他们在招聘板上发布的每个软件工程职位都可能有数百名申请人，他们可能会选择**针对误报进行优化**。假阴性是很不幸的，但是他们的申请人太多了，他们不需要等很久就能得到真正的阳性结果。他们可能认为这是两害相权取其轻。

对于一个小组织来说，他们在招聘栏上发布的每个软件工程职位可能只有少数几个申请人，他们可能会选择优化以避免误判。误报是不幸的，但是雇佣一个表现不佳或者比最初预期需要更多指导的候选人是可以的，如果这个职位空缺了几个月。他们可能也认为这是两害相权取其轻。

# LeetCode 风格问题

## **好人**

LeetCode 风格的问题提高了反馈循环的速度。候选人要么找到最优解，要么找不到。如果他们不这样做，组织可以提前继续前进。这可以防止误报。

LeetCode 风格问题中的表现与技术能力的相关性比许多软件工程师愿意接受的更强。如果你能够理解和解决一个复杂的问题，如“倒置二叉树”，这是一个快速有效的指标，表明你能够理解一个问题，如“如何在一个反应应用程序中管理状态。”

## 坏事

较小的组织已经开始模仿 FAANG*公司的招聘流程，而没有真正考虑为什么 FAANG 公司选择并能够以这种方式招聘。

这造成了焦虑，因为候选人觉得这是对一个不需要相同技能水平和没有相同报酬的角色的不必要的审查。

* *当我写 FAANG 时，我指的是缩写中的公司，以及任何其他精英技术组织或独角兽初创公司。*

## 丑陋的

在面试中被问到的 LeetCode 风格的问题是如此的困难，与一个真正的软件工程师的日常生活相去甚远，事实是候选人需要学习它们。

问非常困难的 LeetCode 风格的问题有利于那些在工作和家庭责任之外有很多时间准备的候选人。多元化和包容性在这里会受到冲击。最好的情况是，他们错过了来自不属于那个阵营的人的反馈和见解。在最坏的情况下，它会影响平等机会。

# 设计风格问题

## 好人

设计风格问题降低了反馈循环的速度。这些问题是开放式的，所以候选人有机会谈论他们所知道的。这优化了对假阴性的防范。

它们是衡量一个候选人对软件和软件开发的理解的一个不错的标准，候选人并不真的需要为它们而学习。

## 坏事

没有已解决或未解决的二元，他们很难防止误报，因为候选人可以编造一个足够好的回答来满足面试官。候选人可能不明白如何实现他们描述的设计，以及它的后果等等。

## 丑陋的

当候选人申请入门级职位时，问他们复杂的设计风格问题是不公平的，特别是如果他们的经验仅限于计算机科学本科学位和可能的实习。我不认为这是一个大问题。

# 带回家作业

## 好人

在带回家的任务中，候选人被给予要求，一段时间，以及使用他们所有的开发工具来构建一些东西，以提交或演示给他们申请的组织。这可能是一个组织在招聘过程中可以包括的最接近真正的软件工程的东西。

## 坏事

这很难防止误报。没有办法确定候选人没有和他们的朋友坐下来完成带回家的作业(这不一定是一件坏事，取决于你如何看待它——这仍然模拟了真正的软件工程)。

## 丑陋的

做 6 个小时的无偿工作来换取*份工作的可能性*被候选人认为是不公平的。

组织甚至可能把带回家的作业作为让求职者免费为他们工作的机会——这是非常不道德的。

# 结论

组织使用特定的招聘实践，因为他们试图在招聘上更务实。他们分析面试的可能结果，以及他们想要优化或反对的结果。

LeetCode 风格题、设计风格题、带回家作业都各有利弊。对一个组织起作用的可能对另一个组织不起作用。
