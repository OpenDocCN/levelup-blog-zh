要成为一名优秀的程序员，我们应该遵循一些易于采用的习惯，以保持我们的编程生涯长久。

在本文中，我们将看看如何对软件项目进度进行估计。

# 估计项目进度

对于像项目这样复杂的东西，我们只需要在开始做项目之前把它们分解成小的可管理的部分。

为了更好地进行评估，我们只需要通过获得经验来变得更好，这样我们就可以从过去的经验中学习。

为了分析和实现一个项目，我们必须检查需求，分析风险，设计，实现，集成，并与用户一起验证项目。

我们可能在开始的时候只有一个模糊的想法去做什么。因此，将需要许多迭代来得到正确的东西。

有人可能希望我们把事情敲定，作为最初计划的一部分。除了最琐碎的项目，这是不可能的。

更好的前进方式是构建和测试第一个增量，然后让它出来，然后我们在后续的迭代中改进它们。

我们在制定项目计划时，应该把这些都考虑进去。

如果我们过于雄心勃勃，我们可能遇到的已知和未知的问题可能会使我们的项目落后。

# 用代码迭代时间表

用代码迭代时间表可能是正确的方法。一个复杂项目的第一次迭代不可能马上被接受。

甚至在此之前，我们可能会遇到问题。我们需要通过发布迭代来获得反馈，以便我们能够处理这个问题。

时间表应该随着代码的进展而更新，这样我们就能知道什么时候会发布。如果要求我们评估进展，我们不应该马上脱口而出一个数字。

在我们给出任何估计之前，最好花一些时间来继续前进。

# 确定目标和承诺

我们应该考虑在给定的时间内一个项目能完成多少。

通过这种方式，我们可以清楚地划分各个部分和任务，这样我们就可以给出估计值，并给出完成每个小任务的总估计时间。

仅仅从整体上对整个项目进行估算是不可能的。

对于更高层次的目标，我们可以合计所有任务的估计时间，然后用它作为目标的估计。

最重要的是，我们清楚地知道要实现既定目标或承诺，我们必须做些什么。

然后，通过将所有目标的评估放在一起，可以提交整个项目。

# 清楚地了解项目的范围

在开始一个项目之前，我们需要了解项目的范围。

否则，由于没有明确的范围，我们不知道如何估计任何东西。如果项目可以向任何方向发展，那么我们无法估计它，因为对于我们必须完成的事情没有明确的目标。

为了确保范围清晰，我们必须做一些事情。

首先，我们必须见面，这样我们才能明确范围。然后，我们必须研究具有类似范围的其他项目，以便我们可以对所涉及的内容有一个大致的了解。

此外，我们必须了解我们正在做的项目类型的细节。最后，我们要对任务进行优先排序，看看我们是否能在给定的时间框架内完成所有任务。

# 非功能性需求也很重要

我们必须记住完成每项任务所需的所有非功能性需求。

我们必须腾出时间来编写测试并清理我们的代码，这样我们才能以一种可维护的方式完成事情。

此外，还有许多我们必须考虑的事情，如可伸缩性、性能、安全性、可用性以及与其他系统的互操作性，这些都是用户和利益相关者没有看到的。

# 结论

如果我们没有对事物进行足够的分解，给出估计会很困难。在我们给出评估之前，我们必须从小任务以及每个任务的功能和非功能方面来考虑它们。

小任务比大目标和大项目更容易评估。因此，在进行估算时，我们必须考虑任务。
