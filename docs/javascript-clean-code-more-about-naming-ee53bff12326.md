编写干净、可读和可维护的代码需要一些努力。程序最基本的部分之一是标识符。为了使阅读和修改代码更容易，它们的命名必须很好。

在这篇文章中，我们将看看一些简单易懂的命名方法。

# 避免心理映射

开发人员不应该将他们的名字映射到他们大脑中的含义。

举个例子，

```
let r;
```

对于保存 URL 字符串的东西来说是一个错误的变量声明，因为名称`r`中没有任何关于 URL 的内容。

我们可以把它命名为`url`,就像我们想节省在大脑中映射它的努力一样。

给事物命名时，清晰是最重要的。

# 类名

在 JavaScript 中，类名是大写的骆驼字母。所以对于一个类来说，`FruitStand`是一个好名字，而`fruitStand`不是一个好名字。

我们应该遵守惯例以避免混乱和错误。

# 方法名称

在 JavaScript 中，方法名应该有骆驼大小写。例如，`getName`是一个函数的好名字，但`GetName`不是。

# 不要用聪明的名字

我们应该使用聪明的名字来迷惑我们代码的读者。例如，`holyCow`是一个不好的变量，因为它没有传达变量的实际内容，所以它保存了水果的数量。

这使得代码难以阅读，也不好笑。

相反，我们应该将 if 命名为`numFruits`，这样人们实际上就知道该变量保存了水果的数量。

我们应该根据事物的含义来命名它们。

# 为每个概念选择一个词

我们应该为每个概念选择一个词。例如，如果我们想表达一个函数得到一个值。那么我们就不应该在不同的函数名中有`fetch`、`get`，或者`grab`。

为每个概念选择一个单词减少了开发人员理解代码所需的脑力，因为一切都是可预测的。

# 没有双关语

在标识符中有两个相同目的的单词会使它们混淆。因此，我们应该避免它们，因为不同的人会对它们有不同的解释。

例如，`augment`对于一个函数来说是一个不好的名字，它可能意味着向数组添加条目，或者向字符串追加内容。所以我们应该用能显示它实际功能的名字来命名它。

# 开发人员理解名称

为开发者理解的事物命名是很重要的。在我们的代码中，我们应该命名与解决方案领域相关的事物。

比如很多开发者都知道 a `MessageQueue`或者`Database`是什么。因此，这些都是好名字。

# 问题域名

如果在技术术语中没有开发人员理解的名称，那么我们应该以描述我们试图解决的问题的方式来命名事物。

# 有意义的语境

我们应该根据标识符名称的上下文来命名事物。例如，`username`和`password`是很好的名字，因为我们知道这些变量是关于用于登录系统的用户实体的。

另一方面，像`number`和`modifier`这样的通用名不是好的变量名，因为它们没有提供太多关于我们将这些变量值设置为什么的上下文。

如果我们用这些名字来构造一个字符串，这个字符串有一些东西的数量，然后是我们要量化的东西的修饰语，那么我们应该这样写:

```
let numberOfItems = 1;
let pluralModifier = numberOfItems ? '' : 's';
let numItemsString = `${numberOfItems} item${pluralModifier}`;
```

我们可以看到，这些名字比`number`和`modifier`有意义得多。它们为我们提供了使用变量的情境。

# 不要添加无用的上下文

没用的语境也不好。例如，如果我们给每个标识符加上前缀`foo`或`Foo`，因为我们正在为公司 Foo 编写软件，这是不好的，因为我们已经知道该软件是为公司 Foo 编写的，所以我们不需要把公司名称添加到所有内容中，占用更多的页面空间和磁盘空间。

我们只需要名字中有足够的信息来表达标识符的含义。如果我们能以更短的方式完成，那就更好了。

用正确的方式给事物命名需要一些思考。我们应该添加足够的上下文，以便人们能够更好地理解代码。

此外，开发人员也不需要在头脑中把名字映射到他们脑海中的某个东西上。如果需要的话，这个名字必须传达更多的意思。

聪明的名字和双关语也不好，因为它们会误导人。

最后，坚持 JavaScript 的惯例是非常必要的，这样可以避免混淆。
