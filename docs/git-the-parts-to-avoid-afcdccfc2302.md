# 获取要避免的部分

> 原文：<https://levelup.gitconnected.com/git-the-parts-to-avoid-afcdccfc2302>

Git 是一个很好的工具，可以管理代码随时间的变化，并允许开发团队集成代码。但是，即使是最有用的工具在被误用时也会出现问题，我们需要了解这些缺点以避免它们。Git 最常见的问题是重写历史和撤销已经做出的更改。

![](img/8b4c69c66e47b35e565a1bcae6ef31b0.png)

让我们看看 Git 中我们应该避免的部分，以使我们作为开发人员的生活变得简单和容易。

# 壁球

使用 Git 的标准过程是克隆一个存储库，在本地进行更改，然后将它们与远程服务器集成。当开发人员向存储库提交多个变更并将代码推送到远程服务器时，在版本历史中将会显示多个提交。这是有益的，这样可以跟踪变化，但有时会被视为混乱或难以阅读。

![](img/c33061fc2e1fbb9c550f2390150c6607.png)

有时开发人员会使用 git squash 功能将多个提交转换成一个提交，这样整个更改就在一个提交中完成了。然而，由于一些原因，应该避免这种情况。

首先，它删除了中间状态，这意味着当出现问题时，我们无法访问变更开始和结束之间的提交。

其次，它重写了变更的历史，我们失去了提交的时间。这导致可用于帮助识别不同分支何时进行变更以及何时出现冲突的信息减少。

第三，它增加了变更的规模，这反过来增加了风险。与不频繁的大变化相比，处理频繁的小变化要容易得多。

最好不要使用 squash，而是保持提交历史不变，并按照提交时的样子呈现每个提交。这使我们能够保留变更的历史，回顾变更中的所有提交，并通过进行较小的变更来管理风险。

# 强制推送

强制推送是用我们的代码版本覆盖远程服务器上的更改的一种方式。但是这样做会给其他团队成员带来问题，并导致我们丢失代码更改。

![](img/a1eff389e12f340b2d9919222885dab2.png)

当我们强制将我们的更改推送到远程服务器时，在相同代码基础上工作的其他开发人员可能会以处理旧代码而告终。如果他们拉最新的拷贝，而您随后又在其上强制推送，他们的客户端将会不同步。如果他们以强迫推动他们自己的改变来回应，给你造成同样的问题，问题就更复杂了。

强制覆盖远程代码还会丢失自上次拉取以来所做的更改。这可能导致旧的 bug 回来，失败的构建，和工作的丢失。

# 摘要

Git 是一个很好的工具，但是当被误用时会导致很多问题。作为使用 Git 的一部分，我们应该避免使用诸如挤压和强制推送之类的功能重写历史或撤销更改。这样做将有助于我们保持工作简单高效。
