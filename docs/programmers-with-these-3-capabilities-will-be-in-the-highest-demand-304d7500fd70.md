在过去的十年里，面试技巧被重新定义。

以前，他们围绕着样板问题(通常在面试前搜索)。由于产品和服务变成了短期项目，公司开始转移他们的选择空间，重新定义面试。

> 没有人在谈论流失率、裁员和基于绩效的解雇

但是很难说他们成功了多少。虽然重新定义的面试已经出现，但没有人在谈论流失率、裁员和基于绩效的解雇。如果雇佣策略有效，有多少被雇佣的开发人员最终会留下来？他们中有多少人最终为公司做出了贡献？

最重要的是，这种差异有多少是有意义的？

如果这些公司真的想迎接这些挑战，他们会努力让事情变得更好。以下是他们希望未来的程序员具备的品质:

# #1:编写规格:

瀑布时代的规范相当冗长。大多数空白区域包含的有用信息很少，包括作者、修订号和修订日期。

许多程序员真的鄙视这些规范。

敏捷废黜了文档，就像文艺复兴推翻了宗教的可行部分，用以商业为中心的科学取而代之。

记录的权力到了产品所有者手中。这种权力的一部分被授予了 UX 的设计师。编码员变得像他们编写的函数:接受他们的输入，输出看起来像最理想条件下的输出*的产品*。

当 bug 追上他们的时候，那些编码员声称修复了更多的票。或者，它们已经被替换了。

> 如果你在编码任务中写了一份好的自述文件，你的面试官选择你而不是你的对手的机会肯定会增加。

如果写规格(不符合任何标准)是强制性的，他们从第一天起就已经拥有了产品特性。首先，他们会设身处地为用户着想，去理解他的问题。然后，他们会冒充创始人，来解决这个问题。

编写规范不仅包括规范，还包括编写读者和发行说明，让非专业客户能够容易理解。如果你在编码任务中写了一份好的自述文件，你的面试官选择你而不是你的对手的机会肯定会增加。

如果有人在想*编码员不会写*的问题:编程毕竟是一种表达的技能。那种语言是人类可以理解的。我们已经超越了集合的日子。如果这是一个挑战，我们必须能够接受它。如果我们不这样做，我们创建的所有软件(其大小达到每天数十亿字节)都将是无主的，并将变成一个具有奇形怪状结构的巨石。

# #2:快递:

> 一个程序只写一次，但要多次阅读。
> 
> —鲍勃叔叔

写作是一种表达形式。但是你的程序表达能力如何呢？它在多大程度上表达了它的目的？它将如何使其未来的维护者受益？

这不仅仅是合理的变量和函数命名。你的程序是一个故事，作为程序员，你是向世界叙述这个故事的人。

## **你的班级足够团结吗？**

这将告诉你如何在一个类的共同目的下绑定相关的项目。例如，在一个旅游预订应用程序中，考虑下面的类:

```
class Booking {
   startDate: Date
   endDate: Date
   property: Hotel
   status: BookingStatus
   cancellationReason: String
}
```

变量 *cancellationReason* 只与预订的一种可能状态(*cancelled*)相关联——为什么它应该是 **Booking** 类的一部分呢？它应该属于某个其他实体(可能是 **BookingStatusChange** 类，该类保存旧的+新的 **BookingStatus** 对象，以及一个时间戳和一个足够通用的原因字段来保存取消。)

## **你的功能被设计成在相同的抽象层次上运行吗？**

这是一个相当高层次的概念，但在较低层次上执行。当你写每一行代码时，你需要小心谨慎。

在我职业生涯的重要阶段，我从 StackOverflow 复制代码。然而，我必须承认，当你这样做时，这是一件你不能有效地做的事。你需要练习到你知道为止。

为什么？因为它需要一个人对每个代码语句进行分类，并把它放在该放的地方。

考虑以下(语法不正确的)代码:

```
function sampleFunction(fileName: String) {
    const fs = readFile(fileName) if !validateFile(fs) {
       return []
    } var byteArray = [] while (fs.bytes()) {
       // Do byte operation 1
       // Do byte operation 2
       // Do byte operation 3
    }
    return byteArray
}
```

这种类型的代码非常普遍——甚至在大型科技公司中也经常出现。它缺少的一点是相同层次的抽象。while 循环中的语句处理字节，而它前面的两条语句处理文件。

为什么不能有一个单独的函数 **parseFileBytes()** 来处理 while 循环的低级字节疯狂？可选地，filehandle fs 也可以传递给这个函数。这样，整个函数都在同一个抽象层次上:它只处理文件句柄。另一方面， **parseFileBytes()** 处理字节级的细节。

额外好处:两种功能都可以单独测试。

```
function sampleFunction(fs: FileHandle) {
   if !validateFile(fs) {
      return []
   }
   return parseFileBytes(fs.bytes())
}function parseFileBytes**(**bytes: [Byte]**)** {    while (bytes) {
       // Do byte operation 1
       // Do byte operation 2
       // Do byte operation 3
    }
    return byteArray
}
```

一些程序员可能会争辩说，包含一个单独的函数调用可能会导致性能下降(取决于语言)。有些人可能也觉得这是微不足道的。

> 花在重构上的一分钟可以节省数百小时的调试时间。

但是如果你考虑调试器程序员的努力，你将能够证明我刚刚解释的原则:[单级抽象](http://principles-wiki.net/principles:single_level_of_abstraction)。当维护人员看到这个函数只处理文件(而不是文件+字节)时，他/她可以在调试过程中迅速决定包含/跳过它。

一个富有表现力的程序员一分钟的仔细重构可以节省数百小时的调试时间。

# #3:放大:

在我早期的职业爱好项目中，我和一个初级同事一起设计了一个图书馆预订网站。注册表单有 6 个字段。他已经验证了其中的大部分。他们中的一些人对这些场的组合起作用。验证的总数是 14！

我要求他重新考虑其中的一些。我认为这是非常糟糕的 UX。对此，他说:“我们上线后不久，他们(1000 名学生)将在大学学期的第一天立即注册。如果他们把坏数据，我们将不得不从后端纠正它。至于坏 UX，每个人都需要它。谁在乎？”

“但是你可以花一些时间来计算验证的实际数量，重构并交付一个质量更好的产品。”我决定照章办事。

对此，他说，“我需要问问图书管理员。他的回答需要时间。如果我们不能在开学前交货，那就没用了。很快，他(图书管理员)就会填写纸质表格，并在全校教职员工面前指责我们。”

他说得有道理。虽然我从来没有同意他半生不熟的，充满验证的用户界面，这迫使我去思考如何在一个真实的生产环境中使用这个网站。

这让我想到了平均负载和峰值统计。它帮助我思考发布后数据迁移的可能性。它帮助我消除了许多无意义的事情，只关注绝对必要的事情。

放大是一项很容易被忽视的技能，因为即使在大学里也很少有人教，更不用说在线教程了。在面试中，这一特质被归入**所有权**公司价值(考虑长期影响与短期效用)。我在[我的高级开发人员访谈电子书](https://tipsnguts.gumroad.com/l/crrzat/zp1vks8)中详细讨论了这个问题(前 100 名买家半价)

> 像史蒂夫·乔布斯这样的远见卓识者之所以受欢迎，仅仅是因为放大。

你的放大能力是面试官试图看穿真实的你的地方(相对于已经在他们邮箱里的你的简历)。正是这一点让你从其他候选人中脱颖而出。没有这一点，就谈不上你是否为 X 工作了 Y 年。

像史蒂夫·乔布斯这样的远见卓识者之所以受欢迎，仅仅是因为这种技能，但他们是从科技领域之外获得这种技能的。它不是通过编写一千个程序或解决最大数量的 LeetCode 挑战来实现的。它带有批判性思维技能。

那些技能并不是真正的技能，而是持续的活动，对于那些真正关心他们编码的产品的人来说是一种享受。

这就是他们在面试和年度绩效评估中所说的*所有权*的含义。

# 结论:

随着人工智能的发展，并不是所有的编码员都会很快被淘汰。但是他们运作的水平将与他们现在运作的水平大不相同。

这三种能力将在区分人类编码员和机器时派上用场。
