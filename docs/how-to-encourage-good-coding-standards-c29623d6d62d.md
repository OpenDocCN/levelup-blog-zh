好的代码对每个从事代码工作的人都有好处。因此，我们应该鼓励这样做。

在本文中，我们将看看如何鼓励良好的编码标准。

# 制定标准时的考虑因素

制定标准在一些组织中比在其他组织中更有用。

我们可能想设定一些标准，这样每个人都在同一页上。

# 给项目的每个部分分配两个人

如果每个项目都有两个人，那么他们可以互相监督，因为如果一个人写了糟糕的代码，另一个人就会遭殃。

# 审查每一行代码

审查代码很重要，因为在审查的压力下，每个人都会写出更好的代码。

如果有什么需要改进的地方，人们会提出改进代码的建议。

# 要求代码签核

代码签署是一些团队做的事情。它让团队领导在代码发布前检查代码，以确保它是好的。

# 传递好的代码示例以供审查

提供好代码的例子也让大家清楚什么样的代码好，什么样的不好。

因此，我们应该提供它们来解释什么是好的。

# 强调代码是公共资产

很多人大部分时间都会在同一个代码库上工作。

所以，既然我们不希望我们的队友因为使用糟糕的代码而遭受痛苦，我们就应该努力让他们变好。

# 奖励好代码

建立一个奖励系统也是奖励写好代码的好主意。

开发人员想要的某种礼物可能是很好的选择。

# 一个简单的标准

一个简单的标准比许多复杂的标准更容易遵循。

因此，我们应该坚持一个，并与它一起生活。

# 结构管理

配置管理是系统地识别项目工件和处理变更以便保持完整性的实践。

我们必须控制这些变化，这样团队成员在进行项目时就不会互相影响。

代码可能未经测试，或者可能不会被使用。

如果这些变化得不到控制，就会浪费很多时间。

# 要求和设计变更

我们必须有一个系统的方法将需求整合到我们的代码中，并将它们转化为现实。

# 遵循系统的变更控制程序

系统化的变更控制程序将使变更的合并变得更加容易，因为变更将以有序的方式添加。

# 分组处理变更请求

分组处理变更会使我们的工作变得更容易，通过对它们进行优先排序，这样我们就可以从一组请求中首先做出重要的变更。

然后，如果我们有时间，我们继续做一些不太重要的改变。

# 估计每个变化的成本

我们需要投入时间和精力来做出改变。

因此，我们应该提供一个估计，以便我们可以轻松地处理它们。

# 警惕高变更量

如果我们有很多变更请求，那么顶层设计可能考虑得不够好。

我们不想扔掉我们经常工作的东西。

这只是浪费时间和精力。

# 以一种对我们的项目有意义的方式建立一个变更控制委员会或者它的等价物

变更控制委员会可以对变更请求进行分类，这样每个人都知道哪些需要完成，哪些不需要。

例如，我们可以创建一个小组来审查所有提议的变更，并批准、不批准或推迟每个变更。

# 官僚主义

官僚主义并不总是坏事。如果我们需要改变控制，那么我们需要一些官僚机构来优先考虑这些改变。

否则，事情可能会很快失控。

# 版本控制软件

版本控制软件对于跟踪我们的应用程序的变更历史是必不可少的。

有了它，我们可以轻松地还原和合并代码。

否则，如果出现问题，就很难撤销我们的代码。

当我们处理同一个文件时，我们也不会互相冒犯，因为冲突可以合并在一起。

每个人都可以用它轻松地完成自己的代码。

版本控制也是一种备份机制。

# 结论

我们可以通过指定人员来确定代码变更的优先级。

此外，我们应该遵循一些管理变更的程序。

版本控制软件对于让多人以有序的方式一起处理相同的代码也很重要。
