这里列出了一些最黑暗或者最滑稽的编程范例，它们被用来让代码和技术屈从于我们的意志。这些范例在队友之间制造了混乱和战争，在整个社区煽动邪恶的思想。

# 货物邪教编程

![](img/59595d2dcf22485b90ae904f5afaa40c.png)

由[罗布森·哈苏卡米·摩根](https://unsplash.com/@robsonhmorgan?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 拍摄的照片

你有没有遇到过被无意义的代码包围的代码块…这些代码可以被删除而不会对其原始行为产生任何副作用？

那个代码块很可能是货物崇拜程序的受害者。这是我们的第一个禁忌范式！

Cargo cult 编程的特点是仪式性地包含没有实际用途的代码。导致膨胀的代码库，其逻辑是 [***你根本不需要(它)***](https://en.wikipedia.org/wiki/You_aren't_gonna_need_it) ***。***

无论何时，Cargo cult 程序都在运行:

*   我们复制代码片段，却没有完全理解它
*   我们调整代码，用新的代码进行小的调整，直到它工作为止。
*   我们离开现场，没有任何反思和/或清理
*   我们不练习代码删除和重构
*   我们并不试图完全理解来自 Stackoverflow.com 的“***2009”***的代码，然而我们继续使用它。

Cargo cult 只会摧毁我们的代码库。避免它，并且总是质疑在互联网、论坛或在线讨论网站上找到的任何代码。

# 复制粘贴编程

![](img/9f614c905b295e577def5dbd93a15404.png)

照片由[克林特·帕特森](https://unsplash.com/@cbpsc1?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

当缺乏技术或语言方面的经验，缺乏从头开始编写代码的意愿，或者只是需要一个快速、现成的解决方案时，通常会使用复制粘贴编程。

本质上，复制粘贴并没有什么大问题。每个人都这样做！实际问题不在于粘贴一段代码的过程，而在于其后的操作。

典型地，当复制粘贴时，人们应该寻找不同代码风格或标准之间的不一致，因为这些可能违背针对该代码库采用的预定义风格。注意那些遵循不一致命名风格的变量、类或函数，或者那些用遗留约定编写的，或者被混淆到无法表达任何真正意义的变量、类或函数。

一定要理解被复制粘贴的代码，注意不要引入任何非故意的行为，以免成为伏都教或魔法编程的受害者。

# 巫毒编程

![](img/af8ece22350b11caeac83fe878926de6.png)

巫毒编程从业者准备部署在 [Giphy](http://giphy.com/)

[巫毒编程](https://en.wikipedia.org/wiki/Voodoo_programming)在没有真正理解潜在问题是什么的情况下，试图通过多次猜测的试错来产生期望的结果或行为。这基本上是一种暴力方法。

应用巫毒编程的典型方式有:

*   尝试一些你知道不应该成功的事情，但是有时候真的成功了
*   重新编译一切，并再次执行应用程序
*   删除您的代码，然后重新编写
*   以一贯的方式做事，却不知道为什么会这样

# 神奇编程

![](img/0e36d3c525e56f12193909c500b56d95.png)

Artem Maltsev 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

[神奇编程](https://en.wikipedia.org/wiki/Magic_(programming))是一种编程风格，它将所有复杂的操作逻辑隐藏在看似简单的代码片段之下。通常，调用者无法预测代码的真实行为，并且在运行时可能会也可能不会出现意外。

哦天哪！像这样的代码通常在没有合理解释的情况下工作。不管我们看多少次。

那些掌握这种范式的人经常会用你永远不会理解的方式解决你从来不知道的问题，用自动运行*的程序。*

*对于我们其余的人来说，我们只是祈祷，这样我们就永远不必在那里修复任何错误。*

# *爬行优雅编程*

*![](img/369da8c5583275092a4a33b9125aae14.png)*

*凯西·霍纳在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片*

*[渐进优雅](https://en.wikipedia.org/wiki/Creeping_elegance)是一种强调极端代码优雅胜过功能性、产品可用性和安全性的范例。*

*极端的代码优雅似乎是一种降低复杂性和增加可读性的尝试，因为这两个方面通常比处理其他系统方面(如安全性、性能、可用性或产品的不断变化和不明确的需求)的负担更受程序员的青睐。*

*然而，代码优雅的问题是，如果不仔细观察，它可能经常导致小而有效的系统被其他大的、臃肿的、过度工程化的系统所取代。*

*请记住，过度设计的通用代码可能会混淆代码库的领域和意图。导致浪费时间去理解它真正做了什么。*

*最后，记住抽象程度越高，代码效率越低。所以也许[保持简单](https://en.wikipedia.org/wiki/KISS_principle)。*

# *霰弹枪手术程序*

*![](img/154ac240d65cf02a7f0bfcdf3e79cdf2.png)*

*由 [Paul Einerhand](https://unsplash.com/@pauleinerhand?utm_source=medium&utm_medium=referral) 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄*

*[霰弹枪手术](https://en.wikipedia.org/wiki/Shotgun_surgery)或编程是一种策略，在这种策略中，有人会一次用几个地方的变化轰炸代码库，希望产生预期的行为、修复错误或引入新功能。*

*对于受复制粘贴编程或巫毒编程影响的大型代码库来说，散弹枪编程似乎是一个不错的策略。*

*尽管很有帮助，但是如果没有容易执行的自动化测试来检测由大量变更引入的 bug，这种策略会非常昂贵。*

*没有适当的 TDD，散弹枪式编程可能会成为一种危险的策略，因为在没有很好地理解正在发生什么的情况下引入了大量的变化。*

# *牛仔编程*

*![](img/3a51751b4b854bba98d9a4661ff46abd.png)*

*普里西拉·杜·普里兹在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片*

*[牛仔编程](https://en.wikipedia.org/wiki/Cowboy_coding)是一个在整个开发过程中给予软件开发者极大自主权的过程。包括使用哪种编程语言、工具、框架、编码风格和项目期限。*

*那些实践牛仔编程的人通常会跳过任何设计或技术讨论。相反，开发人员会尽快开始编写代码。*

*在研究或宠物项目的初始阶段，这种方法是可行的。但是人们很快就会意识到，缺乏结构、标准和编码风格会对解决方案产生负面影响。创建一个产品，要么不能很好地扩展，要么需要在几个月后重写。*

*极限编程对于牛仔编程来说可能是一个更好的选择，因为在它使用软件开发的增量方法的同时，它仍然在产品的每个版本中采用定义的过程。*

*除了笑话，这些都是众所周知的编程风格，在我们作为开发人员的日常生活中经常出现。给它们起个名字只会让我们更能讨论并想出创造性的方法来避免它们。*

*至少我们对名字很有创意！😄*
