我们开始吧。

# ***快速固定器***

Adam 是一名喜欢快速交付票证和功能的开发人员。众所周知，他能在紧迫的期限内完成任务。每当他遇到困难时，他会在谷歌上快速搜索，并应用他找到的第一个解决方案。

这里的问题不是从堆栈溢出或任何其他地方获得第一个解决方案。问题是机械地应用它而不理解后果。在没有完全理解上下文的情况下应用一些东西会在将来导致更多的问题。

我经常看到人们相信快速完成任务会帮助他们脱颖而出。他们也许能在短期内给人留下深刻印象，但从长远来看，他们会悲惨地失败，因为他们对自己解决的问题没有深刻的理解。

# “谁需要测试”

Silvia 是一名出色的开发人员，非常喜欢编程，但她最讨厌的一件事是编写测试。据她说，降低了开发速度，反正有 QA 来测试代码，何必呢。

编写测试有时会很麻烦，但是随着复杂性的增加和需求的不断变化，如果您没有安全网，那么维护代码会变得更加困难。部分原因是缺乏建立测试环境的兴趣，部分原因是缺乏连贯的测试知识。

代码会比你花在上面的时间还要长。你应该确保它做它应该做的，不多也不少。确保下次评估一个特性时，将编写测试考虑在内。

# **总工**

我们都想写出高质量的代码，对最终用户有影响的特性，以及可维护的软件。但是有时开发人员试图创建远远超出需求的解决方案。试图解决可能根本不会发生的问题。很少有例子直接进入微服务架构，甚至不了解系统边界，或者是否需要多个服务，甚至在获得第一个客户端之前就考虑规模，试图使一切都是可配置的，甚至在看到延迟迹象之前就考虑性能优化，等等。

每当遇到这种情况，总是遵循 [YAGNI](https://en.wikipedia.org/wiki/You_aren%27t_gonna_need_it) 和[接吻](https://en.wikipedia.org/wiki/KISS_principle)的原则。本质上，这意味着你应该保持简单，逐步构建，如果你认为有些东西是不需要的，就跳过它。

# 孤独的游侠

成为团队成员是学校教给我们的东西。所有的运动都告诉我们为什么团队合作很重要。在一个团队中，你会为他人着想，在决定一种方法之前咨询他人，你会给队友提供建议。

有些开发人员只是喜欢戴上带噪声消除功能的耳机，然后编码。他们不想和你互动，也对你在做什么不感兴趣。他们在编码和完成任务方面效率很高。但他们同样不善于与他人互动，也不善于向他人解释自己的工作。

# **T5【我讨厌文档】**

有些人认为代码文档不是他们作为开发人员的工作描述的一部分。事实并非如此。编写代码文档和编写代码一样重要。如果你买了一台新电视，却没有安装和正确操作电视的手册，你会有什么感受。你会立刻诅咒制造电视的公司。

编写代码和编写关于代码的文档是两件完全不同的事情。一个开发人员可能在一方面非常擅长，而在另一方面却非常糟糕。一般都是这样。当你不知道如何做的时候，写文档是很无聊的。很多不喜欢写代码的开发人员就是这种情况，因为他们不知道如何写代码。学习写文档需要很长时间。

*这个人编写长达数页的方法和函数。这真的没有任何借口。这个人从来没有想过分解问题，让单个方法更容易被其他类或方法重用。没有缩进。没有一致的编码约定或风格。全局变量到处都是，等等。*

*这是我个人最讨厌的事情。写代码不难，但写好代码很难。大多数人甚至不去尝试。你写的代码透露了很多你的个性。如果你写了笨拙的代码，很可能你在其他事情上也是笨拙的。但是如果你开始朝着写好代码迈出一小步，这将成为一种习惯，并且你会享受这个过程。看着写得很好的代码感觉真好。*

# ****短线投资者****

**吃➞睡➞码➞部署**

*这么多开发者的座右铭。他们只想编码。他们不想学习，他们不希望尝试一些新的框架/库，对领域不感兴趣，他们不关心他们工作的特性是否被客户实际使用。*

*这也是为什么会有那么多的开发者感到累和无聊的原因。有时候，重要的是你要变得自私，从交给你的任务中学习。花些时间阅读关于技术或领域的内容。表现出一定的兴趣，了解客户不喜欢某些功能的原因。向产品经理提问，以了解即将到来的任务和功能。多走一步，积极主动。*

# ****独裁者****

*我的方式或公路是他们的座右铭。非常固执己见。他们对几乎所有事情都有看法。这是他们的想法对你的想法。这是他们的解决方案对你的解决方案。我打赌肯定会有争论。不知何故，它们会不断回到你实现的代码的一部分。即使它工作正常，经过测试，看起来非常好，他们还是会感到不舒服。*

*这个人是生产力的一大瓶颈，将是第一个在压力下崩溃并开始指责的人。这个人不适合团队，但是，他/她可能是一个有经验的/优秀的开发人员。*

# ***“那不是我喜欢的茶”***

*他们写代码。他们不写文档。他们不做平面设计。他们没有完成文书工作。如果不是代码，那就是别人的问题。即使有最后期限，他们也不会去做。*

*Java 开发人员不从事任何非 Java 的工作。他们在得知注册表中的某些内容需要更改时会感到恐慌。他们会因为不得不在数据库中输入东西而畏缩。这些人会尽一切努力避免走出他们的舒适区。*

*我从个人经验中了解到，这种现象在新开发人员中很常见。这正是新开发人员应该避免的事情。准备好跳进去。你只会在做事的时候学习。优秀的开发人员在探索中表现出慢慢/迅速走出舒适区的倾向。*

# *最后的想法*

*从别人的错误中学习被认为是成长的最好方式。我希望这篇文章能帮助你认识到你应该避免的事情，并通过否定而不是增加新的东西来帮助你成为一名更好的开发人员。*

> *有时候，你需要做的只是去除不好的东西，好的东西就会自动闪耀出来。*
