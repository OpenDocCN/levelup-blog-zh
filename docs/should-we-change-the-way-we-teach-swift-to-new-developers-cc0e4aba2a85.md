# 我们是否应该改变向新开发人员教授 Swift 的方式？

> 原文：<https://levelup.gitconnected.com/should-we-change-the-way-we-teach-swift-to-new-developers-cc0e4aba2a85>

## 有一种更好的教学方法可以确保新人从一开始就理解代码质量的概念。

![](img/bf01d8bd4502919513f52c5c75d48711.png)

[斯科特·格雷厄姆](https://unsplash.com/@sctgrhm?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍照

我想对我们教授 Swift 的方式提出一个重要的改变。目前最常见的方法有一个缺陷，可能会将下一代 Swift 开发人员锁定在教程炼狱中。他们可能无法轻松升级。

而我们这些技术写作者，可能也有错。

当我们在写教程、指南和操作指南时，我们可能会养成坏习惯，而不是养成好习惯。我们在教开发人员如何做事情，但我们没有教他们在生产就绪代码中应该如何做这些事情。教程必然会走捷径。教程从上下文中提取代码片段，并将其作为独立于整个应用程序基础结构的东西来呈现。

绝对有一个地方可以提供教程和操作指南。我并不是说它们本身就不好*。有兴趣学习一门新的编程语言的人，需要有一个明确的地方开始他们的旅程，先掌握最基本的概念。*

*但是如果我们继续制作越来越多的教程，我们会教新的开发者只依赖它们。我们会教他们，如果他们不知道如何实现某事，他们只需要搜索一个教程。没有教程怎么办？他们不知道去哪里寻找答案。他们不会有所准备。*

*当我们想到谷歌搜索 StackOverflow 答案时，情况就更糟了。它有时可以用作“给我一个解决方案，我可以复制粘贴”的编码方法。有时候，这已经成为我们做事的事实标准。我们嘲笑我们如何从 SO 复制粘贴。我们嘲笑我们有时不知道代码做什么。我们还嘲笑这样一个事实，即我们的应用程序仍然以某种方式与那些复制粘贴的代码片段一起工作。我们正在规范这种方法，我们说这是一种在某些情况下可以接受的“编写”代码的方式。我们不是已经更了解了吗？*

*同样，在 StackOverflow 上寻找答案本身并没有什么不好。在某些情况下，发帖的用户会很好地解释和记录那里提供的答案。如果你想学，你可以从 SO 那里学到很多东西。我写的不是指责，因为我们都知道事情是怎样的，我写的是当我们延续这种方式时，这种方式形成的后果和习惯。*

*为什么我们不教人们阅读实际的产品代码，而是给他们现成的解决方案？*

*是因为读生产就绪代码很恐怖吗？更可怕的是阅读那些只从教程中学习的人写的代码，并意识到它可能在成千上万的设备上运行。我们的文明和生活越来越依赖软件。*

*除了阅读这类代码，新开发人员没有其他方法可以了解高质量的产品代码。*

*我们应该帮助简化这项任务。我们需要开始养成定期这样做的习惯。我们需要停止给他们鱼，而是在他们学完基础知识后，开始指导他们如何自己制作鱼竿。这就是我们将如何建立一个强大的生态系统，因为这也将增加开发者之间沟通的数量和质量。*

*我希望看到越来越多的故事和文章，一点一点地剖析流行的开源项目，讲述代码背后的故事。加入深刻的观点和评论。帮助理解已经做出的选择。解释使用了什么设计模式，以及为什么。被检查的代码如何适应其余代码库的上下文。所有这些，同时仍然保持我们所写的代码在聚光灯下。*

*我们是已经知道这一切的人。为什么我们不分享这些知识呢？*

*这将最终扭转潮流，并有助于建立一种我们似乎已经忘记的新型习惯。这不是什么新鲜事。这就是我们开源的原因。开源的基础是任何人都可以访问源代码——他们可以阅读它，理解它，从中学习，如果可能的话，改进它。我们争取到了这个机会。让我们不要错过它。*

*我想听听你对此的看法。请在评论中分享你的想法。*
