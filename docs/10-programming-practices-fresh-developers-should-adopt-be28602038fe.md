## 你应该使用的几个简单且最好的习惯

![](img/386d63c2ffaf446158850e1229e3f796.png)

当我还是一名应届毕业生时，我并不知道有许多好的实践可以让你成为一名优秀的开发人员并提高你的效率。我艰难地学会了它们。

在我职业生涯的后期，我雇佣了新生并和他们一起工作。在开始的时候，他们中的大多数是天真的，你花了很多时间回顾他们的工作并训练他们。他们需要 2 到 3 个月的时间来适应。

我列出了一些对新手有帮助的概念和实践:

# 干——不要重复你自己

这个原则是程序员应该知道并时刻牢记的最基本的原则之一。

如果你正在写一个你知道你以前写过的代码，你应该停下来想一想。想想你如何重新利用你已经写好的代码。

这在开始时似乎有点困难，但当你重新审视或重新设计你的代码时，这将变得很容易。

# 好的变量/函数名

应届毕业生有一个习惯，随便给变量和函数起个名字，像 a，b，I，j，或者 fun1。他们从学习的日子里得到这个。我也见过另一个极端，他们给变量和函数起了长名字，看起来像一个句子。

写一个好的变量名，尤其是函数名，是一门艺术，需要你日积月累才能掌握。每次你觉得名字不合适的时候，就花点时间在上面，找一个很棒的名字。

# 可读的代码和有意义的注释

我们都知道注释是如何使代码易于阅读和理解的。经常建议在代码中写很多注释。这会变得很忙乱。

我更喜欢编写易于阅读和理解的代码，并为代码中最复杂的部分编写注释。如果代码写得很好，大多数简单的函数和方法一般不需要注释。

# 遵循惯例和准则

编程语言在如何编写代码方面有自己的惯例。在此之上，团队决定他们的惯例和指导方针。对于一个新的程序员来说，很难理解惯例和准则的重要性。

需要一点时间来理解指导方针的重要性。新手应该遵循指南，阅读前辈写的代码，并适应它。

# 当你学会更好的方法时，重写/重构你的代码

代码不是永远的。每当你意识到你写的代码可以改进或者需要重组，不要犹豫重写/重组。

有时，由于需求或伸缩的原因，您需要重写/重构。虽然还有其他场景，但您可能会预见到一些问题，或者您可能希望让它在未来变得更加健壮。

当有空闲时，这是一个很好的活动。

# 函数/类应该做一件事，并且应该简短

作为幼稚的开发人员，我们都编写了做多件事的长函数。许多团队都有限制一个函数行数的指导原则。

经验法则是“一个函数/类应该做一件事，并且做好”。

# 开闭原理

立体规则的开闭原则规定

> “软件组件应该对扩展开放，对修改关闭”

这意味着软件中的任何组件都应该是可扩展的，而无需对组件本身进行代码更改。

虽然这条规则适用于软件中的任何组件，但它是开发您在代码中使用的库或者如果您想要开源它们的最重要的方面之一。

每次你需要对一个组件进行扩展的时候，你都不需要修改代码。这为您节省了测试和部署组件的工作量。

# 设计模式

在为手头的任务设计解决方案时，很可能有更多的人遇到这个问题，并且有一个经过深思熟虑的解决方案。在大多数情况下，您可能需要根据自己的需求定制解决方案。

学习设计模式可以让你对问题有一个新的认识，或者找到一个快速的解决方案。

# 学习框架

大部分新生都懂一两门编程语言。很少看到使用过框架的新人。

大多数语言都有不少流行的框架，迫使开发人员根据语言的约定编写代码。它们为开发人员编写代码的方式带来了一致性。

# 了解业务

当我们写代码时，我们忘记了更大的图景。开发人员经常过于关注手头的问题，以至于忽略了它的业务方面。

开发人员往往更关注事物的技术方面。这一点也不差，这是他们的工作。但是了解商业方面可以带来一个新的视角来看待问题。
