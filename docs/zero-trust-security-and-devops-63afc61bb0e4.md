作为一个在金融行业工作的人，信息安全显然是任何软件项目最关心的问题。然而，安全问题在软件开发生命周期中出现得太晚，并在项目接近尾声时导致巨大的问题，这种情况经常发生。 **"DevSecOps"** 模型的一部分意味着在应用程序的设计阶段更早地涉及信息安全，以便可以更快地解决安全问题。

*快速上“DevSecOps”:*从现在开始我再也不会提到或使用 DevSecOps 这个词了。我们生活在当今的 DevOps 世界中，使您的 DevOps 实践成熟的一部分本质上意味着创建一种协作文化，在这种文化中，安全性就像运营一样重要。在 DevOps 中特别提到 Sec 是多余的，给营销人员增加了另一个术语。`/rant`

信息安全部门喜欢将“零信任”安全模型的概念引入任何关于云中应用程序的讨论，但这到底意味着什么呢？我们来探索一下。

## 定义零信任

“零信任”是 Forrester 的 John Kindervag 在 2010 年首次提出的一个短语，用来描述让安全领导者远离失败的以边界为中心的方法，并引导他们转向一种依赖于对每个设备、用户和应用程序的信任进行持续验证的模式。这是通过从`“trust but verify”`转向`“never trust/always verify”`的方式实现的。在实践中，该模型认为所有资源都是外部的，并在仅授予所需的访问权限之前不断验证信任。

## 零信任的五大支柱

![](img/8a12be1c15cd9d390738b961dbd64e5d.png)

**设备信任:**对于零信任，作为 IT 管理员，你需要了解你的设备，然后才能信任它们。您必须有一份清单，说明哪些设备归您的公司所有并由其控制。您必须有一个监控、管理和控制这些设备的解决方案。通过询问设备状态，您可以根据预先确定的安全策略确定设备是否可信，以及设备是否兼容。统一终端管理(UEM)解决方案使 IT 团队能够从单一控制台管理、监控和控制所有平台上的所有设备，包括移动设备、桌面设备、加固设备和物联网设备。此外，集成端点检测和响应(EDR)技术可以进一步检测可能的恶意端点活动，从而进一步改善设备安全状况。

**用户信任:**一次又一次，基于密码的用户认证被证明是低效和无效的。因此，作为零信任的一部分，组织必须利用更安全的用户身份验证方法。例如，一个强大的条件访问引擎可以使用动态和上下文数据做出决策。支持强大的条件接收引擎的技术构建块包括无密码身份验证(如生物识别、证书)、多因素身份验证(MFA)、条件接收策略和动态风险评分。

**传输/会话信任:**零信任的另一个关键组件是最低特权访问的概念。其思想是，用户或系统应该只访问那些执行手头任务特别需要的资源。不多不少。通过使用最小特权访问资源的原则，我们限制了对用户的访问，并授予他们执行工作所需的最低权限。帮助实现最低特权访问的技术构造块包括微分段、传输加密和会话保护。作为一个具体的例子，每应用隧道允许特定的应用逐个应用地访问内部资源。这种限制意味着您可以让一些应用程序访问内部资源，而让其他应用程序无法与后端系统通信。

**应用信任:**让员工能够从任何设备安全、无缝地访问任何应用，包括传统的 Windows 应用，这是创建数字工作空间和实施零信任的关键。随着用户身份验证的现代化，允许应用程序单点登录(SSO ),我们获得了安全性和更好的用户体验。对于不是为零信任设计的传统应用程序，我们以隔离的形式增加了保护。为了隔离和现代化传统应用程序，可以利用虚拟桌面或应用程序环境在传统体系结构和基于零信任的未来之间建立一座桥梁。

**数据信任:**归根结底，数据才是最重要的，这也是我们需要强大安全性的全部原因。我们必须防止数据泄露，并确保我们的用户正在交互的是正确的、未经修改的数据。数据丢失防护(DLP)等技术可确保敏感数据的意外泄露或销毁。虽然数据分类和完整性在很大程度上是由应用程序本身处理的，但是在构建零信任架构时，我们应该尽可能地提高信任级别。

## 那么这一切对 DevOps 意味着什么呢？

![](img/df7d118dac434dbdc834065691d893e1.png)

让我首先提醒我们 DevOps 的定义:

> *DevOps 是一种* ***文化*** *鼓励包括开发和运营团队在内的所有利益相关者之间的协作，并通过自动化改进流程，以提高软件交付的质量和速度。*

这里的关键词是“*利益相关者*”，因为在我们部署的任何应用程序中，信息安全都是一个巨大的利益相关者，尤其是那些在云中的应用程序。我们必须在任何应用程序设计决策的早期就让他们参与进来，以确保在我们应用程序的**“持续交付”**方面不会遇到最后一分钟的障碍。

自动化是这里的另一个关键词。显然，当我们将安全性作为策略代码来实施时，我们获得了自动化以及对分析我们的应用程序网络行为的新见解。这跨越了零信任安全模型的 5 个支柱，并固有地将安全控制嵌入到应用程序本身以及我们的 CI/CD 管道中。

我想说的是，安全性是应用程序的一部分，是 CI/CD 管道的一部分，因此，根据定义，它是“DevOps”的一个主要部分通过将这 5 个支柱分成不同的类别，我们可以在安全的这些方面进行深入研究，并了解我们的应用程序。

**TLDR: DevOps ==安全**
