# 1.学习领域驱动的设计

企业希望您解决一个软件问题。你选择方法。大多数开发人员会尽快完成。对现有代码进行修改，不要破坏任何东西，去喝杯咖啡休息一下。

你需要避免这种方法。

糟糕的领域设计会导致很多混乱。我犯了`Util`、`Manager, Helper`类。我们直到今天还保留着它们。

不要只提取代码。把它放在你领域的正确位置。

你需要`CustomerFacadeUtil`吗？检查内部工作，提取逻辑，并创建一个更好的名称。你未来的自己会感谢你。

> 在你的代码中明确领域概念意味着其他程序员可以更容易地收集代码的意图，而不是试图将算法改造成他们所了解的领域。— [丹北](https://github.com/97-things/97-things-every-programmer-should-know/blob/master/en/thing_11/README.md)

开发人员喜欢使用基本类型而不是特定于领域的类型。对此也感到内疚。我用了`String`，而不是`CustomerSpecificEnum`。你应该知道这不会有好结果的。

您应该总是求助于特定于域的类型。提高你的代码质量。减少调试时间。代码变得如此容易解释。

[Einar Landre](https://github.com/97-things/97-things-every-programmer-should-know/blob/master/en/thing_65/README.md) 写了几个要点:

*   代码变得更加可读，因为它表达了领域的概念，而不仅仅是浮点或字符串。
*   随着代码封装了易于测试的行为，代码变得更易于测试。
*   *代码有助于跨应用程序和系统的重用。*

# 2.你知道函数式编程吗？

我的学士论文是“在 Elixir 中开发稳定的软件”。我知道函数式编程范例。每个优秀的开发者都知道。了解它将扩展您的编程视野。

> 掌握函数式编程范式可以极大地提高您在其他环境中编写的代码的质量。如果你深刻理解并应用了函数范式，你的设计将会展现出更高程度的*参考透明性*。——[爱德华·加森](https://github.com/97-things/97-things-every-programmer-should-know/blob/master/en/thing_02/README.md)

我以前的技术负责人说过这句话。*“写纯函数！”。*纯函数不会因输入相同而改变结果。这意味着它们没有副作用。

另一个口头禅:*“你的函数不要超过 4 行。”*。

在写论文项目时，我做了临时测试。4 行代码可以包含多少逻辑？你可以把逻辑放入你的头脑，测试它，然后继续发展。

看 Java 8 里的 [Lambdas。Lambdas 需要了解高阶函数](https://www.w3schools.com/java/java_lambda.asp)。将函数作为参数传递是当今的主流。

函数式编程只是显得陈旧。近年来，它越来越受欢迎，社区也越来越大。

# 3.不要碰(工作)代码

小心使用您的服务器。所有的代码应该首先通过审查，然后部署。不要固定在生产上！

这不用说，但发生了很多。无论是创业还是自由职业项目。对此也感到内疚。

你知道那种感觉。简单的修复。一句话。应用后会破坏一些特征。花费更多的时间来修复。

大多数组织都有 CI/CD。不能在运行时添加代码。修补程序需要规划和重新测试。

也不要接触本地的工作代码。这在某些情况下是正确的。不要在编码中加入你自己的自我。

> 最好尽可能多地重用代码。不管代码有多难看，它都已经被测试过，审查过，等等。— [拉吉斯·阿塔帕图](https://github.com/97-things/97-things-every-programmer-should-know/blob/master/en/thing_06/README.md)

你看到的是低质量的代码结构。您对结构进行了更改。代码得到审查。出现 bug。你加班。你现在需要投入更多的时间来修复 bug。

> 如果有用，就不要碰它。

# 4.试验

> 由于软件的构建成本如此之低，正式的工程验证方法在现实世界的软件开发中没有多大用处。仅仅构建设计并测试它比试图证明它更容易也更便宜。——杰克·里维斯

杰克·里维斯说得好。我们作为软件工程师，不需要证明什么。我们构建和测试，因为这样做很便宜。

想象建一座摩天大楼。你需要做分析，地面测试，做适当的检查。尽管如此，你还是有问题。

然而，我们在这里。懒于测试软件。需要几分钟来测试。尽管如此，这仍然是令人畏惧的工作。

> 好的测试作为他们正在测试的代码的文档。他们描述了代码是如何工作的——杰拉德·梅萨罗什

人们应该总是为了清楚地理解代码而编写测试。测试积极和消极的场景。测试您的测试。触发错误的插件数据。检查错误发生后的情况。

# 5.编写更少的代码—做更多的事情

> *风格之美、和谐之美、优雅之美以及良好的节奏取决于简单。* —柏拉图

您应该始终努力简化代码。简单的代码使测试变得容易，解释速度快，并且是可变的。

遵循 YAGNI 原则。减少代码的混乱。删除所有无用的功能。没有客户端使用它—删除它。

不要创造临时的解决方案。我创造了一个。从没定位过。与其他小问题一起被搁置了。

创造临时解决方案会产生技术债务。这笔债务对这个项目造成了损害。和开发人员，因为他们需要回去。
