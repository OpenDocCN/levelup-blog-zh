# CICD 继续在行业中成长

报告发现，47%的开发人员以某种方式使用持续集成或部署。然而，只有五分之一的人使用持续集成和部署方法来自动化构建、测试和将代码部署到生产环境的所有方面。

![](img/9e3321d6980f3cee7eefea5c5fd105a7.png)

从这些数字中，我们有一个很好的指标，即进入使用开发-运营流程的门槛相对较低，如果可以，开发人员愿意承担这项工作。因为在一天结束时，DevOps 过程可以自动化掉开发人员可能有的一些常规活动，比如验证和测试他们的软件。我们看到了这一点，特别是在监控方面。许多 SAAS 提供商为许多标准需求提供即插即用的解决方案，因此开发人员在开发运维之旅的早期会选择这些解决方案，这是一个轻松的胜利。

当我们向下走到一些排名较低的 DevOps 活动时，由于组织买入所需的复杂性增加，开发人员很难完全实施甚至开始这些活动。例如，自动构建和测试代码变更对于单个团队来说很容易开始，但是在您的管道中创建自动回归测试来测试代码变更，现在需要组织同意什么是测试用例，什么是部署的截止点，什么是回滚计划等等。因此，这些变化需要组织看到这些方法的好处，并进一步开发他们的内部业务逻辑，以允许他们的工程师自动化测试和发布等工作，从而减少代码更改的交付时间。

# 谁在做 DevOps？

对于谁在执行或不执行某种开发运维活动，您可以看到所有规模的组织都在大力实施开发运维。

![](img/f13b45dc00c1758f5cc73e6ad21dc601.png)

同样，DevOps 承诺自动化开发人员的常规活动，这意味着无论组织规模如何，您都将在某种程度上受益于这种自动化。可以理解的是，自由职业者的比例最低，因为当你是唯一的开发者时，自动化一些行为没有意义。然而，即使是这些人也可以从 DevOps 中受益，他们可以利用自己的活动，使用通用开发平台进行开发，或者在与多个客户合作时更高效地部署项目，避免重复任务。

# CICD 让我们更快地发布更好的软件

该报告还发现，采用现代后端技术通常与代码更改的交付时间和恢复服务的时间的性能提高相关，但未能显著提高部署频率。速度和稳定性指标之间也有很强的相关性，当开发人员试图为他们的应用程序恢复服务时，这两个指标之间不会互相妥协。

![](img/9a92f1ca6230addbabc7a1cd1e0af235.png)

作为一名开发人员，这个准备时间非常合理。如果应用程序在生产中失败，如果我们不能更快地部署到它们，我们就不能更快地修复它。然而，在代码变更的发布过程中进行投资，以确保它适合您的开发人员的目的和精益，以便服务可以更快地恢复。

与投资于您的发布过程类似，投资于容器技术将会看到团队在 DevOps 活动中成为顶级执行者的可能性增加一倍。这里的性能是在为生产环境中的应用程序实现代码更改的交付时间内测量的。

![](img/af2c37825983d150ea337cbf1279a518.png)

这些容器工具鼓励对您的开发人员平台进行投资，以充分利用它们，这样团队就可以跨应用程序中可能涉及的多个代码库高效地执行容器构建，并将基础架构和部署配置为接近应用程序代码的代码。然后，开发人员可以更好地利用这个容器平台，通过简单的工作流更改来加快工作速度，例如，可以在本地机器上运行一个容器，用于与生产中运行的容器相同的开发。容器也非常适合绿地工作，在绿地工作中您可以遵循所有的容器最佳实践，在棕地工作中，开发人员可以使用容器来更好地定义和共享他们的本地运行时环境以进行测试，同时使用不同的部署方法仍然可以大大提高开发人员的工作效率并缩短代码更改的时间。

然而，容器不会自动让我们成为部署大师并修复我们所有的错误。

![](img/6de3e2f8a190d5fdc02a33f38db508c4.png)

由此我们可以说一件事，容器和微服务技术是复杂的。它们需要在平台上进行投资，以便在微服务的所有不同操作方面都处于领先地位，如监控、安全性和您正在运行的堆栈的性能。因此，我们看到，开始使用这些技术很容易，但要超越需要真正的投资和团队思维的改变，以遵循所有 DevOps 原则，例如拥有您的基础架构，在团队级别进行部署和监控，以减少代码更改和服务恢复的时间。

# 包扎

总的来说，作为一名 DevOps 工程师，我对这些结果并不感到惊讶。随着新工具变得越来越好，越来越容易集成到开发人员的工作流中，DevOps 作为一种实践仍在增长。几乎每个 SASS git 存储库平台都有一个内置的 CICD 系统，这使得使用 DevOps 比以往任何时候都更容易。虽然开发人员可以通过 DevOps 实践快速解决一些容易实现的结果，例如用于在合并前验证代码的自动化单元测试或源代码控制工作流，但获得回归测试和更快发布等基本的高影响力变化需要更大组织的真正支持，他们需要采用一些 DevOps 原则，并通过定义他们的产品和需求来允许更快速的变化，以便可以对其进行测试和自动化。但是，随着工具的成熟，这些唾手可得的成果几乎成为最佳实践的需求，这些更复杂的 DevOps 挑战正在慢慢得到解决，随着时间的推移，工具可以更好地为我们自动化这些需求。
