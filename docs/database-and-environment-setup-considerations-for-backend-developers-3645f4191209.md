作为后端开发人员，我们有一项艰巨的任务。我们制作管理人们数据的应用程序，我们可以编写代码对他们做很多事情，不管他们是好是坏。

在本文中，我们将看看在开发后端应用程序时需要注意的一些最佳实践。

# 持久性解决方案

大多数后端应用程序接收数据，并将它们保存在一个容易访问的地方。

我们必须考虑如何将我们的数据保存到一些数据库中，以便以后使用。

就像其他事情一样，我们必须通过选择符合我们需求的产品来妥善处理这个问题。否则，我们必须做出巨大的改变，以切换到适合我们需求的正确解决方案。

有几种数据库。最常见的是关系数据库。还有以非关系方式存储数据的 NoSQL 数据库。

关系数据库更常见，因为面向用户的业务应用程序使用的大多数数据都可以用关系方式建模。

此外，它有许多保护措施来确保数据和交易的完整性。

许多关系数据库系统都是 ACID 兼容的，这意味着每个事务都是原子的、一致的、独立完成的，并且是持久的。

原子事务意味着每个事务都是一个单元。

一致性意味着一个事务只能将数据库从一种有效状态带到另一种有效状态。

隔离事务意味着并发执行事务会使数据库处于与顺序执行事务时相同的状态。

持久事务保证一旦事务被提交，那么即使在系统失败的情况下，它也将保持提交状态。

NoSQL 数据库用于水平扩展数据，不需要 ACID 合规性。

有几种 NoSQL 数据库。它们包括文档存储数据库和键值存储。

文档数据库将数据存储为可以嵌套的文档，以父子关系存储数据。

它对于存储大量结构化文档非常有用。例子包括 CouchDB、MongoDB 和 ElasticSearch。

还有键-值存储，顾名思义，将数据存储为键-值对。

NoSQL 数据库更适合利基应用程序，所以大多数商业应用程序应该使用关系数据库作为他们的主要数据库，除非有很强的理由使用 NoSQL 数据库。

# 应对不同的环境

我们需要不同的环境，以便我们可以在不面向用户的环境中开发和测试我们的应用程序。

通常，我们需要应对一些环境。它们是本地开发环境、持续集成环境、测试环境、试运行环境和生产环境。

当地环境顾名思义就是为了当地的发展。这是我们编写代码并检查其工作情况的地方。

它应该有自己的测试数据数据库，我们可以用它来测试我们的东西。

持续集成环境是我们运行来构建我们的应用程序，并在它进入测试环境之前在其上运行测试的地方。

我们可以实现检查，以确保只有在所有测试都通过的情况下，我们的代码才会被合并到主分支中。

测试环境是进行测试的地方。这是一个共享的环境，代码尽可能多地从主分支部署。

暂存通常是生产环境的副本，我们在其中进行健全性检查，以确保它可能在生产中工作。

生产环境是用户使用我们产品的地方。

# 构建工件中的信息

构建工件应该包括一些关于构建的基本信息，以便在我们需要进行故障诊断时可以方便地使用这些信息。

我们应该包括用于构建它的关键工具库的版本。

此外，还应该包括所使用的依赖项。还应该包括构建的全局唯一修订。

如果需要，每个调试都应该包括构建包时使用的环境和变量。

如果我们选择不删除那些失败的构建工件，失败的测试和检查也应该包括在内。

# 结论

我们的构建工件应该包括有用的诊断信息。

此外，我们应该将所有这些环境包含在软件生产管道中，这样我们就可以在不干扰生产环境的情况下开发和测试软件。

最后，应该谨慎选择数据库，但最有可能的是关系数据库。
