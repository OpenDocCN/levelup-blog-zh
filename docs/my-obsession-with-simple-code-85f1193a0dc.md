写简单的代码对我来说已经成了一种困扰。

我发现自己试图简化我写的每一段代码。我经常在 pull 请求上添加注释，我们不需要把事情复杂化或计划得太远；现有的代码很容易维护。如果当前代码只需要一个`switch`语句，为什么要使用模式？如果你以后需要改变它，你会的。

我失去了所有干净的编码员了吗？我想是的。

我想展示对简单代码的痴迷是如何给你的团队带来更好的结果的。更好的是，对简单代码的痴迷通常会让你比试图实现复杂代码时更快地解决问题。这些都不是新的——马丁·福勒、埃里克·埃利奥特以及更多的人已经谈论这个想法很久了。

我想在对话中补充的是，有时候简单的代码*并不是我们经常想到的*干净的代码。简单的代码可能是一个`switch`语句。

# 什么是简单？

根据牛津词典，简单就是:

> 容易理解或做的品质或状态。

在软件中，“易于理解”是关键点。我们想要易于理解的代码。为什么？因为我们读代码的次数比写代码的次数多得多。

此外，如果我们希望我们的软件成功，它必须是可维护的。可维护性需要的不仅仅是简单的代码，但是**必须**需要它。团队会随着时间的推移而变化，需求会变化得更快，甚至我们自己的代码在我们写完几个月后看起来也是陌生的。我们必须保持代码简单，以便未来的开发人员(包括我们未来的自己)能够理解正在发生的事情。

# 简单与琐碎

在关于简单性的讨论中，人们经常会对简单代码的本质产生混淆。简单的代码很容易听起来像琐碎的代码。但是还是有区别的。

回到字典，琐碎的定义是:

> 缺乏严肃性或重要性；无足轻重。

在软件环境中，琐碎代码是指不做任何事情的代码。这不重要。你可以把它拿掉，一切都会好的。

![](img/dd20856e04040d88f5dfadd27a95921f.png)

另一方面，简单代码*做*一些事情！这很重要！能够以简单的方式编写重要的代码是软件工程师成长的最佳指南之一。

# 抽象将会出现

当编写软件时，许多工程师陷入对他们项目未来的猜测。写了一些简单的东西后，他们认为“我应该把它抽象出来以备后用。”通常发生的情况是，构建的抽象相当繁琐，难以使用。

相反，我们需要关注紧急设计或紧急抽象。让抽象随着时间的推移和不断的重构而发生，而不是猜测未来。遵循“三击出局”规则——如果你发现重复的代码三次或更多次，你就需要重构，并可能将其抽象掉。

专注于编写——字面上——完成任务的最简单的代码。然后，再简化一些。

如果你今天能写出最简单的代码，明天你就能更容易地重构这些代码。这就是我痴迷于简单代码的原因。因为未来易于更改对软件的成功至关重要。

# 抽象不仅仅是重复数据删除

谈到抽象，许多工程师认为抽象是“可重用的共享代码的集合”。我们获取一个函数，看到代码被复制，然后将该函数移动到一个公共位置以便重用，并声明我们已经将它抽象掉了。

但是抽象要大得多。更重要的是。

正如 Eric Elliot 指出的，抽象是编写简单代码的**核心。此外，抽象让我们能够以如此快的速度构建现代软件。由于抽象，机器变成了虚拟的，数据中心变成了云。由于抽象，位变成了文件，文件创建了数据库。**

抽象将复杂的事物转化为简单的交互方式。因此，即使在我们的应用程序中，抽象也不仅仅是删除重复的代码。在这篇文章中，我们没有时间来回顾具体的例子(尽管可能稍后会有一个)，所以请查看 [Eric Elliot 的视频](https://youtu.be/F-JvvFlYcts?list=PLWOdyjG6bHl54g4o3V-5ooeGnAfqWH6ja)，查看一个*伟大的*例子。关注要抽象的常见行为。而不是要消除重复的代码。

尽可能编写最简单的代码。然后在构建软件时无情地重构它。现在不要担心抽象——专注于编写简单的代码，抽象就会到来。
