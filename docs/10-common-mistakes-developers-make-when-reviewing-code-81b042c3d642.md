# 开发人员在审查代码时会犯的 10 个常见错误

> 原文：<https://levelup.gitconnected.com/10-common-mistakes-developers-make-when-reviewing-code-81b042c3d642>

审查别人的代码会让你觉得你在骂他们的宝贝孩子丑。但是那个代码可能是坏的，那个婴儿实际上可能是可怕的。当你看另一个开发人员的代码时，你想提供诚实和可操作的反馈。

当涉及到代码评审时，不要手下留情。一定要给作者最纯粹、最有建设性的评论。给他们直接的反馈，就是不要太残忍。

审查代码有时比实际编写代码更困难，这取决于内容。如果你对作者、代码领域甚至语言不是最熟悉的，那么这可能是令人生畏和艰巨的。

在本文中，我们将为新开发人员回顾一些常见的错误。这些都是简单的，看起来无伤大雅的事情，但它们会对评审者、作者和代码库产生有害的长期影响。

## 1.不读代码👀

这是显而易见的，但这是有原因的。你实际上并没有读代码。你只是浏览了一下。没有明显的语法错误。看起来不错，对吧？

*错了。*

阅读代码。阅读提交消息。阅读提供的任何相关文档。试着理解作者到底想要完成什么。这一切在逻辑上有意义吗？变量命名之类的东西看起来怎么样？

一行一行的读，不要检查出来，敷衍了事。除非你阅读代码，否则你不会提供有用的反馈。

## 2.不测试代码🧪

在 PR 中“运行”代码可能并不总是可行的，但是如果你真的可以，那么你会做得更好。如果您正在审查 UI 代码或任何对前端进行更改的内容，您需要看到这些更改。运行代码是唯一真正做到这一点的方法。

即使他们包括一些截图或复制的测试输出，也不意味着你真的自己尝试过代码。

拉树枝，做个造型，试代码。确保它有效。如果作者因为你没有相信他们的话而生气，那么告诉他们不要再回头看了，去多喝点咖啡什么的。

## 3.不留下评论💬

为了让作者得到反馈和建设性的批评，你实际上必须把它变成文字。看到你不认同的东西？*说话*。看到你觉得很酷的东西？*说话*。

当您阅读代码并给出反馈时，您可以留下评论。没有任何代码是 100%完美的，总会有一些问题需要解决。

花时间对他们的代码进行深思熟虑的评论。

## 4.❓，不要问问题

也许你想问一个关于他们做出的选择的问题？获得设计模式的清晰性？询问他们是否考虑了其他语法？

*在评论中提问。*

问他们为什么这样命名而不是那样命名。问他们为什么选择这个图书馆。想问什么就问他们，尤其是有不懂的地方。

## 5.没有提供明确的反馈💎

这几乎和不留下任何反馈一样糟糕。当你评论或质疑某件事时，确保它是集中的和清晰的。如果你不同意一个决定，但只是留下一个情绪化的评论，说它有多糟糕，这给作者提供不了任何有用的东西，你应该感到难过。

假设您不同意开发人员在特定位置使用三元运算符，这是您留下的评论:

```
"I wouldn't use a ternary here"
```

你认为这会让作者有多生气？不是因为他们不应该使用三元组，而是因为你甚至没有提供一个更好的选择或例子。

留下可操作的、清晰的反馈。如果你要建议什么…那就建议吧。

## 6.不要求架构公关变更👷

假设您收到一个包含 2000 行变更的代码审查。所有这些台词都是作者写的。他们已经为此工作了几个星期。公关是*巨大的*。你不可能在不牺牲反馈质量或者完全破坏你自己的开发工作的情况下合理地评审它。

不要害怕在 PR 上要求架构变更。这意味着不要害怕让他们把它分成更小的逻辑块。问他们应该先做哪些改变。哪些是相关的？有些改动可以以后再审核吗？

不要盲目接受庞大的评审(不管工程师多资深)。

## 7.不批评变量/函数名🗒️

获得正确的变量名可能非常困难。在编写代码时，很容易写出随机的、模糊的、有点意义的名字。后来，当波兰继续下去，这些名字可能看起来不那么热。

一定要确保重新评估变量和函数名的准确性。大声读出来。他们是做什么的？有意义吗？如果你不知道另一边是什么，你能猜出来吗？

花时间弄清楚关注点是否被合并到一个变量或函数中。名字用`and`多吗？是不是想做的事情太多？要求拆分或者改名。提供一个新的命名选项或关于如何划分的想法。

## 8.未确认测试、检查和林挺🏁

这是基本的代码审查内务。如果你收到一个新的评论，确保任何自动化测试都通过了。如果测试不是自动化的，那么你可以自己运行它们。确保任何其他自动检查正在运行并且也成功。

不要忘记确保代码被很好地链接。如果棉绒机是自动的，检查它们。如果不是，你可以自己动手或者询问结果。

作者最初应该这样做，但是评审者也应该仔细检查。底线是您需要直观地检查所有应该在代码评审中发生的检查。

## 9.不够灵活💪🏻

这份清单上的每一项看起来都是挑剔和敲打作者代码的武器，但事实并非如此。它们是供您使用的工具，可以提供最高质量的审查过程。它们不仅保护作者和评审者，也保护整个代码库。与评审者一样，这些工具是好代码的看门人。

有了所有这些工具和规则，审查过程有时会变得过于严格。如果发生这种情况，来回奔波已经成为一种负担，那么记住要有一点灵活性是很重要的。

在某些情况下，规则总是可以变通的。如果作者有后续的公关排队解决你剩下的问题，让他们休息一下，放松一下。总是使用你最好的判断，如果事情开始紧张，它可能是。

不要为了强硬而强硬。

## 10.没有放眼全局，🖼️

代码评审过程发生在微观层面。你深入杂草中检查语法、设计和语言使用。进入另一个工程师的大脑是一个复杂的过程。

记住大局是很重要的。记住这段代码的走向。它将如何与其他系统互动。谁会用。

永远保持广角视野，不要戴上眼罩。当你挑选单个函数时，很容易产生狭隘的观点，但是试着每隔一段时间后退一步，看看这个 PR 如何适应代码库的其余部分。也许你对一个内部的 bug 修复或者一个小的一次性脚本过于挑剔。
