# 你认为需要但不需要的 7 类开发人员

> 原文：<https://levelup.gitconnected.com/7-types-of-developers-you-think-you-need-but-you-dont-3a5d84ba8bdf>

## 关于依赖某些开发人员以及如何应对的误区

![](img/851223a3307d7129456ee14dd1d99b39.png)

图片由[皮克斯拜](https://pixabay.com//?utm_source=link-attribution&utm_medium=referral&utm_campaign=image&utm_content=1492561)的 Gerd Altmann 提供

在信息技术领域，开发人员/软件工程师是公司的核心。他们构成了团队的大部分，因此，他们定义了团队文化。

因为我们都是不同的个体，每个开发人员都有他自己的个性，但是在大多数团队中都可以看到某些突出的“类型”。几乎每个团队都有特定的开发人员，他们被认为是经理最喜欢的，或者是管理层一切都依赖的人。在某些情况下，这是真的，但如果允许这种情况持续很长时间，它会给团队注入毒性。

一个团队中总会有杰出的表现者，但当他们成为团队的瓶颈时，那就是管理失败的明显迹象。个人的努力应该得到回报，但重点应该是建立一个强大的团队，拥有超越个人英雄主义的分布式专业知识。

许多经理认为团队没有特定的个人就无法生存，但这是一个神话。这个神话继续盛行，因为没有人敢挑战它。

我列出了某些杰出类型的开发人员，他们通常被认为对团队非常有价值，但实际上可能弊大于利。我相信你能在你的团队中找到一个或多个这样的人。

# 1.最后一个站着的人

在一个软件产品公司中，经常会有技术的转移或架构的改变，导致不同代的产品。然而，旧的产品或“遗留系统”继续拥有客户和一些工作量。剩下的工作不多了，团队转移到其他途径，但有几个或一个留了下来。现在，随着每一个客户问题的出现，世界的中心转移到最后一个了解旧产品的人身上，或者看起来是这样。

如果他们允许他们的客户依赖这个人，这是管理层的失败。过去的普通员工将成为未来的重要资源。作为团队的问题解决者，他们经常得到更高的评价并赢得尊重。管理层开始认为这个人是不可或缺的，但这种一厢情愿的想法并不能解决问题，就像其他人一样，这个人也可以离开。与其陷入“总线因素”的境地，不如努力将遗留系统的所有权和知识保留给团队而不是个人。

# 2.态度恶劣的天才

这是在技术方面非常擅长的开发人员，但是没有人愿意和他们一起工作。作为一名经理，当你需要某人来解决一个复杂的问题或研究新技术时，你会去找他们并得到预期的结果。另一方面，团队中的其他开发人员由于这个人态度不好，不愿意接近他。这位开发人员的技术专长使他对团队至关重要，但大多数人更喜欢避免互动。由于这些人被认为是如此重要，没有人抱怨他们，但团队有很高的流失率。

这些娇生惯养的人本可以成为伟大的专业人士，但却从未得到适当的指导。即使某人是极其有价值的贡献者，也应该考虑对团队的整体影响。忽视糟糕的态度会向团队传递错误的信息，并间接赶走其他优秀员工。总之，对一个人的依赖剥夺了团队拥有多种才能的机会。经理需要与这些人一起工作来指导他们，并为在团队中工作设定正确的期望。任何人都不应该觉得自己凌驾于团队中的任何人之上。

# 3.引人注目的人

如果你在运行幻灯片的任何地方都看到相同的面孔，从 sprint 评审到客户演示，很可能你的团队中就有这种类型的开发人员。也有一些真正的执行者，但是三个团队中的每两个团队都会有一个开发人员，他通过抓住每一个提供可见性的机会(不管有没有工作)来解读成功的公式。

你可能认为团队中需要这样的人，但是要小心，不要让他们抢了别人的功劳，或者排挤其他团队成员的积极性。我的团队中有一个这样的高级架构师，他不让任何其他人展示团队的工作。最初，没有人会在意，但后来这成为一种趋势，团队成员开始感到被忽视，因为只有一个人得到了所有的荣誉。每个人都有不同的沟通风格，在一些关键会议上选择最佳演讲者是没问题的，但作为经理或领导者，培养他人并为他们提供学习这些技能的机会也是他们的责任。

# 4.先锋先生

“嘿，伙计们，为什么不试试这个在谷歌很受欢迎的新 XYZ 框架呢？我听说他们将性能提高了##%”或“这本最新的畅销书解决了您在项目中面临的确切问题”？这是一个了解技术最新趋势的人，他读过某个影响者的每本书/博客，并且总是谈论大多数人不理解的东西。讨论任何新概念或新技术的关键人物。他/她也是聚会中的重要人物，经常提问以展示他/她对主题的理解。大多数高管和经理都对精通技术的工程师有很高的评价，因为他总能分享一些有趣的东西。

最初，你可能还会认为他是个天才，因为他总能从书中的例子中找到每个问题的答案。然而，很快你就会意识到，这个家伙从来没有参与解决任何问题，所有的建议或指导都是肤浅的，没有任何研究或真正的实施。他们唯一的动机是创造一个理论知识的门面，从来不负责实施任何东西。我见过这样的例子，这些人仅仅通过在社交聚会上陪伴高管就获得了晋升。它们之所以糟糕，是因为它们对努力工作的团队成员的士气产生了负面影响。重要的是要区分空谈和解决实际问题的实际工作。

# 5.完美主义者

这是一个很难识别的问题，有些人甚至会支持这些性格类型。我所说的这种开发者并没有恶意，只是有很多不安全感。通常，他们在某一特定领域有丰富的经验，并对某种工作方式有偏见。他们希望每件事都以某种方式完成，并会不遗余力地改变周围每个人的行为，以那种方式工作。他们的代码审查会有一些价值，但随之而来的是大量关于代码风格、评论语言、字体、风格、标点符号等的评论。

代码的一致性总是好的，但是当你开始因为个人喜好而忽略代码的优点时，问题就来了。这些开发人员试图通过延长审查和批准的时间来一次又一次地建立他们在团队中的主导地位。他们让任何事情听起来对任何其他工程师来说都太复杂，并形成了一个封闭的可信任的同事群体。管理者需要评估每项努力的可行性，并努力促进一种开放的文化，在这种文化中，每个人的意见都得到考虑，并得到公平的机会，而不是陷入一个只有一个人可以解决的过于复杂的陷阱。价值和努力之间应该有一个平衡，有时，过多的审查可能不值得输出的价值。

# 6.熟练的骗子

这是一种“吝啬”的开发人员，他们不仅聪明，而且有点聪明过头。他/她通常擅长编程技能，但只使用这些技能来最大化自己的利益。他们会一直做好工作，但会按照自己的意愿去做。他们通常不关心团队的目标或速度，只关心他们自己的时间表。如果您询问任何工作估算，您通常会得到一个回答，“您什么时候需要它？”

原因是他们知道他们可以很快完成，但他们不会接更多的工作，他们会抓住以前的任务，直到它成为优先事项，然后按时交付。他们也不与团队分享他们的专业知识。许多不了解细节的经理认为这个人总是能扭转乾坤。另一方面，这是那种促进有毒行为的人，不是一个好的团队成员，不可靠。你总是怀疑自己是否能信任他们的反应。作为团队的领导者，与其宣扬个人英雄主义，不如更看重那些通过诚实的努力促进团队协作和知识共享的人。

# 7.过量使用的控制器

这是团队中包括管理层在内的每个人都欣赏的最受欢迎的开发人员。他们善于完成自己的工作，参与每个人的工作，帮助他人，甚至自愿承担额外的工作。这一切都很好，使他们成为队中表现最好的球员之一。在某些情况下，这些高绩效者更进一步，最终获得所有工作的所有权。他们设计，他们编码，他们向其他团队成员提供如何编码的逐步指导。他们开始审查每一行代码，检查任何人做的每一件事。在其中一家公司，我看到个人开发者被贴上解决方案正式所有者的标签。

当一切都变得依赖于一个人，而其他表现良好的人开始感到窒息时，问题就来了。没有其他人成长的空间，因为每个人都必须与一个人的思维过程保持一致。最终，人们开始离开，你可能会以团队中除了一个“所有者”之外的高流动率而告终。在大多数情况下，过度承诺的个人都有良好的意愿，每个人都赞赏他们的贡献，这就很难诊断出真正的问题。拥有高绩效的个人对团队总是有好处的，但我们需要在培养和指导其他团队成员时引导他们的优势，并将其转化为对每个人都有利的双赢局面，而不是变得依赖他们。

# 最后的想法

每个团队都有不同的个性，我并不是说我上面列出的类型对团队不好。最终，这一切都归结于整个团队的组合，以及不同的性格类型如何混合。在某些情况下，你可能需要这些不同类型的人，他们可能是你唯一的救星。我想说的是，工程领导者需要识别每个人的价值，并确保他们充分利用每个人的能力。重点应该是建立一个公平公正的工作环境，在这个环境中，人才比“需要”更受重视。这需要不断的评估、反馈、信任和透明。从长远来看，一个产品团队只有在团队的贡献下才能茁壮成长，而不仅仅是一个人。
