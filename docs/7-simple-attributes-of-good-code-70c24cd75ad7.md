在软件开发中，一个长期反复出现的建议是写好代码。作为一名程序员，你一定已经意识到了它的重要性。好的代码有助于在未来很长一段时间内产生可伸缩和可维护的系统。

你可以读一段代码，自己决定好不好。但是好的代码不仅仅是看上去的那样。一个简单的编程原则，用三个简单的词抓住了一个好代码的本质，那就是“易于更改”(等等)。

# ETC —易于更改

像生活中的其他事情一样，软件工程中的变化是不可避免的。无论你做了多少计划，你都无法设计出一个完美的系统——一个永远没有任何问题地持续满足用户需求的系统。

谨慎的方法是开发能够跟上变化步伐的代码。如果你的代码可以很容易地随着用户的行为和需求而改变，你可以称之为好代码。

ETC 原则列出了代码的属性，在需要的时候很容易修改。为了使代码易于修改，它需要是可读的、可测试的和可重用的。它还需要可靠、可维护，并且无需任何用户努力就能优雅地处理故障。

让我们详细讨论一下这七个简单的属性，它们可以帮助你遵循 ETC 原则，并启发你写出好的代码。

# 1.代码必须是可读的

如果你正在写一个一次性的代码，将来没有人需要阅读，包括你自己，那么你可以用你喜欢的方式写。但是大多数有用的软件都需要适应常规的增强。

在团队环境中，一组开发人员需要在相同的代码基础上工作。他们必须扩展彼此的功能或修复其中的缺陷。如果代码不容易阅读，没有人能理解它。

一个好的代码也必须是无需过多解释就可以理解的。能够用更直白的形式编码复杂的逻辑，无疑是一种技能。无论代码多么高效，如果没有易读性，它很快就会成为团队的负担。

# 2.代码必须是可伸缩的

好代码的第二个基本属性是可伸缩性——代码可以随着业务的需要而伸缩。可伸缩性主要是关于代码的效率。可扩展的代码不需要频繁的重新设计来保持处理不同工作负载的性能。

通常，系统的性能越好，就越容易扩展。如果您的可伸缩性很容易预测，那就更好了。一个好的代码应该让它的读者毫不费力地预测它将来是否可以扩展。

但是可读的代码并不总是可伸缩的。例如，以多层嵌套循环为例。当然，任何懂这种语言的人都能看懂。但是你可以确定代码是不可伸缩的。

# 3.代码必须是可测试的

一个可测试的代码是一个好的代码，因为它总是准备好投入生产。在软件开发中，最后一刻的变化是不可避免的。如果你的代码不能被快速和自动的回归测试，它将很难通过任何新的修正。

如果你的代码足够好，你不需要测试你的整个产品来做简单的改变。您可以单独测试修改后的组件，并且仍然可以确定整个流程。因此，如果你能独立测试你最小的组件，你的代码已经达到了很好的可测试性。

# 4.代码完成了所要求的工作

每一个实现的软件都有一个预期的目的。对于了解功能的人来说，符合需求的代码很容易理解。因此，满足功能规范是好代码最重要的属性之一。

每个代码执行的输出都需要是可预测的。当代码不能满足用户的需求时，代码再漂亮也没有用。如果它在工作流程中产生一堆缺陷，或者计算的结果并不总是正确的，那么代码是完全无用的。

不必要的功能对应用程序来说也是无用的。在最好的情况下，它们会使软件膨胀，并使其他开发人员感到困惑。在最坏的情况下，它们会降低性能或者在所需的特性中引入问题。因此，编写准确满足业务需求的代码总是谨慎的。

# 5.代码会优雅地失败

失败和缺陷是任何软件不可避免的部分。无论你在设计和开发过程中多么小心，你都不能预见每一个可能的生产场景。但是，您应该保护您的应用程序免受这些问题的负面影响。

一段糟糕的代码可以毁掉整个应用程序。在负载较重时，执行缓慢的查询会消耗所有数据库连接。外部服务调用失败可能会耗尽您的线程或中断业务工作流，而不会向用户发出适当的错误消息。

好的代码需要适当地处理任何这样的失败，而不会将错误处理问题与正常的应用程序流混淆。代码需要足够可靠，不会将失败传播到执行上下文之外。异常也需要尽可能早地被捕获，并在适当的层优雅地处理。

# 6.代码很容易扩展

易扩展性是好代码的基本属性。可扩展的代码既可重用又可维护。您只需编写一次代码，并确保它按预期运行。需要它的模块可以扩展该特性，并在其上添加新的实现。

可扩展性也减少了测试的范围——不需要一直测试整个实现。只有扩展部分需要测试。它的多个其他扩展可能已经验证了可扩展代码。

因此，如果开发人员可以轻松地扩展您的代码，而无需修改它来增强后代代码中的特定功能，那么它就可以被称为好代码。

# 7.代码是可重用的

对于任何软件产品来说，可重用代码都是必要的，而且非常有价值。它有助于保持你的代码简单，减少重复。从长远来看，可重用的代码可以节省时间，并且更具成本效益。

如果您正在编写实用程序方法，那么它必须是理想的可重用的，无需任何更改。您的共享代码需要修改的越少，它可以实现的稳定性就越好。几乎不会有任何缺陷，因为每一次新的使用都会经过多次测试。

然而，这并不意味着如果你的代码不可重用，它就不好。不是每个代码都需要可重用。根据特定的用例，重写某些特性可能更明智。

> “一点点模仿胜过一点点依赖”——罗布·派克

# 最后的想法

永远记住——好代码的替代品并不是坏代码。如果您的应用程序代码的一部分不符合上面提到的一些属性，这并不意味着它是坏的。

根据环境、手头的工具和项目时间表，开发人员决定编写以后可能不容易更改的代码。只要您理解 ETC 原则，并意识到好代码的属性，您就可以自己决定如何实现您的需求。
