再深入一点，让我们看看为什么这三个断言是正确的。

# 不写测试会让你暴露在错误面前

这一点应该是不言自明的。如果你没有自动测试来确保你的代码正常工作，那么你就要依靠手动测试来验证你的应用程序的行为。手动测试是乏味的，随着应用程序的增长，一个人要记住整个应用程序架构的心智模型，并了解应用程序某个部分的任何给定变化如何一直影响应用程序的其他部分，这变得很难。

未经测试的代码是错误代码。即使代码现在没有错误，也不会保持很长时间。

# 不写测试会让重构更加困难

测试提供了重构过程中所需的支持，以确保应用程序在更改后仍能正常工作。没有适当的测试，你真的不能确信你的重构是安全的。你回到了彻底手动测试整个应用程序的责任，以确保没有倒退发生。让自动化测试为您完成这项工作不是更好吗？

# 从长远来看，不写测试会让你走得更慢

你的应用程序变得越大，隐藏 bug 的地方就越多，你必须花费更多的时间来手动验证你的修改。在手动测试过程中，不可避免地会遗漏一些东西。bug 会被发布到产品中，你的团队会把时间花在救火和修复 bug 上，而不是开发令人兴奋的新特性。

# 有更好的方法

如果我描述的场景听起来对你没有吸引力，很好！幸运的是，有一个更好的方法。始终为您的代码编写测试。如果你的回购目前没有测试，现在就采取措施解决这个问题。确保每一个新的合并请求都包含测试，并在进行的过程中缓慢但稳定地增加你的回购测试覆盖率。你会很高兴你做了。
