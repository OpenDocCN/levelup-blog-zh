交付好的软件变得非常重要。这个我解释过很多次了。除此之外，擅长交付软件也很重要。DevOps 是组织以高效的方式交付高质量软件所遵循的一套实践。

在这篇文章中，我将解释一个组织在交付软件时的挣扎，以及为什么需要 DevOps。然后，我将解释什么是 DevOps，以及一个组织如何采用它的实践。

# 为什么需要 DevOps？

让我们以一个中型组织为例，该组织由于市场上的新竞争对手而表现不佳。它最近记录了糟糕的收入增长和高流失率。该组织已经意识到，它必须通过改进其交付软件的方式来进行改变。

我们来讨论一下面临的一些问题。测试(手动测试)和构建的自动化程度有限。

软件交付缓慢。一个想法一旦被接受，可能需要数年时间才能投入生产。多个团队通常会堆积很多工作。

生产中的软件运行时间很短。当某个东西出现故障时，通常会有很多恐慌，恢复时间真的很长。

大多数安全问题被认为是安全团队的责任，没有人关注安全。有时，为了更快的交付，安全性会受到影响。

最后，公司缺乏对客户的关注，团队没有产品思维。每个人都关注自己的事情，开发人员关注编码，项目经理关注交付时间和预算，QA 团队关注软件质量。

# DevOps 是什么？

根据 Gartner Research 的研究，“DevOps 代表了 IT 文化的变化，通过在面向系统的方法的背景下采用[敏捷](https://medium.com/gitconnected/everything-you-need-to-know-about-agile-dc90cc35108c)、[精益](https://theleanway.net/The-Five-Principles-of-Lean)实践，专注于快速的 IT 服务交付。”

DevOps 更关注自动化，以减少软件开发和交付之间的时间。重点更多的是客户满意度和发现在市场中有效竞争的方法。

现在，我们来讨论一下 DevOps 不是什么。DevOps 不仅仅是一个团队或者一种方法论。一个组织可以有一个 DevOps 团队，但是他们不是唯一遵循其实践的团队。这不是你能买到的产品。此外，它不仅仅是自动化某些流程。

最后，DevOps 绝对不容易。一个团队不能只说“让我们为这一个做 DevOps”。采用 DevOps 时，许多事情需要改变。

我们将在下一节讨论我们的公司如何做到这一点。

# 为组织采用 DevOps

为了采用 DevOps 实践，组织需要做出一些关键的改变。这些变化包括文化和组织的变化，引入新的流程和技术。

主要目标是持续改进和为客户提供价值。就像任何新的实践一样，采用 DevOps 应该是渐进的，只能以增量的方式进行。

## 文化变迁

文化变革的主要部分是开始拥有产品的所有权。整个团队应该致力于追求卓越。例如，软件的质量不仅仅是 QA 团队的责任，甚至开发人员也必须关注质量。

当一个团队拥有一个产品时，团队中的每个人都对产品的成功负责。这可能包括在出现危机时承担额外的责任。这需要团队之间的高度信任。

团队应该采用**产品思维**，并以产品的价值为导向。DevOps 建立在产品思维模式的基础上，关注软件如何交付价值。

这与项目思维模式相反，在项目思维模式中，只关注项目目标的实现。这需要深入了解客户的需求。

重点应该放在客户身上。这包括思考产品试图解决什么问题。组织应当鼓励更多的顾客互动，并定期获得反馈以改进产品。

另一件重要的事情是**做出数据驱动的决策**。在没有任何数据支持的情况下，它不应该做出任何重大决策。这让每个人都知道他们实际上在做什么。然而，这需要团队收集相关的和可量化的数据。

最后，它需要采取持续改进的心态。根据收集的数据，应对产品方法进行任何必要的改进。然而，改进的方向应该是正确的。

## 组织变革

IT 团队通常分为不同的团队和角色，如项目经理、架构师、数据库管理员、开发人员等。这些团队中的每一个都非常有效率，并被驱使着去做他们所做的事情，但是当涉及到提供一致的价值和客户满意度时，他们可能经常做不到。

因为他们每个人都专注于自己的目标，所以很难驱使每个人走向相同的结果。这通常会导致时间或资源的浪费。

基于产品的结构是一种有益的改变。在这里，所有的团队都致力于一个目标，并且是跨职能的。

平台团队通过创建和管理平台来帮助改进软件交付。他们将平台本身视为一种产品，因此他们致力于不断改进它。SRE(现场可靠性工程师)团队为这些平台提供额外的支持。

应该建立开放的交流。良好的沟通可以避免很多问题。持续的反馈和协作有助于让每个人都保持一致。

最后，本组织不应仅仅确立其价值观。他们应该开始实施这些措施。一种方法是用福利和晋升来奖励团队。这有助于保持每个人的积极性。

## 练习

DevOps 和敏捷实际上是相辅相成的。很多敏捷实践都符合 DevOps。然而，DevOps 采取敏捷实践直到集成和交付( [CI/CD](https://www.redhat.com/en/topics/devops/what-is-ci-cd) )。

软件应该以小增量交付，这样可以更快地学习和改进。这为需求和特性的变化提供了更多的空间。这也降低了部署成本。

服务水平目标衡量服务的性能。应该设定这些目标来帮助团队实现其目标。在每次部署之前实现自动化测试。最后，通过以吉拉故事或任务的形式记录下来，开始让所有的工作可见。
