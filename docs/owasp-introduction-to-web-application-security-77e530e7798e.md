**Django 如何帮助减少漏洞。**

在构建 web 应用程序的过程中，Web 开发人员通常不会考虑安全措施，尤其是当您与安全专家一起工作时——我们倾向于让他们处理安全性的全局方面，这并不总是理想的。

> 开放 web 应用程序安全项目(OWASP)是一个在线社区，为 Web 应用程序提供免费的文章、文档、工具或安全建议。我们将要涵盖的列表并没有列举所有与 web 应用程序安全相关的问题，而是根据安全专家在执行渗透测试时看到的情况，列出了前 10 个问题。

Django 的构建是为了最小化这些安全风险，同时也让开发人员能够通过使用更好的实践来避免和减少这些漏洞。让我们看看怎么做。

1.  **注射**

这是技术社区中最常见的安全问题。注入简单地说就是插入一段将在系统内部执行的代码。例如，SQL 注入的一个漏洞，让黑客在我们的数据库中执行任意 SQL 代码，这可以让他们完全控制我们的系统。

Django 中 QuerySet 类的使用在某种程度上保护了我们免受 SQL 注入的攻击。QuerySet 确保发送的所有查询都是参数化的，以便数据库可以区分查询中的 SQL 代码和值。

**2。被破坏的认证&会话管理**

这是指有人将自己认证为另一个用户或接受另一个用户的会话的风险。

Django authentication 总是对密码进行哈希处理，这样即使你的应用程序受到威胁也无法破解。所以作为一名开发人员，总是使用 Django 内置认证或简单地扩展它——虽然它提供了安全性，但也节省了您的时间和精力。

Django 会话 ID 永远不会出现在 URL 中，并且在登录后也会改变。

**3。跨站点脚本**

跨站点脚本(XSS)是指攻击者能够让 web 应用程序显示他自己创建的 HMTL 或 javascript，而不是开发人员创建的。

Django 默认用 HTML 编码保护模板中的所有变量。

**4。不安全的对象引用**

当我们暴露我们的资源引用而没有保护资源不被利用时，就会发生这种情况。例如，使用在 URL 中公开的自动递增主键可以让攻击者遍历数据库中的所有项目。这不是一个安全的做法。

Django 通过在设计模型时使用像 UUIDField 这样的字段来帮助我们避免这种情况。

**5。安全错误配置**

这是在不应该部署正确的安全措施时发生的风险。

例如，在 DEBUG 设置为 True 的情况下运行 Django 应用程序，会向公众暴露太多有价值的信息。

Django 项目网站和社区在解释如何在部署前设置特定的配置，或者如何管理多个配置文件以避免遇到这个问题方面做了大量工作。

**6。敏感数据暴露**

这是数据在没有适当授权的情况下被访问的风险。在项目的开始和整个过程中讨论数据总是好的做法，比如:什么数据是最重要的？数据应该如何处理？或者等等…应该经常放在桌子上。

Django 可以被配置为仅在 HTTPS 提供页面服务，这降低了 dat 被暴露然后被攻击者劫持的风险。

**7。被破坏的访问控制**

这是指功能，或者说业务层不受保护。例如，您不希望在您的 web 应用程序上允许匿名发布产品反馈。这将避免未知用户发送数据。

Django 通过提供像权限类这样的特性来帮助我们处理这些问题。

**8。跨站点请求伪造(CSRF)**

CSRF 可能是 OWASP 列表中最复杂的风险。

Django 提供了一个 CSRF 令牌来确保任何对数据库进行修改的操作都有一个惟一的值，只有我们的系统知道这个值，这个值确认用户是有意使用应用程序来执行操作的。

**9。使用具有已知漏洞的组件**

根据您使用的库或框架，它们可能有已知的漏洞，但您必须始终保持所有安全版本和更新。

Django 项目始终让社区了解影响框架的问题。

**10。错误的重定向和转发**

这是指重定向到第三方网站。例如，Django 让您能够操纵 URL，以便 URL 的下一个目的地只是项目的一部分。

作为一名 web 开发人员，使用一个保证安全性的框架总是很好的，但更好的是在构建应用程序时测量所有涉及的漏洞，并知道如何保护自己免受它们的影响。
